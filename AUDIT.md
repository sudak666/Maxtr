## Критичні проблеми

### 1. Два service worker на одному scope — підтверджено

`sw.js` реєструється завжди при завантаженні сторінки (`index.html:22`,
кореневий scope). `firebase-messaging-sw.js` реєструється окремо, коли
користувач вмикає push (`index.html:5935`), теж без шляху — тобто теж на
кореневому scope. Браузер може мати лише один активний service worker на
scope, тож активація push-воркера конкурує з кеш-воркером за той самий
scope.

Наслідки: кешування PWA може ставати нестабільним після ввімкнення push;
повторний запуск застосунку може повернути `sw.js` назад; фонові
сповіщення можуть працювати непередбачувано.

Треба об'єднати кешування й Firebase Messaging в одному service worker,
або чітко розвести їхні scope (наприклад, зареєструвати один з них під
підшляхом).

### 2. Self-XSS через `showToast()` — підтверджено, з уточненням

`showToast()` (`index.html:4228-4235`) вставляє повідомлення напряму в
`innerHTML`:

```js
t.innerHTML=(icon?window.Icon(icon):'')+'<span>'+msg+'</span>';
```

Підтверджені виклики з даними, введеними самим користувачем, без
екранування:
- `index.html:6422` — `rule.category` (назва категорії в правилі автозаповнення);
- `index.html:5445` — `name` (назва гаманця/категорії при пошуку).

**Уточнення щодо серйозності**: оскільки Firestore-правила ізолюють дані
по `uid` (`firestore.rules:9-11`), це не міжкористувацький XSS — атака
можлива лише через дані, які користувач ввів сам собі (self-XSS). Все
одно варто виправити, бо: (а) синхронізація між пристроями того самого
акаунта означає, що шкідливий payload, введений на одному пристрої,
виконається на іншому; (б) загалом 94 місця з `innerHTML` у файлі — не всі
перевірені індивідуально, і патерн варто виправити системно, а не
точково.

Треба вставляти текст через `textContent`, іконку — окремим DOM-вузлом;
перевірити решту 94 місць з `innerHTML` на аналогічний патерн.

### 3. Push-нагадування рахують час у UTC — підтверджено, але це задокументоване обмеження, не незамічений баг

Cloud Function справді порівнює час користувача з `now.getUTCHours()`
(`functions/index.js:87`), і нагадування на 21:00 у Києві прийде о 18:00
або 19:00 UTC-зсунутого часу залежно від сезону (тобто не о 21:00 за
Києвом).

**Уточнення**: коментар прямо над цим кодом (`functions/index.js:138-145`)
вже пояснює це як свідоме, задокументоване обмеження ("best-effort daily
reminder, not to-the-minute precise"), а не недогляд — автор коду знав
про це й залишив нотатку, що саме треба зробити для фікса (зберігати
timezone/offset користувача в `notifSettings`, рахувати локальний час
через `Intl.DateTimeFormat`). Пріоритет виправлення від цього не
змінюється — просто фрейминг "критичний непомічений баг" неточний.

### 4. Офлайн-режим неповний — підтверджено

`sw.js` кешує лише `./`, `./index.html`, `manifest.json` (і статичні
іконки) зі списку `STATIC_ASSETS`. Firebase SDK завантажується з
`gstatic.com` і не кешується — холодний запуск без інтернету ненадійний,
хоча дані є в `localStorage`.

Треба або локально збирати Firebase SDK, або контрольовано кешувати
дозволені зовнішні залежності.

## UI/UX: мінуси

**Дрібний текст.** У CSS багато тексту 9–11px — замало для навігації,
фільтрів, підписів на мобільному. Мінімум для звичайного тексту —
приблизно 13–14px, для другорядного — 12px.

**Маленькі touch targets.** Частина кнопок (видалення, фільтри, малі
круглі іконки) — приблизно 28–34px. Ціль для телефону — близько 44×44px.

**Перевантажена вкладка фінансів.** Одночасно гаманці, операції, курси,
конвертер, аналітика, графіки, бюджети й цілі. Варто залишити зверху
баланс, швидкі дії й останні операції; аналітику й курси/конвертер
винести в окремий екран/підвкладку.

**Забагато декоративних ефектів.** Різні градієнти на вкладку, glow,
ripple, pop і haptic одночасно створюють візуальний шум. Варто залишити
одну акцентну систему, короткий press feedback, haptic — тільки для
важливих підтверджень.

**Слабка семантика — підтверджено.** У документі відсутні `<main>`,
`<nav>`, `<header>`, `<section>`, `<h1>`–`<h3>`, `<dialog>` (усі перевірено
`grep` — нуль збігів). Заголовки й секції зроблені через `<div>`.

**Клікабельні `<div>` — підтверджено.** 214 атрибутів `onclick` у розмітці
(рахунок точний: `grep -o 'onclick=' index.html | wc -l` = 214). Частина
має `tabindex`/`role="button"` як тимчасовий захід, але лише 0 `role="button"`
насправді знайдено при точному пошуку — варто перевірити це окремо перед
плануванням фіксу доступності, оскільки початкова оцінка "78 елементів" не
підтверджена точним підрахунком.

**Модальні вікна — підтверджено.** Немає жодного `aria-modal` у файлі
(0 збігів). Модалки — звичайні `<div>` без focus trap, повернення фокуса,
гарантованого закриття по Escape.

**Icon-only кнопки — підтверджено частково.** Лише 13 атрибутів
`aria-label` у всьому файлі — для застосунку з 214 `onclick`-елементами і
численними icon-only кнопками видалення це явно недостатньо.

**Контраст.** `--muted2: #68676e` на темному фоні — слабкий контраст, не
варто використовувати для важливих підписів.

## Код: плюси

- Firestore ізолює дані за `uid` — правило `users/{uid}/max_tracker/{document=**}` (підтверджено, `firestore.rules:9-11`).
- Старий публічний шлях явно закрито (`firestore.rules:24-26`, `allow read, write: if false`).
- Firebase API key у фронтенді — нормально, не є секретом.
- Є оптимістичне виявлення конфліктів між пристроями (`lastKnownUpdatedAt`).
- Профілі реалізовані зі збереженням сумісності зі старими даними.
- SDK Firebase зафіксовано на конкретній версії (12.11.0).
- Signing keystore не закомічений у репозиторій.

## Код: мінуси

### Моноліт — підтверджено

`index.html`: 7051 рядок, ~427 КБ (перевірено `wc`), 94 місця з
`innerHTML`, 214 `onclick` у розмітці. Будь-яка зміна може випадково
зачепити інший розділ.

Рекомендований поділ (не обов'язково React, звичайних ES-модулів
достатньо):

```
src/
  app.js
  firebase.js
  state.js
  auth.js
  shifts.js
  finance.js
  debts.js
  shopping.js
  notifications.js
  ui/
  styles/
```

### Забагато глобального стану

Функції залежать від module-scope змінних і `window`-експортів — складно
тестувати частини окремо, відстежувати мутації, уникати конфліктів.

### `finance` — один великий Firestore-документ

В одному документі: транзакції, гаманці, категорії, бюджети, правила,
цілі, покупки, налаштування. Ризики: ліміт 1 MiB на документ; кожне
збереження переписує весь обсяг; конфлікт будь-якого поля блокує весь
документ.

Рекомендація: винести транзакції в підколекцію
`users/{uid}/profiles/{profileId}/transactions/{transactionId}`,
довідники/налаштування залишити окремим документом.

### Cloud Function погано масштабується

`notificationSweep` (щогодини) читає всі `push_tokens`, послідовно
проходить кожного користувача й кожен профіль, для кожного бюджету
перебирає всі транзакції заново. На малій кількості користувачів це
нормально, на тисячах — повільно й дорого. Невалідні FCM-токени після
помилок `registration-token-not-registered` не видаляються автоматично.

### Firestore rules перевіряють лише власника, не структуру — підтверджено

`firestore.rules` містить лише перевірку `request.auth.uid == uid`, без
обмежень на поля, типи, довжини рядків, кількість профілів, формат
`push_tokens`.

### Порожні `catch` — підтверджено

19 точних збігів патерну `catch(e){}` (і подібних з іменованим
параметром) — перевірено `grep`. Частина виправдана (localStorage,
Vibration API), частина може приховувати справжні помилки без
діагностики. Рекомендується централізований логер, що розрізняє
очікувані помилки користувача від реальних збоїв.

## Пріоритети виправлень

**Обов'язково перед публічним запуском:**
1. Об'єднати два service worker або розвести їхні scope.
2. Закрити self-XSS у `showToast()` та перевірити решту 94 місць з `innerHTML`.
3. Виправити timezone push-нагадувань (зберігати offset/timezone користувача).

6. Додати доступність модалок (`aria-modal`, focus trap, Escape) та `aria-label` на icon-only кнопках.

9. Збільшити шрифти й touch targets на мобільному.
10. Спростити вкладку фінансів.

**Можна пізніше:**
