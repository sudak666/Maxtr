# План: винести транзакції з `finance` у Firestore-підколекцію

Статус: **план + неруйнівний backfill-скрипт, не виконано проти продакшн-бази.**
Аудит-пункт 8. Причина відкладення виконання: немає доступу до реальних
Firestore-даних цього акаунту з цієї сесії для перевірки міграції, і 62
місця в `index.html` напряму читають/пишуть масив `transactions` в пам'яті
— повне перенесення застосунку на підколекцію є окремим, більшим фронтом
роботи, а не одноразовим скриптом.

## Чому це потрібно (нагадування з аудиту)

`finance` — один Firestore-документ, що містить масив `data` (усі
транзакції) поряд з десятками інших полів (гаманці, категорії, бюджети,
цілі, налаштування...). Ризики://
- ліміт Firestore 1 MiB на документ — при достатній історії транзакцій
  можна впертися в нього;
- кожне збереження (`fbSaveNow()`) переписує **весь** документ, тобто й усю
  історію транзакцій, навіть якщо змінився один гаманець;
- optimistic-concurrency конфлікт (`lastKnownUpdatedAt`) спрацьовує на
  рівні всього документа — правка бюджету на одному пристрої і додавання
  транзакції на іншому конфліктують, хоча стосуються різних полів;
- повільне первинне завантаження при великій історії (весь масив
  тягнеться одним запитом).

## Цільова схема

Замість нового окремого дерева колекцій (`users/{uid}/profiles/{id}/...`,
як запропоновано в оригінальному аудиті), яке б розійшлося з уже наявною
конвенцією `max_tracker`/`userDoc()`/`@<profileId>`-суфіксів — узгоджена
схема:

```
users/{uid}/max_tracker/{financeDocName}/transactions/{txId}
```

де `{financeDocName}` — те саме, що вже повертає `userDoc('finance')`
(`finance` для дефолтного профілю, `finance@<profileId>` для інших).
Це підколекція **під існуючим `finance`-документом**, не новий top-level
шлях — узгоджується з `firestore.rules`'s `match
/users/{uid}/max_tracker/{document}` без переосмислення всієї моделі
доступу; потрібно додати сусідній `match` для
`max_tracker/{document}/transactions/{txId}`.

`txId` = той самий `id` (число, `Date.now()`-based), що транзакція вже
має в масиві — зберігає стабільність id при переносі й дозволяє
ідемпотентний backfill (записати документ з тим самим id двічі — не
проблема).

## Стратегія міграції (неруйнівна, поетапна)

1. **Backfill (цей крок є, скрипт нижче)** — прочитати `finance.data` для
   кожного акаунту/профілю, записати кожну транзакцію як документ у
   `transactions` підколекції. **Не видаляти й не чіпати** оригінальний
   масив `data` — застосунок продовжує працювати на ньому, як і зараз.
   Backfill повністю паралельний до продакшн-роботи: нуль ризику для
   поточних користувачів, бо жоден існуючий код-шлях не читає нову
   підколекцію.
2. **Dual-write (окрема робота, не виконана)** — оновити всі точки запису
   транзакцій в `index.html` (додавання/редагування/видалення операції,
   recurring auto-add, import CSV тощо — orientovno 10-15 функцій із 62
   знайдених referenced-місць є фактичними мутаціями, решта — читання/
   фільтрація), щоб вони писали одночасно в масив (як зараз) і в
   підколекцію. Це найбільший шматок роботи — вимагає проходу по кожній
   функції, яка мутує `transactions`, і ретельного ручного/Playwright-
   тестування кожної.
3. **Dual-read verification (окрема робота)** — період, коли застосунок
   читає з обох джерел і звіряє розбіжності (логує через щойно доданий
   `logAppError`), перш ніж довіряти підколекції як джерелу правди.
4. **Cutover (окрема робота)** — перевести читання (рендер списку
   транзакцій, аналітика, бюджети, CSV-експорт, автоправила) на
   підколекцію (з пагінацією замість `.some()/.filter()` по всьому
   масиву), прибрати dual-write.
5. **Cleanup (окрема робота, значно пізніше)** — після впевненості:
   прибрати `data` з `finance`-документа.

Кроки 2-5 навмисно не виконані в цій сесії — кожен з них торкається
активного коду застосунку (не лише даних) і має тестуватись проти
реального використання, а не синтетичних прикладів.

## Rollback

Оскільки крок 1 (backfill) нічого не видаляє й не змінює в існуючій
моделі, rollback — просто видалити підколекцію `transactions` під кожним
`finance`-документом (`firebase firestore:delete` рекурсивно) — застосунок
про неї нічого не знає, поки не дійде до кроку 2.

## Backfill-скрипт

`scripts/backfill-transactions.mjs` (див. файл поруч) — Node-скрипт на
`firebase-admin`, призначений для одноразового ручного запуску власником
акаунту з реальними credentials (`firebase-tools`/service account),
**не автоматизований і не викликаний цією сесією**. Підтримує:
- `--dry-run` (за замовчуванням) — рахує й виводить, що буде записано,
  нічого не пише;
- `--uid=<uid>` — обмежити одним акаунтом для тестового прогону;
- перевірку після запису (рахує документи в підколекції == довжині
  масиву) і виводить розбіжності, не падаючи мовчки.

Запуск (коли є credentials):
```
cd scripts
npm install firebase-admin
GOOGLE_APPLICATION_CREDENTIALS=/path/to/service-account.json node backfill-transactions.mjs --dry-run
# перевірити вивід, потім:
GOOGLE_APPLICATION_CREDENTIALS=/path/to/service-account.json node backfill-transactions.mjs
```
