# Налаштування Firebase (обов'язково перед публікацією)

Код тепер використовує справжню автентифікацію (Firebase Auth) замість
захардкодженого PIN, і дані кожного користувача зберігаються окремо в
`users/{uid}/max_tracker/...`. Але це вимагає ручного налаштування в
Firebase Console — без цього реєстрація/вхід не запрацюють.

## 1. Увімкнути способи входу
Firebase Console → проєкт `maxtr-c238f` → **Authentication → Sign-in method**:
- Увімкнути **Email/Password**
- Увімкнути **Google** (вкажи publicly-visible name та support email)

## 2. Додати дозволені домени
**Authentication → Settings → Authorized domains** — додай домен(и), з яких
буде відкриватись застосунок (наприклад `твій-домен.web.app`, кастомний
домен, і `localhost` для тестів). Без цього `signInWithPopup` для Google
видаватиме помилку `auth/unauthorized-domain`.

**Важливо для цього репо:** `*.web.app`/`*.firebaseapp.com` додаються
автоматично, але **GitHub Pages-дзеркало (`sudak666.github.io`) — ні**.
Якщо вхід за номером телефону (або Google popup) не працює саме на
`https://sudak666.github.io/Maxtr/`, але працює на `https://maxtr-c238f.web.app`
— це воно: додай `sudak666.github.io` в Authorized domains вручну. Судячи з
того, що invisible reCAPTCHA на телефоні "блимає і зникає" одразу з
помилкою входу, це схоже саме на цей випадок — Firebase відхиляє домен
ще до того, як капча встигає щось показати.

## 3. Задеплоїти правила безпеки Firestore
У репозиторії є `firestore.rules` — вони обмежують доступ так, що
користувач бачить і редагує **тільки свої** дані:
```
firebase deploy --only firestore:rules
```
(потрібен `firebase-tools`, `firebase login`, `firebase use maxtr-c238f`).

**Це критично.** Поки правила не задеплоєні (або поки в консолі стоїть
"test mode" `allow read, write: if true`), дані будь-якого користувача
може прочитати/змінити хто завгодно — незалежно від того, що написано в
коді застосунку.

## 4. Стара спільна колекція
До реєстрації всі дані (зміни, фінанси, борги) лежали в одному документі
`max_tracker/*` без власника. Цей шлях уже перенесено в основний акаунт і
повністю закрито в правилах (`allow read, write: if false`) — окремих дій
тут більше не потрібно.

## 5. Вхід за номером телефону (опційно)
Firebase Console → **Authentication → Sign-in method** → увімкнути **Phone**.
Потрібен тариф **Blaze** (pay-as-you-go) — безкоштовний Spark не підтримує
відправку SMS взагалі, незалежно від коду застосунку.

**Помилка `auth/error-code:-39` (503 на `sendVerificationCode`) для всіх
номерів.** Підтверджено відповіддю Firebase Support (кейс 10414309, липень
2026): це не білінг, не API-ключ, не регіональна політика і не код
застосунку — це автоматичне блокування Firebase конкретного оператора
зв'язку/регіону через стабільно низький відсоток успішних SMS-доставок
(захист від toll fraud). Перевірені й виключені окремо: активний Blaze
білінг, дозволений регіон у SMS region policy, `Identity Toolkit API` без
обмежень на автоматично створеному API-ключі, кілька різних номерів з
однаковою помилкою, 0 надісланих SMS у Authentication → Usage без
попередження про квоту.

Рішення — увімкнути **reCAPTCHA SMS Defense** (Authentication → Settings →
reCAPTCHA → Manage reCAPTCHA → тумблер SMS Defense), зачекати 1–2 години на
синхронізацію backend-ключів, потім тримати в **Audit Mode** ~тиждень
(поріг довіри 0.8–0.9) перш ніж перемикати на **Enforce**. Ця фіча вимагає
**Web SDK 11+** — репозиторій оновлено до `12.11.0` саме для цього (шість
входжень: чотири `import` в `index.html`, два `importScripts` у
`sw.js`; шукай рядок з номером версії, якщо оновлюєш
SDK ще раз). Додатково Google рекомендує увімкнути Firebase App Check і
прибрати з Authorized domains усе тестове/невикористовуване.

## 6. Push-сповіщення (опційно, потребує коду з `functions/`)
Сповіщення (нагадування про облік, перевищення бюджету, наближення
повторюваного платежу) працюють локально й без цього кроку — але тільки
поки застосунок відкритий у браузері. Щоб вони доходили й коли застосунок
закрито, потрібен Firebase Cloud Messaging + одна Cloud Function:

1. **VAPID-ключ**: уже згенерований і захардкоджений у `index.html`
   (`const VAPID_KEY`) для проєкту `maxtr-c238f` — окремих дій тут не
   потрібно. Заміняти його треба лише якщо ключ ротується або якщо цей код
   деплоїться під **іншим** Firebase-проєктом: Firebase Console →
   **Project settings → Cloud Messaging → Web configuration → Web Push
   certificates → Generate key pair**, і вставити новий ключ замість
   поточного значення `const VAPID_KEY` в `index.html`.
2. **Задеплоїти функцію та оновлені правила Firestore**:
   ```
   cd functions && npm install && cd ..
   firebase deploy --only functions,firestore:rules
   ```
   Потрібен тариф **Blaze** (Cloud Functions на безкоштовному Spark не
   працюють).
3. Готово — кнопка "Push-сповіщення" в Налаштуваннях → Сповіщення тепер
   реєструє токен пристрою, а функція `notificationSweep`
   (`functions/index.js`) щогодини перевіряє всі три умови для кожного
   користувача й надсилає push через FCM.

## 7. Шаблони листів автентифікації (email-брендинг застаріле, потребує ручного фіксу)
**Знайдено 2026-07-13, ще не виправлено** — лист скидання пароля досі
підписаний старою назвою застосунку **"Xamss"** ("Reset your password for
Xamss" / "Your Xamss team"), хоча код і весь інший брендинг тепер
"Rytm" (перевірено грепом по всьому репо — жодного іншого сліду "Xamss"
немає, окрім історичних приміток у CLAUDE.md/CHANGELOG.md; "Zminka" теж
більше ніде немає — репо перейменовано на "Rytm" 2026-07-16). Причина: шаблони листів
автентифікації (скидання пароля, підтвердження email тощо) налаштовуються
в **Firebase Console → Authentication → Templates**, а не в коді цього
репозиторію — агент-сесія без доступу до Console UI не може це виправити
пул-реквестом. Потрібно вручну:
1. **Спочатку перевір/онови Public-facing name проєкту** — **не** в
   Authentication → Settings (це поле там більше не показується, підтверджено
   скріншотом реальної консолі 2026-07-18 — Firebase, вочевидь, переніс його).
   Правильний шлях: шестерня (⚙️) поруч з "Project Overview" у лівому
   верхньому куті → **Project settings** → вкладка **General** → прокрутити
   вниз до поля **Public-facing name** (той самий рядок також показується на
   екрані згоди Google OAuth). Стандартні
   шаблони листів автентифікації вставляють цю назву через плейсхолдер
   `%APP_NAME%`, а не як буквальний текст у кожному шаблоні окремо (див.
   [офіційну документацію](https://support.google.com/firebase/answer/7000714))
   — тобто "Xamss" тут майже напевно означає, що це поле просто не
   оновили під час попереднього перейменування на Zminka (а тепер і на
   Rytm), а не що хтось вручну вписав "Xamss" в текст листа. Зміни цього
   одного поля на "Rytm" зазвичай
   достатньо, щоб виправити **всі** стандартні шаблони одразу (Password
   reset, Email verification, Email change), включно з тими, які ще
   ніколи не надсилались і тому не були протестовані вручну.
2. **Тільки якщо після цього десь усе ще лишається "Xamss"/"Zminka"** — значить
   конкретний шаблон був колись відредагований вручну з буквальним
   текстом замість `%APP_NAME%`: Firebase Console → **Authentication →
   Templates**, відкрити той шаблон і замінити "Xamss"/"Xamss team"/"Zminka" на
   "Rytm"/"Команда Rytm" (і за бажанням додати переклад українською —
   за замовчуванням шаблони англійською).
3. **Окрема, менш пріоритетна примітка**: лист прийшов у Спам у реальному
   тесті (Gmail) — типова поведінка для дефолтного відправника
   `noreply@<project>.firebaseapp.com` без власного домену/SPF/DKIM.
   Виправляється підключенням **кастомного домену для відправки** у тому ж
   розділі Templates (Firebase Console дає покрокову інструкцію з DNS-
   записами) — окрема робота, не критична, поки листи хоч десь доходять.

## 8. Firebase App Check — уже налаштовано, лишається одна ручна дія

**Зроблено повністю з коду в audit-followup сесії** (детально —
`CLAUDE.md`'s "Firebase App Check" секція): reCAPTCHA Enterprise ключ
створено, зареєстровано в App Check для web-застосунку, клієнтський код
(`js/core.js`) уже викликає `initializeAppCheck()` при кожному завантаженні
сторінки. Нічого встановлювати вручну не потрібно.

**Свідомо залишено в режимі "не примусово" (`UNENFORCED`)** — Firestore і
Auth зараз приймають запити з App Check-токеном і без нього однаково, як і
раніше. Це навмисний, безпечний перший крок (офіційно рекомендований
Google порядок розкатки), а не недогляд.

**Коли будеш готовий увімкнути реальний захист** (після того, як
попрацюєш кілька днів і подивишся, що метрики токенів виглядають
здоровими):
1. Firebase Console → **App Check** → перевір вкладку **Metrics** для
   Firestore і Authentication — переконайся, що переважна більшість
   запитів приходить із дійсним токеном (не "unknown"/невалідні).
2. Той самий екран → для кожного сервісу (Firestore, Authentication) →
   перемкнути **Enforce**.
3. Після цього кроку запит без дійсного App Check-токена (наприклад, з
   домену поза списком дозволених — GitHub Pages-дзеркало вже в списку,
   локальний `localhost` для розробки — ні) почне реально відхилятись.
   Якщо колись знадобиться тестувати локально під enforced-режимом —
   Firebase підтримує debug-токени (`self.FIREBASE_APPCHECK_DEBUG_TOKEN`),
   ще не налаштовано, бо поки не потрібно.

## 9. Публікація в Google Play — покроковий чекліст (audit-followup сесія)

Все технічно готове з боку коду й інфраструктури (пакет, keystore,
assetlinks — див. CLAUDE.md's "Android / Google Play packaging"). Це не
може зробити агент-сесія: реєстрація Play Console вимагає реального
платежу й верифікації особи власника акаунта, а сам APK/AAB-білд
блокується мережевою політикою пісочниці (теж задокументовано в
CLAUDE.md). Нижче — точна послідовність кроків для власника, від оплати
до публікації.

**1. Створити акаунт розробника** — [play.google.com/console](https://play.google.com/console),
разовий платіж $25, верифікація особи (документ + іноді відео-дзвінок;
Google попереджає, що це може зайняти до 48 годин, іноді довше).

**2. Створити застосунок у консолі** — назва "Rytm", основна мова
українська (uk-UA), тип "Застосунок" (не гра), безкоштовний.

**3. Заповнити розділ "Вміст застосунку" (App content)** — обов'язковий
блок перед будь-яким релізом:
   - **Політика конфіденційності**: `https://maxtr-c238f.web.app/privacy.html`
     (файл уже є в репо й задеплоєний, нічого готувати не треба).
   - **Доступ до застосунку**: застосунок вимагає входу — познач це і
     вкажи тестові облікові дані для рев'юера Google (можна створити
     окремий тестовий email+пароль прямо в застосунку для цієї мети).
   - **Реклама**: немає (якщо це не зміниться).
   - **Анкета вікового рейтингу (IARC)**: заповнити чесно за фактичним
     вмістом — особистий фінансовий трекер без насильства/азартних
     ігор/чатів з іншими користувачами зазвичай отримує найнижчу
     категорію.
   - **Розділ "Безпека даних" (Data safety)** — задекларувати чесно, що
     збирається: email, номер телефону (опційно, якщо користувач додає),
     фінансові дані введені користувачем (операції/гаманці/борги) —
     зберігаються в Firebase (Google Cloud), передача шифрована (HTTPS).
     **Плюс, який уже готовий**: Play вимагає, щоб застосунки, які
     збирають персональні дані, давали можливість видалити акаунт і дані
     прямо в застосунку — це вже реалізовано (Налаштування → Акаунт →
     Видалити акаунт, `deleteAccountUser()` в `js/auth.js`), тож у цьому
     пункті анкети чесно відповідай "так, є" й посилайся на цей екран.

**4. Сторінка застосунку (Store listing)** — назва, короткий і повний
опис (українською; англійською — опційно), скріншоти з телефону
(мінімум 2, краще 4–8 — зроби напряму з `maxtr-c238f.web.app` в Chrome
DevTools mobile-emulation або з реального телефону), feature graphic
1024×500, іконка 512×512 (`icon-512.png` уже є в репо, підходить напряму),
категорія — "Фінанси" або "Продуктивність".

**5. Зібрати AAB і залити на "Внутрішнє тестування"** — точний рецепт уже
задокументований у CLAUDE.md ("Rebuilding the Android package from
scratch"): pwabuilder.com → `https://maxtr-c238f.web.app` → Package For
Stores → Google Play → Package ID `ua.rytm.app` → Icon URL
`android-icon-rounded.png` / Maskable icon URL `icon-512.png` → Signing
key → "Mine" → завантажити наявний `.jks`/`.p12` і **один** пароль (той
самий для store й key — див. PKCS12-примітку в CLAUDE.md) → Download
Package. Отриманий `.aab` залити в Play Console на трек **Internal
testing** (не одразу в Production).

**6. Перевірити на внутрішньому тестуванні** — додай свій Google-акаунт як
тестера, встанови застосунок за opt-in посиланням, переконайся, що TWA
відкривається **без адресного рядка браузера** (це підтверджує, що
`.well-known/assetlinks.json`'s fingerprint збігається з підписом AAB —
якщо рядок браузера все ж з'являється, дивись CLAUDE.md's assetlinks-
примітку).

**7. Промоутнути в Production** — після успішного внутрішнього тесту.
Перший реальний реліз Google перевіряє вручну — це може зайняти від
кількох годин до кількох днів, на відміну від наступних оновлень
(зазвичай швидше).

**8. Після публікації** — почни звіряти Firebase Console → **App Check →
Metrics** (розділ 8 вище) уже на реальному Android-трафіку з Play, перш
ніж вмикати `Enforce` для Firestore/Auth.
