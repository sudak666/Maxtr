## Project overview

**Zminka** (formerly "Xamss") is a static PWA: a personal tracker for work shifts, multi-currency finances, and debt/settlement schedules. It's Ukrainian-first (`uk`/`en` toggle), backed by Firebase (Auth + Firestore + Cloud Messaging + one scheduled Cloud Function), and deployed as static files via Firebase Hosting — there is no build step, bundler, or frontend framework; native browser ES modules do the file splitting instead. `index.html` holds the markup/CSS and loads four small classic (non-module) scripts plus the module graph; almost all app logic lives under `js/*.js`, loaded via `<script type="module" src="./js/app.js">` (see "`js/` module layout" below). It used to be one 7000-line inline `<script type="module">` — split apart specifically so a change to (say) the debt tab doesn't require reading through finance/calendar/settings code to get there; see that section for why this was safe to do without a bundler and what to watch for when editing across files.

## Architecture

### Layout
- `index.html` — markup, CSS, and four `<script src="...">` tags loading small classic (non-module) scripts from `js/`; loads `js/app.js` as the app's actual logic (see "`index.html` script structure" below).
- `js/` — the app's logic, split into 16 native ES modules (no bundler — the browser resolves the relative `import`s directly). See "`js/` module layout" below for the file list, the shared-state pattern, and gotchas for anyone splitting a file further or adding a new cross-file call.
- `functions/` — a separate Node 20 npm project (Cloud Functions v2 / `firebase-admin` + `firebase-functions`), deployed independently of the static site.
- `firebase.json` / `firestore.rules` / `manifest.json` — Firebase Hosting, Firestore security rules, and PWA manifest.
- `sw.js` — the single service worker, handling **both** PWA asset caching (network-first for HTML, cache-first for everything else) **and** Firebase Cloud Messaging background push display. It used to be split into `sw.js` (caching) + a separate `firebase-messaging-sw.js` (push), both registered at the site's root scope — but a single scope can only ever have one active service worker, so registering `firebase-messaging-sw.js` (which had no `fetch` handler) after `sw.js` silently replaced `sw.js` at that scope and disabled offline caching the moment a user enabled push. Merged into `sw.js` alone (bumped to v4) to remove the conflict; there is no `firebase-messaging-sw.js` file anymore — don't recreate it as a separate registration. `sw.js`'s `FIREBASE_SDK_ASSETS` also precaches (and cache-first-serves) the exact four gstatic SDK URLs `index.html`'s `<script type="module">` statically imports — a static ES module import is all-or-nothing, so without this, a cold start with no network (even with `index.html` itself already cached) would fail the whole module and leave the app inert past its HTML shell. This is a narrow exact-URL allowlist inside the fetch handler's normal cross-origin bail-out, not a general override — if the pinned SDK version in `index.html`'s imports ever changes, update `FIREBASE_SDK_ASSETS` in `sw.js` to match (and bump `CACHE_NAME`). `STATIC_ASSETS` also explicitly lists every file under `js/` for the same reason (same-origin this time, so no allowlist trick needed — they're precached alongside `index.html` itself) — add any new `js/*.js` file to this list too, or a cold start offline can fail to load it.
- `privacy.html` / `terms.html` — standalone static pages, not part of the SPA.
- `SETUP.md` — the manual, non-code Firebase Console setup required before auth/push/phone-sign-in work in production (enabling sign-in providers, authorized domains, deploying Firestore rules, VAPID key, required GCP APIs for the scheduled function). Keep this in sync whenever a change adds a new external dependency.

### `index.html` script structure
Five `<script>` tags, in load order — **all classic (non-module) scripts are now external files under `js/`, not inline** (moved there in the CSP-hardening pass, see "CSP" under Known gaps below; `sw.js`'s `STATIC_ASSETS` precaches all four same as every `js/*.js` module):
1. `js/theme-preinit.js` — applies the saved theme before first paint (must stay first, must stay a blocking classic script, not `defer`/`async`/`type=module`).
2. `js/touch-active-fix.js` — registers a no-op `touchstart` listener; iOS Safari only applies `:active` CSS states to a tap when at least one touch listener exists somewhere in the document, or every `:active`-based press animation (bottom nav, buttons, FAB) silently never shows on iPhone.
3. `js/sw-register.js` — registers `sw.js` on `window load`.
4. `js/classic-globals.js` — the custom monochrome SVG icon library (`ICON_PATHS`, `window.Icon(name)`, `setIcon(id, name)`) plus the I18N object/`window.tr()`/`window.translateStaticDOM()`/`window.setLang()`/`window.setTheme()`, theme/lang toggle wiring, and (added in the CSP pass) a small delegated-click listener for `.theme-opt`/`.lang-opt`/`.modal-card`/`.dlg-card`/`#profile-avatar-preview` that used to be inline `onclick=""` attributes. Runs before the module script. **This is genuinely outside the `js/` module graph** despite living in the same directory as the 16 ES modules — it's a classic script with immediate top-level execution, no `import`/`export`, no `__init_X__()` wrapper. `tr`/`Icon`/`setIcon`/`translateStaticDOM` are real globals (via `window`), not something any `js/*.js` ES module imports; `window.__applyLangDynamic` is the one hook back the other way (`js/core.js` assigns it so `setLang()` can re-render dynamic content after a language switch — grep for it before assuming these scripts don't talk to each other at all).
5. `<script type="module" src="./js/app.js">` — loads the whole `js/` module graph (see below). Functions called from inline `onclick`/`onchange` HTML attributes must be attached to `window` explicitly (`window.foo = function(){...}`) since module-scope declarations aren't global — this is why you'll see both a bare `function foo(){}`/`const foo = function(){}` and a `window.foo=foo;` line for anything wired to markup. **This bit us for real**: a function only called from an inline `onclick` attribute (not from other JS) must be `window.foo`, even a "just renders a list" helper — `renderProfilesUI` was defined as a bare module-scope `function` and worked fine everywhere it was called from other JS, but silently threw `ReferenceError` every time the one inline `onclick="...; renderProfilesUI()"` fired, until it was also assigned to `window`. **As of the CSP-hardening pass, this is now the exception, not the rule** — `index.html`/`js/*.js` templates have essentially zero inline `onclick=""` left (see the window.*/onclick-removal audit item's closing entry); this paragraph's advice only still applies to the ~10 deliberately-kept cross-file `window.*` exports documented there.

### `js/` module layout
`index.html` used to be one 7000-line inline `<script type="module">`; it's now 16 files under `js/`, split along the same `// ── SECTION ──` banners that organized it before (`state.js`/`core.js`/`firebase-sync.js`/`color-picker.js`/`auth.js`/`app-init.js`/`ui-widgets.js`/`calendar.js`/`settings-managers.js`/`goals-profile.js`/`notifications.js`/`finance.js`/`analytics-csv.js`/`debt.js`/`shopping.js`, plus `app.js` as the entry point). This was a mechanical, AST-verified transformation (parsed with `espree` + `eslint-scope`, not manually traced), not a rewrite — the actual code inside each function is unchanged. Three things make cross-file JS in this app different from a typical ES-module codebase, and matter for anyone adding a new cross-file reference or splitting a file further:

1. **All mutable app state lives in one shared object, `AppState` (`js/state.js`)** — `transactions`, `wallets`, `shifts`, `currentUser`, `activeProfileId`, all ~60 of what used to be individual top-level `let` bindings, are now `AppState.transactions`, `AppState.wallets`, etc. **Why one object instead of `export let x` per field**: ES modules only let the module that declared a binding reassign it, but many of these fields are reassigned from several different files — e.g. `fbLoadNow()` (`js/firebase-sync.js`) resets almost all of them, `switchProfile()`/`fbLoadNow()` write `AppState.transactions` from one file while `addTransaction()`/`deleteTransaction()` (`js/finance.js`) write it from another. A plain property write (`AppState.x = value`) has no such restriction; `export let x` per field would. **`js/state.js` has zero imports of its own, deliberately** — it's the one file every other file depends on, so if it depended on anything back, that circular edge risks a temporal-dead-zone `ReferenceError` on its own top-level `const AppState = {...}` initialization (this happened during development: `AppState`'s `widgetOrder`/`MONTHS`/`WEEKDAYS` fields needed `WIDGET_ORDER_DEFAULT`/`LANG_CALENDAR`, which briefly lived in `core.js` — which itself transitively imports `AppState` — and that circular edge threw). Both constants were moved into `state.js` itself instead. **If you add a new module-level mutable field, put it in `AppState`, not a fresh top-level `let`** — a bare new `let` works fine within its own file but silently can't be reassigned from anywhere else without going through this same problem.
2. **Every chunk that does something immediately at the top level (not inside a function) has that code wrapped in an exported `__init_<chunk>__()` function, called from `app.js` after all 16 files have finished importing.** The original single file ran strictly top-to-bottom, so e.g. `const auth = getAuth(fbApp)` (near the top) had always already run by the time a much later `onAuthStateChanged(auth, ...)` line executed. Once split across files with a real (partly circular) import graph, module evaluation order is not guaranteed to match the original file's order — an unwrapped top-level statement in one file can run before another file's declaration it depends on has executed, which is a temporal-dead-zone crash (hit during development with exactly this `auth`/`onAuthStateChanged` pair). **If you add new top-level side-effecting code to one of these files** (a `document.addEventListener(...)` call, a `setInterval(...)`, anything that isn't a `function`/`class`/`const`/`let` declaration), put it inside that file's `__init_X__()` function, not bare at the top level, or it risks running before some other file it depends on has initialized.
3. **A name only ever exposed as `window.foo = function(){...}` (no separate bare `function foo(){}`) still needs a real export/import if another file references the bare identifier `foo`** — this looks like it should be safe to skip (relying on `window` being shared across every script on the page) and an earlier version of this split tried exactly that, but it breaks for the specific case of the `window.foo = foo;` assignment statement itself: that line needs `foo` to already be a correctly-resolving lexical binding in the file it's written in, and "resolves via `window` later" doesn't help the statement that's supposed to be *setting* `window.foo` in the first place. Every `window.foo = function(){...}`-only name was rewritten to a real `const foo = function(){...}; window.foo = foo;` pair during the split for exactly this reason (except `closeManagers`, which already had a separate same-named bare wrapper function with different code — see that function if you need the exact shape). **When adding a new function only meant to be called from inline HTML (`onclick=`), still give it a real top-level declaration** (`function foo(){}` or `const foo = function(){}`) in whichever file it lives in, plus the `window.foo = foo;` line — don't write `window.foo = function(){...}` as the only place `foo` is defined, since any other `js/*.js` file that later needs to call it by name (not just via `onclick=`) will need to `import` that declaration, and there won't be one to import.

Verification for this split (documented for whoever next needs to add a 17th file or further split an existing one): `tests/smoke.mjs` + a manual Playwright pass exercising a transaction add, opening a settings modal, a language switch, and a shift-detail modal, all with zero console/page errors — see git history around the module-split commit(s) for the exact verification script if you need to reproduce it. The AST-analysis + code-generation scripts used to produce this split were one-off tooling (not committed — they read a since-deleted single-file snapshot of the module script) rather than a repo-maintained codegen step; treat `js/*.js` as hand-edited source from here on, the same as `index.html` always was.

### Firebase data model
- Firestore layout: `users/{uid}/max_tracker/{doc}` where `{doc}` is one of `shifts`, `finance`, `debt`, `backup_v2` (a one-time raw snapshot taken whenever config gets re-seeded), `profiles_meta` (see Multiple profiles below). `DCOL = 'max_tracker'`; `userDoc(name)` builds the path. **The Firestore rule for this collection is no longer a full wildcard** — `firestore.rules` matches a single path segment (`match /users/{uid}/max_tracker/{document}`, not `{document=**}`, since there are no subcollections here) and requires `document` to match `^(shifts|finance|debt|backup_v2)(@[A-Za-z0-9_]+)?$|^profiles_meta$`, plus shallow validation (top-level key count, `updatedAt`/`at` must be numbers if present — see the rules file's own header comment for why this stops short of a full per-field schema). **Adding a new doc *base name* now needs a matching `firestore.rules` change**, unlike before; a new `@<profileId>`-suffixed variant of an existing base name needs none. Before touching this file, run `tests/firestore-rules.mjs` (see Commands above) — it asserts against a real emulator and has already caught a rules bug (duplicate `allow write` blocks silently canceling validation via OR semantics) that a read-through missed. A legacy pre-auth shared `max_tracker/*` collection is explicitly denied in `firestore.rules` — do not resurrect it.
- The `finance` doc is the big one: transactions, wallets, categories, budgets, subcategories, currencyRates, tags, autoRules, recurring, `shoppingList`, goals, profile, subscription, widgets, **and `notifSettings`** (synced here specifically so the server-side Cloud Function can see each user's notification preferences — it used to be localStorage-only; now also carries `notifSettings.timeZone`, an IANA zone name captured client-side for the push sweep's local-time calculations).
- `push_tokens/{uid}` is a separate top-level collection (not nested under `users/{uid}`) holding `{token, updatedAt, sentDaily, sentBudget, sentRecurring, profileState}` for the FCM push sweep; it has its own Firestore rule scoped to `request.auth.uid == uid`, plus type/size checks on the client-writable `token`/`updatedAt` fields (not a full key allowlist, since the Cloud Function's Admin SDK writes its own dedup fields onto the same doc, bypassing rules entirely).
- Client-side caching/sync: every piece of state is mirrored into `localStorage` under per-user keys via `lsKey(name)` → `mx_<name>_<uid>` (loaded first for instant paint), then reconciled against Firestore in `fbLoadNow()`. Writes go through `scheduleSave()` (debounced) → `fbSaveNow()`, which uses optimistic concurrency (`lastKnownUpdatedAt` per doc, compared against each doc's `updatedAt`) and prompts the user on conflict rather than silently overwriting another device's newer save. A few internal localStorage keys still carry the old `xamss*` prefix from before the Zminka rename (e.g. `xamssTheme`, `xamssLang`) — these were left as-is intentionally since renaming them would silently reset existing users' local prefs for zero benefit.

### Multiple profiles per account
One Firebase Auth account can hold several independent data-profiles (e.g. "Я"/"Дружина") with their own finances/shifts/debts, switchable in Settings → Профілі.
- `activeProfileId` (module-level, default `'default'`) drives both `userDoc(name)` and `lsKey(name)`: for the **default** profile these resolve to the original unsuffixed doc names/localStorage keys (zero migration, existing users are untouched); for any other profile, `userDoc` suffixes the doc name `"@<profileId>"` (e.g. `finance@abc123`) and `lsKey` suffixes the localStorage key the same way. Because every save/load path already funnels through these two functions, nothing else needed to change to make the whole app profile-aware.
- `profiles_meta` (account-wide, never suffixed even when a non-default profile is active) holds `{list:[{id, name, avatar, createdAt}], updatedAt}` — just labels/avatars for the switcher, not real data. Each profile's actual identity (nickname + avatar shown in Settings → Профіль and the topbar) lives inside *that profile's own* `finance` doc, since `profile.nickname`/`profile.avatar` is per-profile automatically once `finance` is suffixed.
- `activeProfileId` itself is a **per-device** choice (`localStorage['mx_activeProfile_'+uid]`), not synced — switching profiles on one device doesn't affect others.
- `switchProfile(id)` flushes the current profile (`fbSaveNow()`) before reassigning `activeProfileId` and calling `fbLoadNow()` again — this reuses all the existing seeding/rendering logic verbatim, so a brand-new profile gets the same default wallets/categories a brand-new account gets.
- The scheduled Cloud Function (`functions/index.js` `notificationSweep`) reads `profiles_meta` and sweeps every profile on the account (default + suffixed), not just the default one — each profile's `finance`/`finance@<id>` doc is checked independently. Per-profile dedup state lives in `push_tokens/{uid}.profileState.<profileId>` (`{sentDaily, sentBudget, sentRecurring}`); pre-existing token docs' flat `sentDaily`/`sentBudget`/`sentRecurring` fields are read as the default profile's initial state until the sweep writes `profileState.default` for the first time, so old single-profile users need no migration. Local in-app notifications work fine for whichever profile is active since they read the same in-memory state that switching already repopulates.
- Any render function that bails out early when its collection is empty (`if(!items.length) return;`) needs to explicitly clear its previously-rendered DOM first — otherwise switching to a profile with no data leaves the *previous* profile's numbers on screen instead of blanking them (this bit us once with the Debt tab; check for the same pattern before adding another profile-scoped collection).

### Auth
Three sign-in methods, all in the AUTH / PHONE SIGN-IN sections: email+password, Google popup, and phone number (SMS code via `signInWithPhoneNumber` + an invisible `RecaptchaVerifier`). Layered on top, independent of Firebase Auth: a local PIN (SHA-256 hash in localStorage, `mx_pin_<uid>`) and optional WebAuthn biometric unlock — these gate the UI locally but never replace the Firebase session.

Settings → Акаунт additionally lets an already-signed-in user (e.g. one that signed in via Google) **link/unlink a phone number** as an extra credential (`linkWithPhoneNumber(currentUser, ...)` / `unlink(currentUser,'phone')`) — a separate flow from the pre-auth sign-in one, with its own `link-recaptcha-container`/`RecaptchaVerifier` instance so it doesn't collide with the lock-screen's.

**reCAPTCHA retry gotcha**: `RecaptchaVerifier.clear()` can itself throw partway through on a failed attempt, leaving the widget's DOM node behind; the next attempt's fresh `RecaptchaVerifier.render()` then finds a stale widget already in the container and throws `"reCAPTCHA has already been rendered in this element"`. Both phone flows now force the container's `innerHTML` empty on error (`resetRecaptchaContainer(containerId)`) regardless of whether `.clear()` succeeded — do this for any new reCAPTCHA-backed retry flow too.

### UI conventions worth following
- Custom-styled `<select>`/`<input type=date>`: any new one just needs `enhanceSelect()` / `enhanceDateInput()` called on it (or add it before `enhanceAllSelects()` runs) — don't hand-roll another dropdown.
- **Never use `<input type=color>`** — it opens the browser's native, unstylable OS color-picker popup, which looks jarringly out of place against the app's dark custom UI. Use the shared `openColorPicker(kind, id)` / `color-pick-modal` pattern instead (a `PALETTE`-swatch grid modal, same idea as the avatar picker) — wire a new color-editable list (wallets/shiftTypes/tags all already do this) by adding a case to `selectPickedColor()` and rendering a plain `<button class="mgr-color" style="background:${color}" onclick="openColorPicker('kind','${id}')">` instead of a color input.
- Similarly, don't rely on native browser affordances that can't be restyled or that read as "not part of the app" — e.g. no `title="..."` tooltips (native OS tooltip box); prefer a visible label/subtitle in the UI itself, like the rest of the settings rows do.
- "Settings manager" modals (wallets, categories, budgets, goals, recurring, tags, rules, rates, widgets, pin, premium, profiles, link-phone, avatar/color pickers) all follow the same pattern: a `.modal-overlay` + `.modal-card`, an `open*Manager()`-style function that renders a list into it and sets `display:flex`, and the shared `closeManagers()` that hides all of them by id — add new manager ids to that shared close-list (an easy miss: forgetting this leaves a modal that won't close on backdrop-click).
- All icons are the custom monochrome SVG set (`window.Icon(name)` / `setIcon(id, name)`) — no emoji, no icon fonts. Add new glyphs to `ICON_PATHS` rather than reaching for an emoji. **Check for duplicates within the same visible list/row-group** before reusing an existing glyph — two *different* features sitting next to each other (e.g. two quick-action buttons, two settings-card icons in the same screen) with the same icon reads as a bug to users even though it isn't one; reusing an icon for the *same* feature across contexts (a tab + its settings entry + its quick-action) is fine and intentional. Generic action icons (`plus` for add, `trash` for delete, `pencil` for edit, `chevron`, `refresh`) are meant to repeat everywhere — that's a consistent affordance, not a duplicate.
- i18n: a single `I18N = {uk:{...}, en:{...}}` object plus `tr(key)`; static markup uses `data-i18n`/`data-i18n-placeholder` attributes applied by `translateStaticDOM()`. There is no `data-i18n-title` — dynamic titles/tooltips must be set in JS.
- The bottom nav (`.tabs`) currently has 5 tabs (Фінанси/Графік змін/Розрахунки/Покупки/Налаштування) — `.tab-icon-wrap` is 42px (38px under the ≤600px breakpoint). Adding a 6th tab means re-checking these still fit comfortably; there's no overflow/scroll handling on that bar.
- `sw.js`'s `fetch` handler must bail out (`return` without calling `respondWith`) for any cross-origin request and let the browser handle it natively — it used to re-issue *every* request including third-party ones through `respondWith(fetch(event.request))`, which throws for some no-cors/redirect combinations (this broke reCAPTCHA's own CSP-report beacon with a console `TypeError: Failed to fetch`). Don't reintroduce a `respondWith` path for anything outside `self.location.origin`.

The patterns below are now the established convention — follow them for anything new rather than reverting to the older per-screen styles they replaced.
- **Hero metric + chip row** (`.hero-metric`/`.hero-balance` + `.chip-stat-row`/`.chip-stat`): replaces "3-4 equal-weight stat cards in a grid" (the old `.stats-row` of `.stat-card`s). ### Android / Google Play packaging (TWA, July 2026)
The app ships to Google Play as a **Trusted Web Activity (TWA)** — a thin Android wrapper that opens the live `maxtr-c238f.web.app` site full-screen, not a separate native codebase. There is no local Android project checked into this repo; it's generated on demand.
- **Canonical domain for the Android package is `maxtr-c238f.web.app`**, not the GitHub Pages mirror — chosen because Play's Digital Asset Links verification needs one fixed origin, and the Firebase domain has no subpath (GitHub Pages serves under `/Maxtr/`, which would complicate `start_url`/asset-link matching).
- **`.well-known/assetlinks.json`** (repo root) is what lets Android show the app without a browser URL bar — it declares the app's package ID + the signing certificate's SHA-256 fingerprint, and Google Play/Android verifies it against `https://maxtr-c238f.web.app/.well-known/assetlinks.json`. `firebase.json`'s hosting `ignore` list originally excluded all dotfiles (`**/.*`), which would have silently dropped this file from every deploy — it now has an explicit `"!/.well-known/**"` un-ignore plus a `Content-Type: application/json` header override for that path. If this file's fingerprint and the actual signing key ever drift apart (e.g. the key gets regenerated), the app degrades to showing a Chrome URL bar instead of failing outright — annoying but not broken, and the fix is always "update the fingerprint in this file, redeploy hosting."
- **Package ID is `ua.zminka.app`** — currently a placeholder (app name wasn't finalized when this was set up). It's freely changeable in whatever tool generates the package (twa-manifest.json / PWABuilder's Package ID field) up until the *first* real Play Store upload; after that, changing it means publishing as a brand-new listing, not an update.
- **The actual APK/AAB build cannot happen from a Claude Code cloud sandbox.** Confirmed by hitting explicit 403 policy denials from this environment's egress proxy on `dl.google.com` (Android SDK/build-tools — blocks even the offline `bubblewrap build` path), `pwabuilder.com`, and even `maxtr-c238f.web.app` itself. Only `github.com`/npm/pypi-class hosts and Google's OAuth/Firebase *API* endpoints (as opposed to the Hosting *content* domain) were reachable. What **did** work fully offline from the sandbox: generating the Android Studio project skeleton via `@bubblewrap/core`'s `TwaManifest`/`TwaGenerator` classes directly (bypassing the CLI's interactive prompts, feeding it the local `manifest.json` instead of fetching it live, and serving the icon files over a `127.0.0.1` static server so `TwaGenerator` had something reachable to pull them from), and generating the release signing keystore with the locally-installed JDK's `keytool` (no network needed at all for that part). The actual compile step then has to happen either in Android Studio on a real machine, or — the path actually used — **pwabuilder.com** from the user's own browser, pointed at `https://maxtr-c238f.web.app`.
- **Signing keystore is intentionally not committed to this repo** — it was generated once (`keytool -genkeypair`, RSA/3072, PKCS12 format) and handed directly to the account owner as a downloadable file; losing it means never being able to publish an update under `ua.zminka.app` again. If it ever needs regenerating, `.well-known/assetlinks.json`'s fingerprint has to be updated and hosting redeployed to match, or the "no browser URL bar" behavior breaks.
- **PKCS12 keystore gotcha, cost real debugging time**: `keytool` on JDK 9+ defaults to PKCS12-format keystores, which — unlike the older JKS format — **do not support a separate password for the private key entry**; `keytool -genkeypair -keypass X -storepass Y` with `X != Y` silently *succeeds* but prints `Warning: Different store and key passwords not supported for PKCS12 KeyStores. Ignoring user-specified -keypass value.` and just reuses the store password for the key too. Any consumer that's later told "the key password is X" (PWABuilder, `apksigner`, etc.) will then fail with `javax.crypto.BadPaddingException: Given final block not properly padded` — a decryption error, not an authentication-rejected error, which reads like file corruption rather than a wrong-password problem. **Always generate PKCS12 signing keys with the same value for both `-storepass` and `-keypass`**, and only document/hand off one password, not two.
- **`android-icon-rounded.png`** (repo root) is a squircle-rounded-corner variant of `icon-512.png`, generated with Pillow — used *only* by pasting its URL into PWABuilder's Android package form "Icon URL" field, not referenced anywhere in `manifest.json`. Reason: the TWA's launch splash screen centers the raw icon on the manifest's `background_color` with **no adaptive-icon masking applied** (unlike the home-screen launcher icon, which Android masks itself), so `icon-512.png`'s hard square corners looked jarring specifically on that one screen. The form's separate "**Maskable icon URL**" field must keep pointing at the original full-bleed `icon-512.png` — pre-rounding that one would break the safe-zone contract Android's own adaptive-icon masking expects.
- Rebuilding the Android package from scratch (e.g. after a manifest/icon change) is a manual, browser-based process: pwabuilder.com → enter `https://maxtr-c238f.web.app` → Package For Stores → Google Play → fill Package ID / Icon URL / Maskable icon URL → Signing key → "Mine" → upload the existing `.jks` and its one password → Download Package. There's no CI/automation for this yet.
- As of this session: Google Play Console developer account **not yet created** by the account owner ($25 one-time fee, manual step at play.google.com/console) — nothing has been uploaded to Play yet, only the hosting-side asset-link verification and a locally-tested `.apk` exist so far.

- **Cloud Function scaling pass, and a real production bug found only by testing against a real Firestore emulator** (PR [#117](https://github.com/sudak666/Maxtr/pull/117), merged (squash) as `79cc789`, deployed 2026-07-11): split `sweepProfile()`/`sweepToken()` out of `functions/index.js` into `functions/lib/sweep.js` (same reasoning as `lib/pure.js` — testable with no firebase-admin credentials or `node_modules` install), taking `db`/`sendPushFn` as explicit parameters. This made two things possible: (1) **parallelized the previously-sequential per-user Firestore reads** — `profiles_meta` and the default profile's finance doc now fetch concurrently (every account has at least the default profile regardless of what `profiles_meta` says), and any additional profiles' finance docs fetch in parallel with each other too — roughly halving per-user latency for the common single-profile case, which is what actually bounds how many users one `notificationSweep` invocation can get through before its own execution timeout as the account count grows; (2) added `tests/functions-sweep.mjs` (13 cases, fake db + fake sendPush, zero external dependencies, wired into CI's `smoke` job) covering all three notification types, dedup, invalid-token handling, and the parallel-fetch behavior itself (verified via timing — a two-read scenario with 100ms artificial delays on each must finish in well under 200ms, not ~200ms+). **The real find**: after the fake-based tests passed, ran the same refactored functions against a *real* Firestore emulator via the actual `firebase-admin` Admin SDK (one-off manual check this session, not new committed CI infra — `firebase emulators:start --only firestore`, confirmed reachable and working in this sandbox) with only `sendPush` faked out. This immediately threw `Value for argument "data" is not a valid Firestore document. Cannot use "undefined" as a Firestore value` — a bug the hand-rolled fake `db` had silently let through because it never validated writes the way real Firestore does. Root cause: `nextProfileState[profileId] = {...prevState, ...profileUpdates}` — for a brand-new `push_tokens` doc (or one that simply never triggered a budget/recurring alert before), `prevState`'s legacy-fallback fields (`sentDaily`/`sentBudget`/`sentRecurring` destructured straight off the token doc) include real `undefined`s for whichever notification types have never fired, and those explicit `undefined`s land in the final write payload — which the real Firestore Admin SDK rejects outright, for *any* document write, not just this one. **This bug predates this PR entirely** — the merge logic itself was unchanged by the refactor, it just had never been exercised against a real Firestore implementation before (previously verified only by code review + the AUDIT.md pass). Practical impact: this would fail on the very first successful notification for almost any account (since having all three dedup fields already populated before that point is the unlikely case), turning into a `sweepToken failed` log entry via the existing `mapWithConcurrency`/`Promise.allSettled` error handling — the push itself would still have sent (dedup-state write happens last), but the dedup state would never persist, risking a duplicate push on every subsequent hourly run for that account indefinitely. Fixed by stripping `undefined`-valued keys before the merge. **Also fixed while writing test cases** (not found via the emulator, just noticed by inspection): the daily-reminder hour check used `h || 21`, which silently treats an explicit midnight (`"00:00"`) reminder time as hour `0` → falsy → falls back to 21:00 instead of respecting the user's actual choice; switched to `Number.isFinite(h)`. **Lesson, worth repeating from the CSP-verification entry above but even more directly demonstrated here**: a hand-rolled fake for a database in a unit test proves your *logic* is right but not that your *data* is valid for the real store — for anything that writes to Firestore, a real (even emulated) Firestore run is the only check that catches serialization-level constraints a fake won't know to enforce.
