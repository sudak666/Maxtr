## Project overview

**Rytm** (formerly "Xamss", then "Zminka" — renamed to Rytm 2026-07-16, the account owner's final branding decision) is a static PWA: a personal tracker for work shifts, multi-currency finances, and debt/settlement schedules. It's Ukrainian-first (`uk`/`en` toggle), backed by Firebase (Auth + Firestore + Cloud Messaging + one scheduled Cloud Function), and deployed as static files via Firebase Hosting — there is no build step, bundler, or frontend framework; native browser ES modules do the file splitting instead. `index.html` holds the markup/CSS and loads four small classic (non-module) scripts plus the module graph; almost all app logic lives under `js/*.js`, loaded via `<script type="module" src="./js/app.js">` (see "`js/` module layout" below). It used to be one 7000-line inline `<script type="module">` — split apart specifically so a change to (say) the debt tab doesn't require reading through finance/calendar/settings code to get there; see that section for why this was safe to do without a bundler and what to watch for when editing across files.

## Architecture

### Layout
- `index.html` — markup, CSS, and four `<script src="...">` tags loading small classic (non-module) scripts from `js/`; loads `js/app.js` as the app's actual logic (see "`index.html` script structure" below).
- `js/` — the app's logic, split into 16 native ES modules (no bundler — the browser resolves the relative `import`s directly). See "`js/` module layout" below for the file list, the shared-state pattern, and gotchas for anyone splitting a file further or adding a new cross-file call.
- `functions/` — a separate Node 20 npm project (Cloud Functions v2 / `firebase-admin` + `firebase-functions`), deployed independently of the static site.
- `firebase.json` / `firestore.rules` / `manifest.json` — Firebase Hosting, Firestore security rules, and PWA manifest.
- `sw.js` — the single service worker, handling **both** PWA asset caching (network-first for HTML, cache-first for everything else) **and** Firebase Cloud Messaging background push display. It used to be split into `sw.js` (caching) + a separate `firebase-messaging-sw.js` (push), both registered at the site's root scope — but a single scope can only ever have one active service worker, so registering `firebase-messaging-sw.js` (which had no `fetch` handler) after `sw.js` silently replaced `sw.js` at that scope and disabled offline caching the moment a user enabled push. Merged into `sw.js` alone (bumped to v4) to remove the conflict; there is no `firebase-messaging-sw.js` file anymore — don't recreate it as a separate registration. `sw.js`'s `FIREBASE_SDK_ASSETS` also precaches (and cache-first-serves) the exact four gstatic SDK URLs `index.html`'s `<script type="module">` statically imports — a static ES module import is all-or-nothing, so without this, a cold start with no network (even with `index.html` itself already cached) would fail the whole module and leave the app inert past its HTML shell. This is a narrow exact-URL allowlist inside the fetch handler's normal cross-origin bail-out, not a general override — if the pinned SDK version in `index.html`'s imports ever changes, update `FIREBASE_SDK_ASSETS` in `sw.js` to match (and bump `CACHE_NAME`). `STATIC_ASSETS` also explicitly lists every file under `js/` for the same reason (same-origin this time, so no allowlist trick needed — they're precached alongside `index.html` itself) — add any new `js/*.js` file to this list too, or a cold start offline can fail to load it.
- `privacy.html` / `terms.html` — standalone static pages, not part of the SPA.
- `SETUP.md` — the manual, non-code Firebase Console setup required before auth/push/phone-sign-in work in production (enabling sign-in providers, authorized domains, deploying Firestore rules, VAPID key, required GCP APIs for the scheduled function). Keep this in sync whenever a change adds a new external dependency.

### `index.html` script structure
Five `<script>` tags, in load order — **all classic (non-module) scripts are now external files under `js/`, not inline** (moved there in the CSP-hardening pass, see the CSP entries in `CHANGELOG.md`; `sw.js`'s `STATIC_ASSETS` precaches all four same as every `js/*.js` module):
1. `js/theme-preinit.js` — applies the saved theme before first paint (must stay first, must stay a blocking classic script, not `defer`/`async`/`type=module`).
2. `js/touch-active-fix.js` — registers a no-op `touchstart` listener; iOS Safari only applies `:active` CSS states to a tap when at least one touch listener exists somewhere in the document, or every `:active`-based press animation (bottom nav, buttons, FAB) silently never shows on iPhone.
3. `js/sw-register.js` — registers `sw.js` on `window load`, **and drives reliable auto-update**: a `controllerchange` listener reloads the open page exactly once when a newly-deployed worker takes control (guarded against reload loops, and skipped on the first-ever install when there was no prior controller), plus a `reg.update()` poll on load and on tab-refocus. This exists because the app is a long-lived PWA/TWA users keep open without navigating: `sw.js` already uses `skipWaiting()`+`clients.claim()` so a new worker activates immediately, but the already-open page keeps executing the JS it loaded at first paint until it actually reloads — so without this, a user on an open session never picks up a deploy's new JS (this caused repeated "still broken" reports for bugs that were already fixed — their tab was running the previous version). **The transition onto this mechanism costs one manual reload/reopen** (the already-running old code can't retroactively gain the listener); every deploy after that lands automatically. Paired with `firebase.json` serving `sw.js` and `/js/**` as `Cache-Control: no-cache` (revalidate-before-use, not "never cache") so Firebase Hosting's default `max-age=3600` can't (a) delay the SW update check or (b) let a freshly-installed worker's `cache.addAll()` re-cache hour-stale JS from the browser HTTP cache. **When debugging a "my fix isn't working on the phone" report, suspect stale JS delivery first** — reproduce against the actual deployed behavior (or under the real CSP, see the CSP note below) before assuming the code is wrong; the local test suite serves fresh files every run and is blind to caching.
4. `js/classic-globals.js` — the custom monochrome SVG icon library (`ICON_PATHS`, `window.Icon(name)`, `setIcon(id, name)`) plus the I18N object/`window.tr()`/`window.translateStaticDOM()`/`window.setLang()`/`window.setTheme()`, theme/lang toggle wiring, and (added in the CSP pass) a small delegated-click listener for `.theme-opt`/`.lang-opt`/`.modal-card`/`.dlg-card`/`#profile-avatar-preview` that used to be inline `onclick=""` attributes. Runs before the module script. **This is genuinely outside the `js/` module graph** despite living in the same directory as the 16 ES modules — it's a classic script with immediate top-level execution, no `import`/`export`, no `__init_X__()` wrapper. `tr`/`Icon`/`setIcon`/`translateStaticDOM` are real globals (via `window`), not something any `js/*.js` ES module imports; `window.__applyLangDynamic` is the one hook back the other way (`js/core.js` assigns it so `setLang()` can re-render dynamic content after a language switch — grep for it before assuming these scripts don't talk to each other at all).
5. `<script type="module" src="./js/app.js">` — loads the whole `js/` module graph (see below). Functions called from inline `onclick`/`onchange` HTML attributes must be attached to `window` explicitly (`window.foo = function(){...}`) since module-scope declarations aren't global — this is why you'll see both a bare `function foo(){}`/`const foo = function(){}` and a `window.foo=foo;` line for anything wired to markup. **This bit us for real**: a function only called from an inline `onclick` attribute (not from other JS) must be `window.foo`, even a "just renders a list" helper — `renderProfilesUI` was defined as a bare module-scope `function` and worked fine everywhere it was called from other JS, but silently threw `ReferenceError` every time the one inline `onclick="...; renderProfilesUI()"` fired, until it was also assigned to `window`. **As of the CSP-hardening pass, this is now the exception, not the rule** — `index.html`/`js/*.js` templates have essentially zero inline `onclick=""` left (see the window.*/onclick-removal audit entries in `CHANGELOG.md`); this paragraph's advice only still applies to the ~10 deliberately-kept cross-file `window.*` exports documented there.

**A strict Content-Security-Policy IS deployed** (`firebase.json` hosting header): `script-src 'self' https://www.gstatic.com https://apis.google.com` — crucially **no `'unsafe-inline'` and no `'unsafe-eval'`**. This means **inline event-handler attributes (`onclick=""`/`onchange=""` baked into rendered HTML) SILENTLY DO NOT EXECUTE on the live site** (Firebase Hosting / the `.web.app` TWA), and `eval`/`new Function` are blocked too. It's dangerously invisible: the sandbox's `python3 -m http.server` sends no CSP, so inline handlers work in every local test, then no-op only in production. This shipped a real bug — the tx-list delete button (inline `onclick="event.stopPropagation();deleteTransaction()"`) no-op'd on the live site so the tap fell through to the row's click→edit listener ("delete opened edit"), and every `emptyStateHtml` "add first item" CTA was dead ("add doesn't work" on an empty list). All were converted to `data-action` (real `addEventListener` delegation, which CSP allows). **NEVER add an inline `onclick=""`/`onchange=""` to `index.html` or a `js/*.js` template string** — wire it as a `data-action` dispatched by the file's delegated capture-phase listener (or `el.onclick = fn` set in JS, which CSP does NOT block since it isn't an inline attribute), the same as `deleteTransaction`/`openColorPicker`/the empty-state CTAs now are. `tests/csp-handlers.mjs` (in CI) loads the page under the real firebase.json CSP and exercises these flows specifically so this can't regress silently again — the rest of the suite is blind to it (no CSP header locally). `el.onclick=`/`cell.onclick=`/`chip.onclick=` DOM-property assignments in JS are fine (not inline attributes); only attribute strings in HTML are blocked.

### `js/` module layout
`index.html` used to be one 7000-line inline `<script type="module">`; it's now 16 files under `js/`, split along the same `// ── SECTION ──` banners that organized it before (`state.js`/`core.js`/`firebase-sync.js`/`color-picker.js`/`auth.js`/`app-init.js`/`ui-widgets.js`/`calendar.js`/`settings-managers.js`/`goals-profile.js`/`notifications.js`/`finance.js`/`analytics-csv.js`/`debt.js`/`shopping.js`, plus `app.js` as the entry point). This was a mechanical, AST-verified transformation (parsed with `espree` + `eslint-scope`, not manually traced), not a rewrite — the actual code inside each function is unchanged. Three things make cross-file JS in this app different from a typical ES-module codebase, and matter for anyone adding a new cross-file reference or splitting a file further:

1. **All mutable app state lives in one shared object, `AppState` (`js/state.js`)** — `transactions`, `wallets`, `shifts`, `currentUser`, `activeProfileId`, all ~60 of what used to be individual top-level `let` bindings, are now `AppState.transactions`, `AppState.wallets`, etc. **Why one object instead of `export let x` per field**: ES modules only let the module that declared a binding reassign it, but many of these fields are reassigned from several different files — e.g. `fbLoadNow()` (`js/firebase-sync.js`) resets almost all of them, `switchProfile()`/`fbLoadNow()` write `AppState.transactions` from one file while `addTransaction()`/`deleteTransaction()` (`js/finance.js`) write it from another. A plain property write (`AppState.x = value`) has no such restriction; `export let x` per field would. **`js/state.js` has zero imports of its own, deliberately** — it's the one file every other file depends on, so if it depended on anything back, that circular edge risks a temporal-dead-zone `ReferenceError` on its own top-level `const AppState = {...}` initialization (this happened during development: `AppState`'s `widgetOrder`/`MONTHS`/`WEEKDAYS` fields needed `WIDGET_ORDER_DEFAULT`/`LANG_CALENDAR`, which briefly lived in `core.js` — which itself transitively imports `AppState` — and that circular edge threw). Both constants were moved into `state.js` itself instead. **If you add a new module-level mutable field, put it in `AppState`, not a fresh top-level `let`** — a bare new `let` works fine within its own file but silently can't be reassigned from anywhere else without going through this same problem.
2. **Every chunk that does something immediately at the top level (not inside a function) has that code wrapped in an exported `__init_<chunk>__()` function, called from `app.js` after all 16 files have finished importing.** The original single file ran strictly top-to-bottom, so e.g. `const auth = getAuth(fbApp)` (near the top) had always already run by the time a much later `onAuthStateChanged(auth, ...)` line executed. Once split across files with a real (partly circular) import graph, module evaluation order is not guaranteed to match the original file's order — an unwrapped top-level statement in one file can run before another file's declaration it depends on has executed, which is a temporal-dead-zone crash (hit during development with exactly this `auth`/`onAuthStateChanged` pair). **If you add new top-level side-effecting code to one of these files** (a `document.addEventListener(...)` call, a `setInterval(...)`, anything that isn't a `function`/`class`/`const`/`let` declaration), put it inside that file's `__init_X__()` function, not bare at the top level, or it risks running before some other file it depends on has initialized.
3. **A name only ever exposed as `window.foo = function(){...}` (no separate bare `function foo(){}`) still needs a real export/import if another file references the bare identifier `foo`** — this looks like it should be safe to skip (relying on `window` being shared across every script on the page) and an earlier version of this split tried exactly that, but it breaks for the specific case of the `window.foo = foo;` assignment statement itself: that line needs `foo` to already be a correctly-resolving lexical binding in the file it's written in, and "resolves via `window` later" doesn't help the statement that's supposed to be *setting* `window.foo` in the first place. Every `window.foo = function(){...}`-only name was rewritten to a real `const foo = function(){...}; window.foo = foo;` pair during the split for exactly this reason (except `closeManagers`, which already had a separate same-named bare wrapper function with different code — see that function if you need the exact shape). **When adding a new function only meant to be called from inline HTML (`onclick=`), still give it a real top-level declaration** (`function foo(){}` or `const foo = function(){}`) in whichever file it lives in, plus the `window.foo = foo;` line — don't write `window.foo = function(){...}` as the only place `foo` is defined, since any other `js/*.js` file that later needs to call it by name (not just via `onclick=`) will need to `import` that declaration, and there won't be one to import.

Verification for this split (documented for whoever next needs to add a 17th file or further split an existing one): `tests/smoke.mjs` + a manual Playwright pass exercising a transaction add, opening a settings modal, a language switch, and a shift-detail modal, all with zero console/page errors — see git history around the module-split commit(s) for the exact verification script if you need to reproduce it. The AST-analysis + code-generation scripts used to produce this split were one-off tooling (not committed — they read a since-deleted single-file snapshot of the module script) rather than a repo-maintained codegen step; treat `js/*.js` as hand-edited source from here on, the same as `index.html` always was.

### Firebase data model
- Firestore layout: `users/{uid}/max_tracker/{doc}` where `{doc}` is one of `shifts`, `finance`, `debt`, `backup_v2` (a one-time raw snapshot taken whenever config gets re-seeded), `profiles_meta` (see Multiple profiles below). `DCOL = 'max_tracker'`; `userDoc(name)` builds the path. **The Firestore rule for this collection is no longer a full wildcard** — `firestore.rules` matches a single path segment (`match /users/{uid}/max_tracker/{document}`, not `{document=**}`) and requires `document` to match `^(shifts|finance|debt|backup_v2)(@[A-Za-z0-9_]+)?$|^profiles_meta$`, plus shallow validation (top-level key count, `updatedAt`/`at` must be numbers if present — see the rules file's own header comment for why this stops short of a full per-field schema). **Adding a new doc *base name* now needs a matching `firestore.rules` change**, unlike before; a new `@<profileId>`-suffixed variant of an existing base name needs none. Before touching this file, run `tests/firestore-rules.mjs` (see Commands above) — it asserts against a real emulator and has already caught a rules bug (duplicate `allow write` blocks silently canceling validation via OR semantics) that a read-through missed. A legacy pre-auth shared `max_tracker/*` collection is explicitly denied in `firestore.rules` — do not resurrect it.
- **Transactions live in their own subcollection, not the `finance` doc.** `users/{uid}/max_tracker/{financeDocName}/transactions/{txId}` (`txId` = `String(tx.id)`, the same `Date.now()`-based id transactions have always used) — this is the one subcollection under `max_tracker`, with its own sibling `match` block in `firestore.rules` scoped to a `finance`/`finance@<profileId>` parent only. See `MIGRATION_PLAN_transactions.md` for the full history/rationale (1 MiB doc-size risk, whole-doc rewrite on every save, doc-level-not-field-level optimistic concurrency) — completed as a **direct cutover**, not the originally-planned phased dual-write/dual-read rollout, since the app was confirmed pre-launch with no real users to protect through a slow migration (see Operating conventions above). All Firestore access for transactions goes through `js/firebase-sync.js`'s TRANSACTIONS SUBCOLLECTION helpers (`txCollection()`, `saveTransactionDoc()`, `deleteTransactionDoc()`, `loadTransactionsFromSubcollection()`, `batchWriteTransactions()`, `batchDeleteTransactionsByIds()`, `deleteAllTransactionDocs()`) — every function that adds/edits/deletes a transaction, or cascades a change onto many transactions at once (tag deletion, category rename), calls one of these directly instead of `scheduleSave()`, so a single transaction edit no longer rewrites the whole `finance` doc. `fbLoadNow()` does a one-time client-side migration per account/profile: if the subcollection is still empty but the legacy `finance.data` array has entries, it batch-copies them in once and sets `AppState.txMigrated`/`finance.txMigrated` (mirrors the existing `catBackfillDone`/`catLegacyMerged` flag pattern) so a later load — even after every transaction has since been deleted, legitimately leaving the subcollection empty again — never re-migrates the stale array. `fbSaveNow()` no longer writes a `data` field onto the `finance` doc at all (dropped automatically by `{merge:false}` replacing the whole doc). Account deletion (`js/auth.js`'s `deleteAccountUser()`) explicitly deletes every doc in the subcollection too — Firestore never cascade-deletes a subcollection when its parent doc is deleted. The Cloud Function (`functions/lib/sweep.js`) reads the subcollection via the Admin SDK in parallel with the finance/debt doc reads it already did, falling back to the legacy `finance.data` array only when the subcollection is empty (a pre-migration account, or a race with a client that hasn't loaded yet). `scripts/backfill-transactions.mjs` (an operator-run one-time bulk migration script) predates this client-side automatic migration and is now optional — useful only for pre-migrating in bulk before rollout rather than waiting for each account's next app load.
- The `finance` doc still holds everything else: wallets, categories, budgets, subcategories, currencyRates, tags, autoRules, recurring, `shoppingList`, goals, profile, subscription, widgets, **and `notifSettings`** (synced here specifically so the server-side Cloud Function can see each user's notification preferences — it used to be localStorage-only; now also carries `notifSettings.timeZone`, an IANA zone name captured client-side for the push sweep's local-time calculations).
- **`renderFinance()`'s transaction list (`js/analytics-csv.js`) does a targeted DOM update, not a full rebuild.** `renderFinance()` is called from ~20 places across `js/*.js` for state changes that often have nothing to do with any specific transaction (renaming a wallet, editing a budget, switching tabs back), so tearing down and recreating every `.tx-item` on every call was wasteful and also visibly replayed the row entrance animation on unrelated re-renders. It now matches existing `.tx-item` nodes by `data-id` (`dataset.txId`, a stringified transaction id) against the new `visible` list: a matched row gets its inner content regenerated in place via `txItemInnerHtml()` (so nothing ever goes stale — wallet/tag/category names are recomputed fresh every render, same as before) but keeps the *same outer element*, so its click/keydown listeners stay bound without re-registration and the CSS entrance animation (which only triggers on fresh insertion, not on content/attribute mutation of an already-connected node) doesn't replay; only genuinely new transactions get a freshly created node. Existing nodes are repositioned via `insertBefore` if their order changed, and any node whose transaction is no longer in `visible` (deleted, or filtered/collapsed out) is removed. One gotcha this pattern has to guard against explicitly: transitioning *from* the empty-state (`emptyStateHtml()`) or cold-start skeleton (`renderFinanceSkeleton()`) markup leaves non-`.tx-item` children in `#tx-list-container` that the diff wouldn't otherwise know to clear (it only tracks `.tx-item`/`.tx-view-all-btn`) — handled by an explicit `:not(.tx-item):not(.tx-view-all-btn)` cleanup pass before the diff runs. Covered by `tests/tx-list-targeted-update.mjs`, which tags rows with a custom attribute that only survives on a *reused* node (proving identity, not just matching content) across an unrelated re-render, an add, and a delete.
- `push_tokens/{uid}` is a separate top-level collection (not nested under `users/{uid}`) holding `{token, updatedAt, sentDaily, sentBudget, sentRecurring, profileState}` for the FCM push sweep; it has its own Firestore rule scoped to `request.auth.uid == uid`, plus type/size checks on the client-writable `token`/`updatedAt` fields (not a full key allowlist, since the Cloud Function's Admin SDK writes its own dedup fields onto the same doc, bypassing rules entirely).
- `error_reports/{uid}` (another separate top-level collection, added in the audit-followup session) mirrors the capped 20-entry client-side error log that `js/core.js`'s `logAppError()` already kept in `localStorage` (`mx_errlog_<uid>`) — before this, a crash on a real user's phone was invisible to the account owner unless the user sent a screenshot. `logAppError()` now also (best-effort, rate-limited to once per 30s via a module-level `lastErrorSyncAt` so an error loop can't turn into a write storm) whole-doc-replaces `error_reports/{uid}` with `{entries:[...], updatedAt}`, stack traces truncated to 500 chars. Only fires when `AppState.currentUser` is set — signed-out/anonymous errors stay local-only, not worth relaxing the per-uid rule for. Nothing in the client ever *reads* this collection — the account owner checks it via the Firebase console (Admin SDK bypasses rules), so `firestore.rules`' `error_reports/{uid}` match block only needs to gate writes (owner-only, `entries` must be a ≤20-item list, `updatedAt` a number). Deliberately **not** added to `deleteAccountUser()`'s cleanup `Promise.all` — same reasoning as `push_tokens/{uid}` already being left alone there: it's a small, capped, harmless-if-orphaned doc outside the `max_tracker` doc-name scoping that cleanup targets, and account deletion already has enough moving parts.
- Client-side caching/sync: every piece of state is mirrored into `localStorage` under per-user keys via `lsKey(name)` → `mx_<name>_<uid>` (loaded first for instant paint), then reconciled against Firestore in `fbLoadNow()`. Writes go through `scheduleSave()` (debounced) → `fbSaveNow()`, which uses optimistic concurrency (`lastKnownUpdatedAt` per doc, compared against each doc's `updatedAt`) and prompts the user on conflict rather than silently overwriting another device's newer save. A handful of device-global (not per-user) preference/dedup keys — theme, language, hide-amounts toggle, exchange-rate source/timestamp, and local-notification dedup state — live under a separate `mx<Name>` naming (e.g. `mxTheme`, `mxLang`, `mxNotif`; see `js/theme-preinit.js`, `js/classic-globals.js`, `js/settings-managers.js`, `js/notifications.js`) rather than through `lsKey()`, since they apply before any user is authenticated (`theme-preinit.js` runs pre-auth) or aren't meaningfully per-account. These used to carry a legacy `xamss*` prefix from before the Zminka (now Rytm) rename; renamed to `mx*` while the app was still pre-launch with no real users to reset (see Operating conventions above) — don't reintroduce the old prefix. `mx*` is an internal storage prefix decoupled from the marketing brand name on purpose, so it did not need to change again when the brand was renamed Zminka→Rytm.

### Multiple profiles per account
One Firebase Auth account can hold several independent data-profiles (e.g. "Я"/"Дружина") with their own finances/shifts/debts, switchable in Settings → Профілі.
- `activeProfileId` (module-level, default `'default'`) drives both `userDoc(name)` and `lsKey(name)`: for the **default** profile these resolve to the original unsuffixed doc names/localStorage keys (zero migration, existing users are untouched); for any other profile, `userDoc` suffixes the doc name `"@<profileId>"` (e.g. `finance@abc123`) and `lsKey` suffixes the localStorage key the same way. Because every save/load path already funnels through these two functions, nothing else needed to change to make the whole app profile-aware.
- `profiles_meta` (account-wide, never suffixed even when a non-default profile is active) holds `{list:[{id, name, avatar, createdAt}], updatedAt}` — just labels/avatars for the switcher, not real data. Each profile's actual identity (nickname + avatar shown in Settings → Профіль and the topbar) lives inside *that profile's own* `finance` doc, since `profile.nickname`/`profile.avatar` is per-profile automatically once `finance` is suffixed.
- `activeProfileId` itself is a **per-device** choice (`localStorage['mx_activeProfile_'+uid]`), not synced — switching profiles on one device doesn't affect others.
- `switchProfile(id)` flushes the current profile (`fbSaveNow()`) before reassigning `activeProfileId` and calling `fbLoadNow()` again — this reuses all the existing seeding/rendering logic verbatim, so a brand-new profile gets the same default wallets/categories a brand-new account gets.
- The scheduled Cloud Function (`functions/index.js` `notificationSweep`) reads `profiles_meta` and sweeps every profile on the account (default + suffixed), not just the default one — each profile's `finance`/`finance@<id>` doc is checked independently. Per-profile dedup state lives in `push_tokens/{uid}.profileState.<profileId>` (`{sentDaily, sentBudget, sentRecurring}`); pre-existing token docs' flat `sentDaily`/`sentBudget`/`sentRecurring` fields are read as the default profile's initial state until the sweep writes `profileState.default` for the first time, so old single-profile users need no migration. Local in-app notifications work fine for whichever profile is active since they read the same in-memory state that switching already repopulates.
- Any render function that bails out early when its collection is empty (`if(!items.length) return;`) needs to explicitly clear its previously-rendered DOM first — otherwise switching to a profile with no data leaves the *previous* profile's numbers on screen instead of blanking them (this bit us once with the Debt tab; check for the same pattern before adding another profile-scoped collection).

### Auth
Three sign-in methods, all in the AUTH / PHONE SIGN-IN sections: email+password, Google popup, and phone number (SMS code via `signInWithPhoneNumber` + an invisible `RecaptchaVerifier`). Layered on top, independent of Firebase Auth: a local PIN (SHA-256 hash in localStorage, `mx_pin_<uid>`) and optional WebAuthn biometric unlock — these gate the UI locally but never replace the Firebase session.

Settings → Акаунт additionally lets an already-signed-in user (e.g. one that signed in via Google) **link/unlink a phone number** as an extra credential (`linkWithPhoneNumber(currentUser, ...)` / `unlink(currentUser,'phone')`) — a separate flow from the pre-auth sign-in one, with its own `link-recaptcha-container`/`RecaptchaVerifier` instance so it doesn't collide with the lock-screen's.

**reCAPTCHA retry gotcha**: `RecaptchaVerifier.clear()` can itself throw partway through on a failed attempt, leaving the widget's DOM node behind; the next attempt's fresh `RecaptchaVerifier.render()` then finds a stale widget already in the container and throws `"reCAPTCHA has already been rendered in this element"`. Both phone flows now force the container's `innerHTML` empty on error (`resetRecaptchaContainer(containerId)`) regardless of whether `.clear()` succeeded — do this for any new reCAPTCHA-backed retry flow too.

**Phone auth troubleshooting** (not code issues, but easy to mistake for one): (1) needs the **Blaze** plan (documented below already); (2) needs **Authentication → Settings → SMS region policy** to explicitly allow the country being tested (Firebase blocks SMS to unlisted regions by default); (3) **Brave browser disables Web Push by default** (`brave://settings/privacy` → "Use Google services for push messaging") — causes `AbortError: Registration failed - push service error` on `getToken()`/`PushManager.subscribe()` that looks like a bug but isn't; (4) **GitHub Pages mirror isn't auto-authorized** — `*.web.app`/`*.firebaseapp.com` are automatically in Firebase's Authorized domains, but `sudak666.github.io` is not, so invisible reCAPTCHA fails (flashes and vanishes) and phone sign-in/link dies with a generic error specifically on that mirror while working fine on `maxtr-c238f.web.app`. Add `sudak666.github.io` to Authentication → Settings → Authorized domains manually (see SETUP.md §2). `authErrorMessage()` in `index.html` now maps `auth/captcha-check-failed`, `auth/unauthorized-domain`, and `auth/network-request-failed` to specific messages instead of the generic fallback, to make this diagnosable from the UI itself next time; (5) **`auth/error-code:-39` (503 on `sendVerificationCode`) for every number tried** — not billing, not the API key, not the region policy, not a specific number (confirmed by process of elimination and then by Firebase Support directly, case 10414309, July 2026): Firebase's automated anti-toll-fraud system has temporarily blocked the carrier/region for low SMS-success rates. `authErrorMessage()` maps this (and `auth/internal-error`) to a dedicated message, scoped to phone-flow call sites only via an `isPhoneFlow` param — see SETUP.md §5 for the real fix (enable reCAPTCHA SMS Defense, which is why the Firebase JS SDK is pinned to `12.11.0` rather than `10.x`: SMS Defense requires Web SDK 11+).

### Premium / free-tier limits
`subscription = {plan, expiresAt}` lives in the `finance` doc. `FREE_LIMITS`, `canAddMore(kind, count)`, and `showPremiumUpsell(kind)` still exist (kept for when a payment provider ships), but **no "add" action currently calls `canAddMore`** — the wallets/categoriesPerType/autoRules/recurring/goals caps were deliberately removed, because with no way to actually purchase premium, a hard cap was just a dead end for every user (this is what was silently blocking "add category" for any account with ≥8 categories — a brand-new account already starts at exactly 8 default expense categories). If a payment provider gets wired up, the caps should come back by re-adding the `if(!canAddMore(...)){ showPremiumUpsell(...); return; }` guard at the top of each `add*()` function (see git history around the "Remove remaining free-tier caps" commit for the exact call sites) rather than reinventing the gating pattern.

Adding a new toggleable widget to the main scroll means updating the default in all 3 of those places — grep for `widgets.goals` to find them — plus `WIDGET_ORDER_DEFAULT`/`WIDGET_SECTION_IDS`/`WIDGET_DEFS`. `TX_LIST_COLLAPSED_COUNT` (`js/analytics-csv.js`) is 5, not 3 — how many transactions the main scroll shows before requiring a "view all" tap.

### Notifications (two independent delivery paths, same three conditions)
Both paths check: daily "did you log anything today" reminder, budget-exceeded, and upcoming-recurring-payment.
1. **Local** (`NOTIFICATIONS` section): fires only while the app is open, via the `Notification` API, checked on load / tab-visibility-change / a 5-minute interval. Dedup uses localStorage keys.
2. **Push** (`PUSH NOTIFICATIONS` section + `functions/index.js`): the client registers an FCM token (`push_tokens/{uid}`) via `getToken()` + `sw.js` (see Layout above — `sw.js` now also carries the FCM background-message handler). A scheduled Cloud Function (`notificationSweep`, hourly) re-implements the same three checks server-side against the `finance` doc and sends real pushes via the Admin SDK, with its own dedup fields stored back on the token doc. If you change the local-notification logic, check whether `functions/index.js` needs the equivalent change — the two are intentionally kept in sync but are separate implementations (client JS vs. Cloud Function), not shared code. **Themed per-type icons (a later session)**: `functions/index.js`'s `sendPush(token, title, body, type)` takes a 4th `type` argument (`'daily'|'budget'|'recurring'|'debt'`, one per `functions/lib/sweep.js` check) and looks it up in `NOTIF_ICONS` to set `webpush.notification.icon` on the FCM payload, instead of every push showing the same generic `icon-192.png`. `sw.js`'s `onBackgroundMessage` handler reads `payload.notification.icon` and passes it straight to `showNotification()`'s own `icon` option (falling back to `icon-192.png` if absent — e.g. a manually-sent test push with no icon field). The 4 icon PNGs (`notif-icon-daily.png`/`-budget.png`/`-recurring.png`/`-debt.png`, repo root) are hand-generated (Pillow, purple-gradient rounded-square background matching `icon-192.png`'s own branding, simple white glyph — bell/warning-triangle/repeat-circle/bank-building) rather than reusing the SVG `ICON_PATHS` set, since notification icons need to be static image URLs, not inline SVG; same-origin, so added to `sw.js`'s `STATIC_ASSETS` precache list like every other same-origin asset. Covered by `tests/functions-sweep.mjs`'s 4 new `type='...'` assertions (one per push check) — the display side (`sw.js`) isn't covered by an automated test, since it needs a real FCM background-message delivery this repo's stubbed-Firebase Playwright recipe can't simulate; verify manually via a real device if this ever needs re-checking.

### UI conventions worth following
- Custom-styled `<select>`/`<input type=date>`: any new one just needs `enhanceSelect()` / `enhanceDateInput()` called on it (or add it before `enhanceAllSelects()` runs) — don't hand-roll another dropdown.
- **Never use `<input type=color>`** — it opens the browser's native, unstylable OS color-picker popup, which looks jarringly out of place against the app's dark custom UI. Use the shared `openColorPicker(kind, id)` / `color-pick-modal` pattern instead (a `PALETTE`-swatch grid modal, same idea as the avatar picker) — wire a new color-editable list (wallets/shiftTypes/tags all already do this) by adding a case to `selectPickedColor()` and rendering a plain `<button class="mgr-color" style="background:${color}" onclick="openColorPicker('kind','${id}')">` instead of a color input.
- Similarly, don't rely on native browser affordances that can't be restyled or that read as "not part of the app" — e.g. no `title="..."` tooltips (native OS tooltip box); prefer a visible label/subtitle in the UI itself, like the rest of the settings rows do.
- "Settings manager" modals (wallets, categories, budgets, goals, recurring, tags, rules, rates, widgets, pin, premium, profiles, link-phone, avatar/color pickers) all follow the same pattern: a `.modal-overlay` + `.modal-card`, an `open*Manager()`-style function that renders a list into it and sets `display:flex`, and the shared `closeManagers()` that hides all of them by id — add new manager ids to that shared close-list (an easy miss: forgetting this leaves a modal that won't close on backdrop-click). **Separately, every `.modal-overlay` in `index.html` (23 of them, including the two — `#shift-modal`, `#ui-dialog` — that `closeManagers()`'s id list doesn't cover) already gets focus-trap/Escape-to-close/focus-restore for free**, via `setupModalAccessibility()` (`js/settings-managers.js`, called once from `js/app-init.js`'s cold-init block): a `MutationObserver` watches every `.modal-overlay`'s `style` attribute, and on open pushes the overlay onto `AppState.openModalStack`, remembers `document.activeElement` as the element to restore focus to, and auto-focuses the first focusable element inside the `.modal-card`; a single `document`-level `keydown` listener traps `Tab`/`Shift+Tab` within the topmost open modal's card and closes it on `Escape` (by synthetically `.click()`-ing the overlay itself, which reuses whatever close logic that specific modal already has, rather than duplicating it). This is a *different*, more robust choke point than `closeManagers()`'s hard-coded id list — any new modal gets this accessibility behavior automatically just by having `.modal-overlay`/`.modal-card` markup present in the DOM at `init()` time, no id-list registration needed (though it still needs adding to `closeManagers()` separately for backdrop-click/close-all). Covered by `tests/e2e-modals.mjs`.
- All icons are the custom monochrome SVG set (`window.Icon(name)` / `setIcon(id, name)`) — no emoji, no icon fonts. Add new glyphs to `ICON_PATHS` rather than reaching for an emoji. **Check for duplicates within the same visible list/row-group** before reusing an existing glyph — two *different* features sitting next to each other (e.g. two quick-action buttons, two settings-card icons in the same screen) with the same icon reads as a bug to users even though it isn't one; reusing an icon for the *same* feature across contexts (a tab + its settings entry + its quick-action) is fine and intentional. Generic action icons (`plus` for add, `trash` for delete, `pencil` for edit, `chevron`, `refresh`) are meant to repeat everywhere — that's a consistent affordance, not a duplicate.

When adding a built-in shift type, give it a `code`.

- i18n: a single `I18N = {uk:{...}, en:{...}}` object plus `tr(key)`; static markup uses `data-i18n`/`data-i18n-placeholder` attributes applied by `translateStaticDOM()`. There is no `data-i18n-title` — dynamic titles/tooltips must be set in JS.
- The bottom nav (`.tabs`) currently has 5 tabs (Фінанси/Графік змін/Розрахунки/Покупки/Налаштування) — `.tab-icon-wrap` is 42px (38px under the ≤600px breakpoint). Adding a 6th tab means re-checking these still fit comfortably; there's no overflow/scroll handling on that bar.

### Android / Google Play packaging (TWA, July 2026)
The app ships to Google Play as a **Trusted Web Activity (TWA)** — a thin Android wrapper that opens the live `maxtr-c238f.web.app` site full-screen, not a separate native codebase. There is no local Android project checked into this repo; it's generated on demand.
- **Canonical domain for the Android package is `maxtr-c238f.web.app`**, not the GitHub Pages mirror — chosen because Play's Digital Asset Links verification needs one fixed origin, and the Firebase domain has no subpath (GitHub Pages serves under `/Maxtr/`, which would complicate `start_url`/asset-link matching).
- **`.well-known/assetlinks.json`** (repo root) is what lets Android show the app without a browser URL bar — it declares the app's package ID + the signing certificate's SHA-256 fingerprint, and Google Play/Android verifies it against `https://maxtr-c238f.web.app/.well-known/assetlinks.json`. `firebase.json`'s hosting `ignore` list originally excluded all dotfiles (`**/.*`), which would have silently dropped this file from every deploy — it now has an explicit `"!/.well-known/**"` un-ignore plus a `Content-Type: application/json` header override for that path. If this file's fingerprint and the actual signing key ever drift apart (e.g. the key gets regenerated), the app degrades to showing a Chrome URL bar instead of failing outright — annoying but not broken, and the fix is always "update the fingerprint in this file, redeploy hosting."
- **Package ID is `ua.rytm.app`** (changed from the placeholder `ua.zminka.app` on 2026-07-16 once the account owner made the final "Rytm" branding decision — see Project overview). Still freely changeable in whatever tool generates the package (twa-manifest.json / PWABuilder's Package ID field) up until the *first* real Play Store upload, which per the "Google Play packaging" status below hasn't happened yet; after that, changing it means publishing as a brand-new listing, not an update.
- **The actual APK/AAB build cannot happen from a Claude Code cloud sandbox.** Confirmed by hitting explicit 403 policy denials from this environment's egress proxy on `dl.google.com` (Android SDK/build-tools — blocks even the offline `bubblewrap build` path), `pwabuilder.com`, and even `maxtr-c238f.web.app` itself. Only `github.com`/npm/pypi-class hosts and Google's OAuth/Firebase *API* endpoints (as opposed to the Hosting *content* domain) were reachable. What **did** work fully offline from the sandbox: generating the Android Studio project skeleton via `@bubblewrap/core`'s `TwaManifest`/`TwaGenerator` classes directly (bypassing the CLI's interactive prompts, feeding it the local `manifest.json` instead of fetching it live, and serving the icon files over a `127.0.0.1` static server so `TwaGenerator` had something reachable to pull them from), and generating the release signing keystore with the locally-installed JDK's `keytool` (no network needed at all for that part). The actual compile step then has to happen either in Android Studio on a real machine, or — the path actually used — **pwabuilder.com** from the user's own browser, pointed at `https://maxtr-c238f.web.app`.
- **Signing keystore is intentionally not committed to this repo** — it was generated once (`keytool -genkeypair`, RSA/3072, PKCS12 format) and handed directly to the account owner as a downloadable file; losing it means never being able to publish an update under `ua.rytm.app` again (the keystore itself isn't tied to the package ID, so the 2026-07-16 `ua.zminka.app`→`ua.rytm.app` rename didn't require regenerating it — only `.well-known/assetlinks.json`'s `package_name` field needed updating, which it now has). If the keystore itself ever needs regenerating, `.well-known/assetlinks.json`'s fingerprint has to be updated and hosting redeployed to match, or the "no browser URL bar" behavior breaks.
- **PKCS12 keystore gotcha, cost real debugging time**: `keytool` on JDK 9+ defaults to PKCS12-format keystores, which — unlike the older JKS format — **do not support a separate password for the private key entry**; `keytool -genkeypair -keypass X -storepass Y` with `X != Y` silently *succeeds* but prints `Warning: Different store and key passwords not supported for PKCS12 KeyStores. Ignoring user-specified -keypass value.` and just reuses the store password for the key too. Any consumer that's later told "the key password is X" (PWABuilder, `apksigner`, etc.) will then fail with `javax.crypto.BadPaddingException: Given final block not properly padded` — a decryption error, not an authentication-rejected error, which reads like file corruption rather than a wrong-password problem. **Always generate PKCS12 signing keys with the same value for both `-storepass` and `-keypass`**, and only document/hand off one password, not two.
- **`android-icon-rounded.png`** (repo root) is a squircle-rounded-corner variant of `icon-512.png`, generated with Pillow — used *only* by pasting its URL into PWABuilder's Android package form "Icon URL" field, not referenced anywhere in `manifest.json`. Reason: the TWA's launch splash screen centers the raw icon on the manifest's `background_color` with **no adaptive-icon masking applied** (unlike the home-screen launcher icon, which Android masks itself), so `icon-512.png`'s hard square corners looked jarring specifically on that one screen. The form's separate "**Maskable icon URL**" field must keep pointing at the original full-bleed `icon-512.png` — pre-rounding that one would break the safe-zone contract Android's own adaptive-icon masking expects.
- **`icon-192-rounded.png`/`icon-512-rounded.png`** (repo root, same 20%-corner-radius Pillow squircle mask as `android-icon-rounded.png` above, generated independently rather than reusing that file so its own single-purpose documentation above stays accurate) exist for the same underlying reason as `android-icon-rounded.png`, but for the desktop-PWA-install case instead of the Android TWA case: `manifest.json`'s `icons` array used to list `icon-192.png`/`icon-512.png` once each with `"purpose": "any maskable"` combined — but "any" (used as-is, no OS masking — e.g. the Windows desktop shortcut Chrome creates on `chrome://apps` "Install" or a taskbar pin) and "maskable" (Android applies its own adaptive-icon shape mask and needs a full-bleed, un-rounded source) are contradictory requirements on the same file. The combined purpose meant Windows showed the raw square `icon-512.png` with hard corners on desktop shortcuts (reported directly by the account owner via a screenshot). **Fix**: split into four `icons` entries — `icon-*-rounded.png` at `"purpose": "any"`, original `icon-*.png` (unchanged, still full-bleed square) at `"purpose": "maskable"` only. Both new files added to `sw.js`'s `STATIC_ASSETS` (same reasoning as every other same-origin asset there). Deliberately did **not** touch `index.html`/`landing.html`'s `apple-touch-icon` link tags or `js/notifications.js`'s notification icon references — both still point at the original square `icon-192.png` on purpose: iOS applies its own squircle mask to `apple-touch-icon` automatically (pre-rounding it would double-mask/look wrong), and OS notification-tray icons are typically shown small enough that corner rounding doesn't matter and isn't worth another asset variant.
- Rebuilding the Android package from scratch (e.g. after a manifest/icon change) is a manual, browser-based process: pwabuilder.com → enter `https://maxtr-c238f.web.app` → Package For Stores → Google Play → fill Package ID / Icon URL / Maskable icon URL → Signing key → "Mine" → upload the existing `.jks` and its one password → Download Package. There's no CI/automation for this yet.
- As of this session: Google Play Console developer account **not yet created** by the account owner ($25 one-time fee, manual step at play.google.com/console) — nothing has been uploaded to Play yet, only the hosting-side asset-link verification and a locally-tested `.apk` exist so far.
