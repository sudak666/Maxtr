## Project overview

**Rytm** (formerly "Xamss", then "Zminka" — renamed to Rytm 2026-07-16) is a static PWA: a personal tracker for work shifts, multi-currency finances, and debt/settlement schedules. It's Ukrainian-first (`uk`/`en` toggle), backed by Firebase (Auth + Firestore + Cloud Messaging + one scheduled Cloud Function), and deployed via Firebase Hosting. `index.html` holds the markup/CSS and loads four small classic (non-module) scripts plus the module graph; almost all app logic lives under `js/*.js`, loaded via `<script type="module" src="./js/app.js">` (see "`js/` module layout" below).

## Architecture

### Layout
- `index.html` — markup, CSS, four classic `<script src>` tags plus `js/app.js` as the module-graph entry point (see "`index.html` script structure" below).
- `js/` — app logic, 16 native ES modules (no bundler needed for the source tree — browser resolves relative `import`s directly). See "`js/` module layout" below.
- `functions/` — separate Node 20 npm project (Cloud Functions v2, `firebase-admin`+`firebase-functions`), deployed independently.
- `firebase.json` / `firestore.rules` / `manifest.json` — Hosting, Firestore rules, PWA manifest.
- `sw.js` — the single service worker: PWA asset caching (network-first HTML, cache-first everything else) **and** FCM background push display, merged into one file because a scope can only have one active SW (there is no separate `firebase-messaging-sw.js` — don't recreate one). `FIREBASE_SDK_ASSETS` precaches the exact gstatic SDK URLs `index.html` statically imports (a static ES module import is all-or-nothing) — if the pinned SDK version in `index.html` changes, update this list and bump `CACHE_NAME`. `STATIC_ASSETS` lists every same-origin file needed for a cold offline start — add any new `js/*.js` file here too.
- `privacy.html` / `terms.html` — standalone static pages, not part of the SPA.
- `SETUP.md` — manual, non-code Firebase Console setup (sign-in providers, authorized domains, VAPID key, GCP APIs). Keep in sync with new external dependencies.

### `index.html` script structure
Five `<script>` tags, in load order:
1. `js/theme-preinit.js` — applies saved theme before first paint (must stay a blocking classic script, first).
2. `js/touch-active-fix.js` — no-op `touchstart` listener; iOS Safari only applies `:active` CSS when at least one touch listener exists anywhere.
3. `js/sw-register.js` — registers `sw.js` on load, and drives auto-update: `controllerchange` reloads the page once when a new worker takes control, plus `reg.update()` on load/tab-refocus. Needed because the app is a long-lived PWA/TWA — without this, an already-open tab never picks up a deploy's new JS. Paired with `firebase.json` serving `sw.js`/`/js/**` as `Cache-Control: no-cache`. **When debugging "my fix isn't working on the phone," suspect stale JS delivery first** — the local test suite serves fresh files every run and is blind to this.
4. `js/classic-globals.js` — the monochrome SVG icon library (`ICON_PATHS`, `window.Icon(name)`, `setIcon`) plus I18N/`window.tr()`/`translateStaticDOM()`/`setLang()`/`setTheme()`, and a delegated-click listener for `.theme-opt`/`.lang-opt`/`.modal-card`/`.dlg-card`/`#profile-avatar-preview`. Runs before the module script and is genuinely outside the `js/` module graph (no `import`/`export`, no `__init_X__()`). `window.__applyLangDynamic` is the one hook back from `js/core.js` so `setLang()` can re-render dynamic content.
5. `<script type="module" src="./js/app.js">` — the whole `js/` module graph. Functions called from inline `onclick`/`onchange` attributes must be attached to `window` explicitly, since module-scope declarations aren't global — give it a real top-level declaration (`function foo(){}`) plus `window.foo = foo;`, not `window.foo = function(){...}` as the only definition (any other file that later needs to `import` it by name won't have one to import). **As of the CSP-hardening pass, inline `onclick=""` is the exception, not the rule** — see the CSP paragraph below; the ~10 deliberately-kept cross-file `window.*` exports are documented in CHANGELOG.md's onclick-removal-audit entries.

**A strict Content-Security-Policy IS deployed** (`firebase.json` hosting header): `script-src 'self' https://www.gstatic.com https://apis.google.com` — no `'unsafe-inline'`, no `'unsafe-eval'` (plus `https://www.google.com` for App Check's reCAPTCHA script and `'wasm-unsafe-eval'` for Tesseract's WASM — see those sections below). **Inline event-handler attributes silently do not execute on the live site**, and `eval`/`new Function` are blocked. The sandbox's `python3 -m http.server` sends no CSP, so inline handlers work in every local test and only no-op in production — this shipped real bugs before the onclick-removal audit (see CHANGELOG.md). **NEVER add an inline `onclick=""`/`onchange=""`** to `index.html` or a `js/*.js` template string — wire it as a `data-action` dispatched by the file's delegated capture-phase listener, or `el.onclick = fn` set in JS (a DOM-property assignment, not an inline attribute — CSP allows this). `tests/csp-handlers.mjs` (CI) loads the page under the real CSP and exercises these flows so this can't regress silently.

### `js/` module layout
`index.html` used to be one 7000-line inline `<script type="module">`; it's now 16 files under `js/` (`state.js`/`core.js`/`firebase-sync.js`/`color-picker.js`/`auth.js`/`app-init.js`/`ui-widgets.js`/`calendar.js`/`settings-managers.js`/`goals-profile.js`/`notifications.js`/`finance.js`/`analytics-csv.js`/`debt.js`/`shopping.js`, plus `app.js` as entry point) — a mechanical AST-verified split, not a rewrite. Three things matter for anyone splitting further or adding a cross-file call:

1. **All mutable app state lives in `AppState` (`js/state.js`)** — `transactions`, `wallets`, `shifts`, `currentUser`, `activeProfileId`, etc. (`AppState.x`, not per-field `export let x`), because many fields are reassigned from several different files and ES modules only let the declaring module reassign a plain binding. **`js/state.js` has zero imports of its own, deliberately** — a circular dependency back into it risks a temporal-dead-zone `ReferenceError` on its own top-level `const AppState = {...}`. **If you add a new module-level mutable field, put it in `AppState`, not a fresh top-level `let`** — a bare `let` can't be reassigned from another file.
2. **Every chunk with top-level side-effecting code is wrapped in an exported `__init_<chunk>__()`, called from `app.js` after all 16 files finish importing** — module evaluation order across a real (partly circular) import graph isn't guaranteed to match a single-file's original top-to-bottom order. **If you add new top-level side-effecting code** (`document.addEventListener`, `setInterval`, anything that isn't a `function`/`class`/`const`/`let` declaration), put it inside that file's `__init_X__()`, not bare at the top level.
3. **A name only ever exposed as `window.foo = function(){...}` still needs a real export/import if another file references the bare identifier `foo`** — the `window.foo = foo;` assignment line itself needs `foo` to already be a real lexical binding. **When adding a function only meant for inline HTML**, still give it a real top-level declaration plus a separate `window.foo = foo;` line (except `closeManagers`, which has a separate same-named bare wrapper with different code).

Verification for this split: `tests/smoke.mjs` + a manual Playwright pass (transaction add, settings modal, language switch, shift-detail modal) with zero console/page errors. Treat `js/*.js` as hand-edited source from here on, same as `index.html` always was — the original AST-analysis/codegen scripts weren't committed.

### Build tooling (Vite bundler)
Landed across several independently-verified, independently-deployed phases (full history in CHANGELOG.md); this section is the durable summary.
- **`vite.config.js` bundles only the ES module graph rooted at `js/app.js`** (`build.rollupOptions.input`, not `index.html`) — feeding Vite the whole `index.html` was tried first and found to fingerprint/rewrite `<link rel="manifest">`/`apple-touch-icon` paths, breaking GitHub Pages. The 4 classic scripts and `js/vendor/tesseract/**` are untouched by Vite.
- **`js/receipt-ocr.js`'s dynamic `import()` of vendored Tesseract needed `/* @vite-ignore */`** (so Vite never bundles the vendor library) and `entryFileNames`/`chunkFileNames` config forcing the bundled chunk under `dist/js/` (matching the source depth the relative specifier depends on).
- **`scripts/build-site.mjs`** (run via `npm run build`, after `vite build`) copies every static file Hosting needs into `dist/`, patches `dist/index.html`'s `<script type="module">` tag to the real bundle filename, and regenerates `dist/sw.js`'s `STATIC_ASSETS` from the actual build output — the "forgot to add a new file" `sw.js` bug class is now structurally impossible for `dist/`. `sw.js`'s `CACHE_NAME` for `dist/` hashes **every** precached file's bytes (not just the JS bundle's hashed filename), since non-hashed copied assets (manifest, icons) would otherwise leave the SW text byte-identical across deploys and never trigger an update.
- **`firebase.json`'s `hosting.public` is `dist`** (was `.`), with `hosting.predeploy: ["npm install", "npm run build"]` so a deploy always regenerates `dist/`.
- **GitHub Pages still serves the unbundled source tree** — standing, intentional divergence (`maxtr-c238f.web.app` = bundled `dist/`, `sudak666.github.io/Maxtr/` = raw source); no tool available to an agent can flip Pages' build-source toggle.
- **`window.__RYTM_TEST_HOOKS__`** (`js/app.js`, attached after every `__init_X__()`): `{ AppState, addTransaction, scanReceiptImage, setMonobankSyncGapMsForTesting }` — replaces the old `page.evaluate(() => import('./js/state.js'))` pattern, which only worked against unbundled source. **If a test needs one more module-scoped internal, add it to this hook**, don't reach for a dynamic `import('./js/...')` — it breaks against `dist-smoke`.
- **CI has 3 jobs**: `smoke` (source), `dist-smoke` (bundled `dist/`, full parity via the test-hooks fix above), `firestore-rules` (emulator, unrelated).

### TypeScript adoption (gradual, JSDoc-based)
Type-checking plain `.js` files via inferred types + JSDoc, zero runtime change, zero file renames — the natural low-risk followup to the Vite work.
- **`tsconfig.json`**: `allowJs: true`, **`checkJs: false` project-wide** (`include: ["js/**/*.js"]`), `noEmit: true` (this never emits — Vite's esbuild still does the real build). Deliberate, not a placeholder — a global `checkJs: true` sweep against zero prior annotations would flood with errors.
- **Per-file opt-in via `// @ts-check` at the top of the file** — a standard TypeScript feature. Opted in so far: `js/tx-validation.js` (zero imports, no `AppState`/DOM), `functions/lib/pure.js` (needed its own `functions/tsconfig.json` — CommonJS, `module:"CommonJS"`, scoped to `include:["lib/**/*.js"]`, omit `moduleResolution` — TS7 removed `Node10`), `functions/lib/monobank.js` (already covered by the same config).
- **`npm run typecheck`** runs both configs (`tsc --noEmit && tsc --noEmit -p functions/tsconfig.json`), in CI right after the build step.
- **Next candidate**: `functions/lib/sweep.js` — a meaningfully bigger step (coupled to firebase-admin's Firestore snapshot types and the finance doc's complex shape), not a quick pass. A `js/*.js` file that imports `AppState` or the Firebase SDK's `https://` URL specifiers carries the same caveat on the root-tsconfig side.

### Preact adoption (proof of concept)
Third piece of the "doing this properly" initiative, deliberately scoped to one proof-of-concept widget, not a rewrite (confirmed with the account owner before starting). **Preact** (~4KB gzip core), plain hyperscript `h()` — no JSX, no build-time transform.
- **Target: the Debt tab's payoff-forecast widget** (`renderDebtForecast()`/`DebtBurndownChart`/`DebtForecastText` in `js/debt.js`) — pure, read-only, one call site, previously hand-rolled `innerHTML=` template concatenation. No other widget touched.
- **Vendored under `js/vendor/preact/preact.module.js`, not a bare npm import** — a real architectural constraint: GitHub Pages serves `js/debt.js` unbundled, where a bare package specifier isn't a valid URL and throws (all-or-nothing static import failure). Vendoring resolves identically on both deploy targets — GitHub Pages loads it as a normal same-origin module, Vite inlines it into the bundle normally (a genuine static import, not `@vite-ignore`d). No CSP change needed (self-hosted, same-origin).
- **`scripts/build-site.mjs` excludes `js/vendor/preact/**` from `dist/` and strips its `STATIC_ASSETS` line** (`ALWAYS_BUNDLED_VENDOR_DIRS`) — unlike Tesseract, Preact is always fully inlined into the bundle, so a standalone `dist/` copy would be dead weight and, worse, would make the dist SW's `cache.addAll()` 404. If a future always-statically-imported vendor file is added, add its directory to this same set.
- **Every code path touching a Preact-managed container must go through `render(h(...), targetEl)`, never a raw `.innerHTML=`** — mixing manual DOM writes with Preact-managed content desyncs Preact's diff tracking. `renderDebtForecast()`'s two text-only early-return paths go through `DebtForecastText` for exactly this reason.
- Followed since by 3 more Preact-rendered widgets (analytics donut chart, FX rates widget, crypto-top sparkline — see Finance-tab widgets below), same small stateless-`h()`-tree pattern each time. Expanding Preact further into the app is an explicit follow-up conversation, not decided by this proof of concept.

### Firebase data model
- Firestore layout: `users/{uid}/max_tracker/{doc}` where `{doc}` is one of `shifts`, `finance`, `debt`, `backup_v2`, `profiles_meta` (see Multiple profiles below). `DCOL = 'max_tracker'`; `userDoc(name)` builds the path. **`firestore.rules` matches a single path segment** (`match /users/{uid}/max_tracker/{document}`, not `{document=**}`) and requires `document` to match `^(shifts|finance|debt|backup_v2)(@[A-Za-z0-9_]+)?$|^profiles_meta$`, plus shallow validation (top-level key count, `updatedAt`/`at` must be numbers if present). **Adding a new doc *base name* needs a matching `firestore.rules` change**; a new `@<profileId>`-suffixed variant of an existing base name needs none. Run `tests/firestore-rules.mjs` (real emulator, not a read-through) before touching this file. A legacy pre-auth shared `max_tracker/*` collection is explicitly denied — do not resurrect it.
- **Transactions live in their own subcollection, not the `finance` doc**: `users/{uid}/max_tracker/{financeDocName}/transactions/{txId}` (`txId` = `String(tx.id)`). See `MIGRATION_PLAN_transactions.md` for rationale (1 MiB doc-size risk, whole-doc rewrite on every save) — completed as a direct cutover since the app was pre-launch with no real users to protect through a phased rollout. All access goes through `js/firebase-sync.js`'s TRANSACTIONS SUBCOLLECTION helpers (`txCollection()`, `saveTransactionDoc()`, `deleteTransactionDoc()`, `loadTransactionsFromSubcollection()`, `batchWriteTransactions()`, `batchDeleteTransactionsByIds()`, `deleteAllTransactionDocs()`) — never `scheduleSave()` for transaction writes. `fbLoadNow()` does a one-time client-side migration per account/profile from the legacy `finance.data` array, gated by `AppState.txMigrated`/`finance.txMigrated` so an empty subcollection after all transactions are deleted never re-migrates. Account deletion explicitly deletes every subcollection doc too (Firestore doesn't cascade-delete). `functions/lib/sweep.js` reads the subcollection via Admin SDK, falling back to `finance.data` only when empty (pre-migration account or race). `scripts/backfill-transactions.mjs` is an optional one-time bulk-migration operator script, superseded by the automatic client-side migration.

Only fires when `AppState.currentUser` is set. Not cleaned up on account deletion — small, capped, harmless-if-orphaned, same as `push_tokens`.

- Client-side caching/sync: every piece of state mirrors into `localStorage` via `lsKey(name)` → `mx_<name>_<uid>` (loaded first for instant paint), reconciled against Firestore in `fbLoadNow()`. Writes go through `scheduleSave()` (debounced) → `fbSaveNow()`, optimistic concurrency (`lastKnownUpdatedAt` per doc) prompting on conflict. ### Multiple profiles per account
One Firebase Auth account can hold several independent data-profiles (e.g. "Я"/"Дружина"), switchable in Settings → Профілі.
- `activeProfileId` (module-level, default `'default'`) drives both `userDoc(name)` and `lsKey(name)`: default profile → unsuffixed doc names/keys (zero migration); any other profile → `"@<profileId>"` suffix on both. Every save/load path funnels through these two functions, so nothing else needed to change.
- `profiles_meta` (account-wide, never suffixed) holds `{list:[{id, name, avatar, createdAt}], updatedAt}` — labels/avatars only; each profile's real identity (nickname/avatar) lives in *that profile's own* `finance` doc.
- `activeProfileId` itself is a **per-device** choice (`localStorage['mx_activeProfile_'+uid]`), not synced.
- `switchProfile(id)` flushes the current profile (`fbSaveNow()`) before reassigning and calling `fbLoadNow()` again, reusing all seeding/rendering logic.
- The Cloud Function sweeps every profile in `profiles_meta` independently. Per-profile dedup state lives in `push_tokens/{uid}.profileState.<profileId>`; pre-existing flat fields are read as the default profile's initial state until the sweep writes `profileState.default` for the first time.
- **Any render function that bails early on an empty collection must explicitly clear its previously-rendered DOM first** — otherwise switching to a profile with no data leaves the previous profile's numbers on screen (bit us once with the Debt tab).

### Shared profiles
A second Firebase Auth account can be invited to read/write one of this account's own profiles (shifts/finance/debt + transactions) — built for a couple sharing one household's finances, not a general multi-tenant model. Verified against a real local emulator (`tests/firestore-rules.mjs`) before any client code was written.

- **Reuses the existing `users/{uid}/max_tracker/{doc}@{profileId}` tree** — no parallel `shared_profiles` collection; the owner's side behaves exactly like any other profile. The only client-side generalization is in `js/firebase-sync.js`: `userDoc(name)` resolves against `AppState.activeProfileOwnerUid || AppState.currentUser.uid` (never for `profiles_meta`), and `lsKey(name)` returns `null` (no local caching) whenever `activeProfileOwnerUid` is set — a shared profile always loads fresh from Firestore.
- **Membership lives in a new doc-name kind, `shared_members`** (suffixed the same as shifts/finance/debt), holding `{members:[ownerUid, ...], updatedAt}` under the **owner's own** tree.
- **Joining is invite-code based, never open self-join.** `profile_invites/{code}` (new top-level collection — a joiner has no relationship to the owner's account yet) holds `{ownerUid, profileId, profileName, createdBy, createdAt, expiresAt, usedBy}`. 8-char unambiguous-alphabet code, client-generated, 24h TTL, single-use. **Redeeming is a genuine two-step, rules-enforced sequence**: step 1 marks the invite used (`usedBy: null → joinerUid`); only after that commits does step 2 (joining `shared_members@{profileId}`) become possible, because that write's rule does a `get()` on the invite doc and requires `usedBy == request.auth.uid`. A single-write "just check the members array shape" rule would leave a real hole — self-add to `shared_members` for any guessed `{ownerUid, profileId}` pair with no real invite, since Firestore rules can't cross-check "a different document was also written" any other way. Two joiners racing the same code can't both win — commits are serialized per document.
- **`firestore.rules`' main block gained**: `isSharedMember(ownerUid, document)` helper; read/write grant for `shifts`/`finance`/`debt` (never `backup_v2`/`profiles_meta`/`shared_members` itself) to any member; join/leave branches for `shared_members`. **Deletion of the profile's data docs stays owner-only even when shared** — a member can edit or leave, never delete the doc or forcibly remove another member (`removeAll` restricted to your own uid). The transactions subcollection got the identical extension.
- **Client functions** (`js/firebase-sync.js`): `shareCurrentProfile()`, `redeemSharedInvite(code)` (returns `{ok:false, reason}` rather than throwing), `leaveSharedProfile(ownerUid, profileId)`. UI: Settings → Профілі "Поділитись поточним профілем" (plain `uiAlert` with the code) and "Приєднатися за кодом" (`uiPrompt`); each shared profile row shows a badge and leave button.
- **Still out of scope, disclosed**: `notifSettings` lives inside the shared `finance` doc, so it's effectively shared too (a member changing the reminder time changes it for the owner). No cleanup of `shared_members`/`profile_invites` on account deletion. Granular permissions and push support (both listed here as v1 gaps) were implemented later — see the two sections below.

### Granular permissions on shared profiles
Two roles: **editor** (default, full read/write) and **viewer** (read-only).
- **`shared_members` gained a `roles` map**: `{members:[...], roles:{uid:'editor'|'viewer'}, updatedAt}`. A member absent from `roles` defaults to `'editor'` — backward-compatible, no migration, no existing shared profile silently loses write access.
- **Enforcement is in `firestore.rules`, not the client** — `memberRole()`/`isEditorMember()` gate `create`/`update`/`delete` on the main doc and the transactions subcollection; `isSharedMember()` alone (any role) still gates `read`. A viewer's client also gets read access to `shared_members` itself, to look up its own role. `canEditActiveProfile()` (`js/core.js`) is a client-side UX convenience mirroring this, never the security boundary.
- **Role changes are owner-only, applied after the fact, never during redemption** — every new member always joins as editor; the owner downgrades afterward via the **"Учасники" (Members) manager** (`js/color-picker.js`'s `openSharedMembersManagerUI()`/`renderSharedMembersList()`/`toggleMemberRoleUI()`, `#shared-members-modal`), a button on each owner-side profile row.
- **`AppState.activeProfileRole`** (`js/state.js`, default `'editor'`), loaded via `loadActiveProfileRole()` on `switchProfile()` and at cold-init. `canEditActiveProfile()` guards `addTransaction()`/`deleteTransaction()`, `saveModalSelection()`, `addDebtEntry()`/`deleteDebtEntry()` with a `shared_profile_readonly` toast instead of a doomed write. `applyProfileReadOnlyUI()` (`js/ui-widgets.js`) toggles `body.profile-readonly`, hiding `.fin-fab`. **Deliberately not individually guarded**: secondary settings mutations (renaming a wallet, editing a budget/category/tag/rule) — a viewer attempting one still fails safely via `firestore.rules`, just with the generic `sync_autosave_error` toast.

### Push notification support for shared profiles
`functions/lib/sweep.js` resolves a `dataOwnerByProfile` map before fetching each profile's docs: `profiles_meta.list` carries `{id, name, kind:'shared', ownerUid}` for a shared-profile reference, and `dataOwnerFor()` returns that `ownerUid` (or the token holder's own uid otherwise) — every `users/${uid}/...` path in the "other profiles" fetch uses this instead of always the token holder's own uid. `sweepProfile()` takes an optional `dataOwnerUid` parameter (default `= uid`). The Admin SDK bypasses `firestore.rules` entirely, so this needed no rules change — only which path is read. A member's push works the same regardless of viewer/editor role. **`notifSettings` is still the owner's** — a member's push uses whatever reminder time/toggles the owner last saved, a disclosed quirk (see Shared profiles above).

### Monobank integration
"Прив'язати Monobank" (Settings → Фінанси) — pulls transactions from the account owner's Monobank cards/jars via Monobank's Open API (`api.monobank.ua`). Scope decided upfront: **manual "Синхронізувати" button, not a webhook**, and **links every card and jar on the token at once**, not a per-account picker.

- **Cloud Function proxy needed because `api.monobank.ua` never sends CORS headers** (same root cause as `PRIVAT_RATES_URL`). `functions/index.js`'s `monobankProxy` (`onRequest`, exposed same-origin via the `"/api/monobank"` Hosting rewrite) relays `action=client-info` and `action=statement&account=&from=&to=`, attaching the caller-supplied `X-Monobank-Token` header. **Unlike `privatRates` (public, no auth), this proxy requires a valid Firebase ID token** (`Authorization: Bearer <idToken>`, verified via `firebase-admin`) — otherwise it'd be a free anonymous relay riding on this project's quota. Query-building/window-validation logic is split into `functions/lib/monobank.js` for unit-testability (`tests/monobank-proxy.mjs`).
- **The token is a live bank credential**, stored in `AppState.integrations.monobank = {token, clientName, accounts:[{id, kind:'account'|'jar', label, currencyAlpha}], mapping:{monobankAccountId: walletId}, lastSyncAt}` inside the existing `finance` doc — same per-uid rules protection as every other field, no new doc kind/rules change. Only ever sent to this app's own `monobankProxy`.
- **Connecting** (`connectMonobankUI()`, `js/monobank.js`) calls `client-info` once, auto-creates one wallet per account/jar (`createWalletForMonobankAccount()`), records `monobankAccountId → walletId` in `mapping`. No manual per-account picker.
- **Syncing** (`syncMonobankUI()`) walks `mapping`, calling `statement` for `[lastSyncAt ?? now-31days, now)`, chunked into ≤31-day-+1h windows (Monobank's hard limit), capped at 14 chunks as a defensive backstop. **Every call is paced at least 61s apart** (`paceMonobankRequest()`, `SYNC_REQUEST_GAP_MS`) — Monobank enforces "1 request/60s per token" across the whole API, so a multi-account sync can take minutes; `#monobank-sync-status` shows live progress text. `hold:true` entries (pending preauth) are skipped. **Dedup is by Monobank's own stable per-entry `id`**, stored as `monobankId` on the created transaction — re-syncing an overlapping window never creates duplicates.
- **Category**: existing keyword auto-rules (`findMatchingRule()`) get first say; unmatched falls back to `"Інше"`. No MCC→category mapping. **No transfer detection** — an internal move between two Monobank accounts shows as separate expense+income, same simplification as CSV import.
- **Testing**: `tests/monobank-proxy.mjs` (plain node — URL/window validation) and `tests/monobank-connect.mjs` (stubbed-Firebase Playwright + a `page.route()` stub for `/api/monobank`, since there's no Functions emulator in this test setup). `setMonobankSyncGapMsForTesting()` lets the test skip the real 61s pacing gap.
- **CSP**: none needed — client only calls same-origin `/api/monobank` (already covered by `connect-src 'self'`).

### Receipt scanning (OCR)
"Сканувати чек" button on the new-transaction form (`#receipt-scan-group`/`#fin-scan-receipt-btn`, hidden for transfers). Picks/photographs an image (native camera on mobile via `capture=environment`) and runs **Tesseract.js** (WASM, on-device OCR) to prefill amount + date — never blocks the form; an unreadable photo leaves fields as they were, with a toast.
- **Self-hosted, not CDN-loaded — a hard CSP requirement.** Tesseract.js's defaults pull from `cdn.jsdelivr.net`, but this app's `script-src` has no CDN allowance. Vendored into `js/vendor/tesseract/` (~9MB: `tesseract.esm.min.js`, `worker.min.js`, `tesseract-core-lstm.wasm.js` — LSTM-only, non-SIMD for universal compat — plus `lang-data/{ukr,eng}.traineddata.gz`), pulled via `npm pack` (registry.npmjs.org is reachable from this sandbox; jsdelivr/unpkg are not). **The separate `.wasm` binary file is not needed** — `tesseract-core-lstm.wasm.js` already embeds it inline as base64; confirmed via network-request logging during a real E2E run, then deleted (cut ~12MB→~9MB vendored footprint).
- **`corePath`/`workerPath`/`langPath` point at exact same-origin files, and `workerBlobURL:false` is required** — Tesseract.js's browser default spawns its worker via a `Blob`+`URL.createObjectURL()`, which this app's `worker-src 'self'` (no `blob:`) blocks; `workerBlobURL:false` forces the direct same-origin-URL path instead. Passing `corePath` as an exact `.js` file also sidesteps Tesseract.js's own SIMD-feature-detection branching, avoiding the need to vendor both variants.
- **CSP needed one addition: `'wasm-unsafe-eval'` on `script-src`** — `WebAssembly.instantiate()` is blocked without it (narrower than `'unsafe-eval'`, doesn't enable `eval()`/`new Function()`). No other CSP directive needed — image data never touches `img-src`/`connect-src`/blob URLs (read via `FileReader.readAsArrayBuffer()` internally).
- **Two different base paths for the same directory, on purpose**: the dynamic `import()` specifier (`./vendor/tesseract/...`) resolves relative to the importing module's own URL; the plain path strings Tesseract.js's `createWorker()` takes (`workerPath`/`corePath`/`langPath`) are resolved by the library relative to the page's URL (`/` on Firebase Hosting, `/Maxtr/` on GitHub Pages), so those need `./js/vendor/tesseract/...`. Mixing the two 404s silently.
- **Amount-extraction gotcha**: the fallback ("largest number in the text," when no total/сума keyword line matches) used to lose to a receipt's own date (`16.07.2026` contains a bare `2026`, numerically larger than the real total). `parseReceiptText()` masks the matched date span out before amount extraction, and prefers decimal-formatted numbers (`X.XX`) over bare integers. Locked in as a regression case in `tests/receipt-ocr-parse.mjs`.
- **Real phone photos (3000-4000px, several MB) can hang the non-SIMD WASM core with no JS-catchable error, and a long real scan has no UI feedback once the toast auto-hides** — both fixed: `downscaleImage()` (createImageBitmap + canvas, capped 1800px long side, JPEG) runs before every `recognize()`; `handleReceiptFile()` swaps the button's own label to a persistent "processing" state for the scan's duration; a hard 45s timeout (`withTimeout()`) rejects a genuinely stuck call, re-enabling the button with a `receipt_scan_timeout` toast. `scanReceiptImage()` takes an optional `timeoutMs` override for fast test verification (unused by the real call site).
- **CI/local test stubs the dynamic `import()` boundary via `page.route()`** rather than running real WASM OCR. **Needs `browser.newContext({serviceWorkers:'block'})`, not plain `newPage()`** — `sw.js`'s `clients.claim()` can claim the page before this test's late dynamic import fires, so its `fetch` handler intercepts the request before Playwright's route ever sees it (silently loading the real ~9MB library) — worth remembering for any future test whose stubbed fetch happens late in page load, not in the first moment.
- `js/vendor/tesseract/*` is **not** in `sw.js`'s `STATIC_ASSETS` (only reached via a dynamic `import()` inside a function body — eager-precaching ~9MB for a rarely-used feature would be wasteful); it's cached opportunistically on first real use. `js/receipt-ocr.js` itself (the wrapper module, statically imported) **is** in `STATIC_ASSETS`.
- Apache-2.0 licensed — `LICENSE-tesseract.js.md`/`LICENSE-tesseract.js-core.txt` kept alongside the vendored files.

### Notifications (two independent delivery paths, same three conditions)
Both paths check: daily "did you log anything today" reminder, budget-exceeded, upcoming-recurring-payment.
1. **Local** (`NOTIFICATIONS` section): fires only while the app is open, via the `Notification` API, checked on load/visibility-change/a 5-minute interval. Dedup via localStorage keys.
2. **Push** (`PUSH NOTIFICATIONS` + `functions/index.js`): client registers an FCM token (`push_tokens/{uid}`) via `getToken()` + `sw.js`. A scheduled Cloud Function (`notificationSweep`, hourly) re-implements the same three checks server-side and sends via the Admin SDK, with its own dedup fields on the token doc. **If you change the local-notification logic, check whether `functions/index.js` needs the equivalent change** — intentionally kept in sync but separate implementations.

`sendPush(token, title, body, type)` (`functions/index.js`) takes a `type` (`'daily'|'budget'|'recurring'|'debt'`) and looks it up in `NOTIF_ICONS` to set `webpush.notification.icon`. `sw.js`'s `onBackgroundMessage` reads `payload.notification.icon`, falling back to `icon-192.png`. The 4 icon PNGs (`notif-icon-{daily,budget,recurring,debt}.png`, repo root) are hand-generated (`cairosvg`, purple-gradient rounded-square background, white stroke-based glyphs matching `ICON_PATHS`' style — bell/warning/repeat/bank — supersampled 4× then Pillow `LANCZOS`-downscaled for smooth edges, plus a subtle `feDropShadow` on the glyph layer), same-origin so listed in `sw.js`'s `STATIC_ASSETS`. Covered by `tests/functions-sweep.mjs`'s `type='...'` assertions; the display side (`sw.js`) isn't automated — needs a real FCM background delivery, verify manually via a real device if this needs re-checking.

**`badge` (the small Android notification-shade glyph) is a separate image from `icon`, and needs a real alpha channel** — Android renders `badge` using *only* its alpha channel as a white silhouette; a fully opaque RGB PNG (like `icon-192.png`) renders as one solid white blob. `badge-96.png` (repo root, `cairosvg`-rendered from the same `bell` path data at 96×96, no drop shadow — it must stay a pure alpha-channel silhouette for Android's masking to work) is the correct asset. **If a new notification call site is ever added, give it `badge:'badge-96.png'`, never `icon-192.png` or any other opaque image.** The `maxtr-c238f.web.app` origin text shown next to "Rytm" in the notification header is Chrome/Android's own trust-indicator for web push and **not suppressible from app code** — goes away only once this ships as a real installed Android app via a published TWA. `sw.js` `CACHE_NAME` is currently `rytm-v72` — bump it whenever a same-origin precached asset (icon, manifest, etc.) changes, since browsers detect a new SW purely by byte-comparing the script.

### UI conventions worth following
- Custom-styled `<select>`/`<input type=date>`: call `enhanceSelect()` / `enhanceDateInput()` on it (or add it before `enhanceAllSelects()` runs) — don't hand-roll another dropdown.
- **Never use `<input type=color>`** — opens an unstylable native OS popup. Use the shared `openColorPicker(kind, id)` / `color-pick-modal` pattern instead — wire a new color-editable list by adding a case to `selectPickedColor()` and rendering `<button class="mgr-color" style="background:${color}" onclick="openColorPicker('kind','${id}')">`.
- No native `title="..."` tooltips or other unstylable browser affordances — prefer a visible label/subtitle in the UI, like the rest of the settings rows.
- **"Settings manager" modals** (wallets, categories, budgets, goals, recurring, tags, rules, rates, widgets, pin, premium, profiles, link-phone, avatar/color pickers) share one pattern: `.modal-overlay`+`.modal-card`, an `open*Manager()`-style function, and the shared `closeManagers()` that hides all of them by id — add new manager ids to that list or backdrop-click/close-all won't cover them. **Separately, every `.modal-overlay` (23 of them) already gets focus-trap/Escape-to-close/focus-restore for free** via `setupModalAccessibility()` (a `MutationObserver` on `style`, `AppState.openModalStack`) — automatic for any new `.modal-overlay`/`.modal-card` markup, no id-list registration needed, though it still needs `closeManagers()`'s list separately for backdrop-click. Covered by `tests/e2e-modals.mjs`.
- All icons are the custom monochrome SVG set (`window.Icon(name)`/`setIcon`) — no emoji, no icon fonts. Add new glyphs to `ICON_PATHS`. Check for duplicate icons within the same visible list/row-group before reusing a glyph (reusing across contexts for the *same* feature is fine). **`docs/material-symbols/` vendors Google's Material Symbols icon set (Outlined, weight 400) as an offline reference for drawing new glyphs** — not loaded/shipped by the app (`docs/` is excluded from both the Vite build and Firebase Hosting), just a local source of real path data to hand-adapt into `ICON_PATHS`. Most hand-drawn stroke icons stay stroke-based, but a Material Symbols source path is a solid filled shape — wrap it in `<g transform="translate(0,24) scale(0.025)">` (remaps its native `0 -960 960 960` viewBox into this app's uniform 24×24 one) with `fill="currentColor" stroke="none"` on the path, same as `repeat`/`pharmacy` already do. See `docs/material-symbols/README.md` for the full recipe. Confirmed with the account owner as an icon-library-only addition — not a Material Design 3 adoption (colors/typography/components untouched).
- **"Hide amounts" is an opt-in CSS class list, not a generic money-detector** — `filter:blur(7px)` on a fixed selector list (grep `body.amounts-hidden` in `index.html`) — **any new money display needs its own dedicated class added to that list, or it silently leaks.** Don't reuse a shared row/field class also used for non-money text (blurring it would hide the non-money part too) — give the money-only span its own class instead. Covered by `tests/hide-amounts.mjs`.
- **Calendar day cells show shifts as compact colored letter tokens** (`.shift-token`), not word-badges — `shiftCode(t)` prefers an explicit `code` field on the shift type (set on built-ins: Д/Д+/Н/Нр/Нп/В) and falls back to a derived first letter. Give a new built-in shift type a `code`.
- **Debt tab has a payoff-forecast card** (`#debt-forecast`, `renderDebtForecast()`): inline-SVG burndown chart + a payment-count-based pace estimate. Shown only with enough signal (startAmount set, ≥2 payments, positive downward trend). Pure-derived, no schema change.
- i18n: `I18N = {uk:{...}, en:{...}}` plus `tr(key)`; static markup uses `data-i18n`/`data-i18n-placeholder`, applied by `translateStaticDOM()`. No `data-i18n-title` — dynamic titles must be set in JS.
- The bottom nav (`.tabs`) has 5 tabs; `.tab-icon-wrap` is 42px (38px ≤600px). A 6th tab needs re-checking fit — no overflow/scroll handling on that bar.
- `sw.js`'s `fetch` handler must bail out (`return`, no `respondWith`) for any cross-origin request — `respondWith(fetch(...))` throws for some no-cors/redirect combinations (broke reCAPTCHA's CSP-report beacon once). Don't reintroduce a `respondWith` path for anything outside `self.location.origin`.

### Mobile UI redesign
A design review plus several rounds of live-phone-screenshot feedback drove a rework of the densest screens (multi-session, July 2026). These patterns are the established convention — follow them for anything new.
- **Hero metric + chip row** (`.hero-metric`/`.hero-balance` + `.chip-stat-row`/`.chip-stat`): one large primary number + a horizontal scrollable strip of small pill chips, replacing equal-weight `.stats-row` stat-card grids. Used on Finance/Shifts/Debt/Shopping. `.stat-card`/`.stats-row` still exist and are correct where weighting really is equal (Analytics tab's 4 metrics) — not deprecated.
- **Compact manager row** (`.budget-row`/`.rate-row`/`.cat-row`): single line, icon badge + ellipsis-truncated name (needs `min-width:0` on *every* ancestor in the flex chain, not just the text element, or a nowrap row can overflow its own card) + right-aligned input/value. Auto-rules and recurring managers use the same collapsed-summary-row-with-pencil-toggle-to-expand shape (`AppState.expandedRuleId`/`.expandedRecurringId`). **`.mgr-num` gotcha**: `flex:1 1 90px;width:90px` inside a `flex-direction:column` parent (`.mgr-field`) applies `flex-basis` to the *vertical* axis, silently making the input 90px tall too (a solid near-black square) — fixed by dropping the `flex` shorthand, keep `width:90px` alone.
- **FAB + bottom-sheet modal** (`.fin-fab` + `#tx-form-modal`): the Finance "Нова операція" form is a modal opened by a floating `+` button, not a permanently-expanded inline form. Apply the same pattern to any future "always-expanded form on a primary tab."
- **PWA shortcuts + Web Share Target**: `manifest.json`'s `shortcuts` (long-press → "Нова операція") and `share_target` (GET-based) both land on `js/app-init.js`'s `handleLaunchParams()`, called last in `init()` (after `fbLoadNow()`). Reads `URLSearchParams`, strips the query via `history.replaceState`, clicks the same `[data-action="open-new-tx-modal"]` FAB. A shared `title`+`text` prefills the comment and, best-effort, the amount (first plain number found). Covered by `tests/pwa-launch-params.mjs`.
- **Swipe-to-delete list rows** (Telegram/Gmail-style): the pattern for any destructive delete action — Finance tx list (`.tx-item`/`setupTxSwipe()`) and Debt payment rows (`.debt-row`/`setupDebtRowSwipe()`), intentionally duplicated per-file rather than shared. Outer element is a `position:relative;overflow:hidden` clipping shell; the visible card is an `*-inner` child that shrinks its own `width` (never `transform:translateX` — clips left-edge content) to reveal a red delete button behind it. **Flexbox gotcha**: a flex item with `overflow` other than `visible` gets an automatic minimum size of 0 — a swipe-clipped row inside a `display:flex;flex-direction:column;overflow-y:auto` list needs an explicit `flex-shrink:0` or it silently crushes to a sliver once content exceeds `max-height` (hit both `.debt-row` and `.tx-item`; also silently broke pull-to-refresh, since a non-overflowing container reads as non-scrollable). Both are covered by geometry regression assertions (`tests/stress-tx-list.mjs`, `tests/debt-swipe-delete.mjs`). **Give any new swipe-clipped row `flex-shrink:0` and its own `border-radius` matching the row** (the outer clip only rounds the far edge for free).
- **Floating pill bottom nav**: `.tabs` is a detached rounded capsule, not edge-flush. `.app`'s bottom padding (112px) and `.fin-fab`'s offset (104px) are tuned to this bar's height+gap — re-check both if the bar changes.
- **Icon-based avatars, not emoji**: `BUILTIN_AVATARS` uses `window.Icon(a.icon)`. Exactly three render call sites (`renderAvatarPicker`, `renderProfileAvatarPickerGrid`, `renderProfilesUI`) plus `fillAvatarEl` — when this array's shape changes, grep `BUILTIN_AVATARS` **and** the field name itself (`.emoji`/`.icon`), not just the array name, to catch every destructured/aliased read.
- **Collapsed-by-default secondary panels**: e.g. Shifts' "Швидке заповнення" quick-fill, collapsed by default (`toggleQuickFill()`). Don't reuse `min-width:0` on a `.cs-sm` custom-select without either a real minimum or a guaranteed-wrapping container — it can shrink below its own text's width instead of wrapping.
- **Haptic feedback**: `showToast(msg, icon)` fires `navigator.vibrate(...)` (feature-detected) automatically — no need to call it separately.
- **Accessibility touch targets**: `.filter-chip`/`.settings-group-chip`/`.debt-chip`/`.subcat-toggle` and `.mgr-color`/`.mgr-del`/`.mgr-edit`/`.debt-chip-del` bumped to clear ~44px (later reconciled against the real WCAG 2.2 AA floor, SC 2.5.8, which is a more lenient 24×24 CSS px — most "left out" items already clear that). `--muted2` contrast raised to `#98979e` dark / `#626269` light (was under 4.5:1 AA for normal text). `.subchip button`'s small "×" got an asymmetric hit-slop pseudo-element instead of a size bump (safe there specifically because of the safe zone on its trailing edge).
- **Font sizes**: every sub-12px `font-size` was raised to 12px, then (separate pass, direct account-owner feedback that content still read small) the whole small-text scale shifted +1px again: `12px→13px`, `13px→14px`, `14px→15px` (and half-steps), `15px`+ untouched. **Current baseline — check the live value in the file, not any older paragraph's px numbers.**
- **Pull-to-refresh**: `setupPullToRefresh()` (`js/app-init.js`) — drag down from `scrollTop===0` reveals `#ptr-indicator`, snaps back under a 70px threshold or triggers the same `fbLoadNow()`+toast the topbar refresh button uses. Relies on `body{overscroll-behavior-y:contain}` to suppress native pull-to-refresh, not JS `preventDefault()` (uses `{passive:true}`). Covered by `tests/pull-to-refresh.mjs`.
- **Sheet-grabber drag-to-dismiss** (`initSheetDrag()`, `js/ui-widgets.js`): `.sheet-grabber` is a full-width 28px-tall hit-box (visible bar is a `::after`), not just the visible bar itself — a narrow hit-box caused inconsistent-feeling drags from the sheet's rounded corners. Also dismisses from within `.modal-card-body` itself for content-heavy sheets: a downward drag only counts as dismiss if the first unambiguous movement (past 8px slop) is downward **and** the body was already at `scrollTop===0` — any other case is left alone with no `preventDefault()`, so normal scrolling is untouched. Covered by `tests/sheet-drag-dismiss.mjs`.
- **Empty-state illustrations**: `emptyStateHtml()` (`js/ui-widgets.js`) wraps the monochrome icon in a larger radial-gradient glow circle + floating accent dots, built from the app's own icon set/palette (no emoji/external artwork), plus a `prefers-reduced-motion`-aware entrance animation. All 5 existing call sites picked this up automatically.

### Android / Google Play packaging (TWA)
Ships as a **Trusted Web Activity (TWA)** — a thin Android wrapper opening the live `maxtr-c238f.web.app` full-screen, not a separate native codebase. No local Android project checked into this repo; generated on demand.
- **Canonical domain is `maxtr-c238f.web.app`** (not GitHub Pages — Play's Digital Asset Links verification needs one fixed origin with no subpath).
- **`.well-known/assetlinks.json`** (repo root) lets Android show the app without a URL bar — declares the package ID + signing cert's SHA-256 fingerprint, verified against `https://maxtr-c238f.web.app/.well-known/assetlinks.json`. `firebase.json`'s hosting `ignore` has an explicit `"!/.well-known/**"` un-ignore (dotfiles are excluded by default) plus a `Content-Type: application/json` override. If the fingerprint and the actual signing key ever drift, the app degrades to a Chrome URL bar rather than failing outright — fix is "update the fingerprint, redeploy hosting."
- **Package ID is `ua.rytm.app`** — freely changeable up until the *first* real Play Store upload (hasn't happened yet); after that, changing it means a new listing, not an update.
- **The actual APK/AAB build cannot happen from a Claude Code cloud sandbox** — this environment's egress proxy 403s `dl.google.com`, `pwabuilder.com`, and `maxtr-c238f.web.app` itself. What works offline from the sandbox: generating the Android Studio project skeleton via `@bubblewrap/core`'s `TwaManifest`/`TwaGenerator` directly, and generating the signing keystore with the local JDK's `keytool`. The actual compile has to happen either in Android Studio on a real machine, or — the path actually used — **pwabuilder.com** from the user's own browser, pointed at `https://maxtr-c238f.web.app`.
- **Signing keystore is intentionally not committed** — generated once (`keytool -genkeypair`, RSA/3072, PKCS12) and handed to the account owner as a downloadable file; losing it means never publishing an update under `ua.rytm.app` again. If it ever needs regenerating, `.well-known/assetlinks.json`'s fingerprint must be updated and hosting redeployed to match.
- **PKCS12 keystore gotcha**: `keytool` on JDK 9+ defaults to PKCS12, which does **not** support a separate private-key password — `-keypass X -storepass Y` with `X != Y` silently succeeds and just reuses the store password, printing only a warning. A consumer later told "the key password is X" fails with `javax.crypto.BadPaddingException` (reads like corruption, not a wrong password). **Always generate PKCS12 signing keys with the same value for both `-storepass` and `-keypass`**, and only document/hand off one password.
- **`android-icon-rounded.png`** (repo root, Pillow squircle mask of `icon-512.png`) is pasted into PWABuilder's Android package form "Icon URL" field only — not referenced in `manifest.json`. The TWA splash screen centers the raw icon with no adaptive-icon masking, so the hard square corners looked jarring there specifically; the form's separate "Maskable icon URL" field must keep pointing at the original full-bleed `icon-512.png`.
- **`icon-192-rounded.png`/`icon-512-rounded.png`** (repo root) solve the same problem for desktop-PWA-install (Windows shortcuts): `manifest.json`'s `icons` array has 4 entries — `icon-*-rounded.png` at `"purpose":"any"`, original `icon-*.png` at `"purpose":"maskable"` only (a combined `"any maskable"` purpose was contradictory — "any" wants a pre-rounded source, "maskable" wants full-bleed square). Both rounded files are in `sw.js`'s `STATIC_ASSETS`. `apple-touch-icon`/notification icons deliberately still point at the original square `icon-192.png` (iOS applies its own squircle mask; notification-tray icons are too small for rounding to matter).
- Rebuilding the Android package after a manifest/icon change is manual: pwabuilder.com → `https://maxtr-c238f.web.app` → Package For Stores → Google Play → fill Package ID/Icon URL/Maskable icon URL → Signing key → "Mine" → upload the existing `.jks` + its one password → Download. No CI/automation for this.
- **Google Play Console developer account created, pending Google's own verification review** — nothing can be uploaded until it clears (Google's queue, no action available here). Once cleared: finish the Store listing (screenshots, feature graphic, Data safety form, content rating), upload the already-generated `.aab` to Internal testing, then Production — see `SETUP.md` §9.
