# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

**Zminka** (formerly "Xamss") is a static PWA: a personal tracker for work shifts, multi-currency finances, and debt/settlement schedules. It's Ukrainian-first (`uk`/`en` toggle), backed by Firebase (Auth + Firestore + Cloud Messaging + one scheduled Cloud Function), and deployed as static files via Firebase Hosting ‚Äî there is no build step, bundler, or frontend framework; native browser ES modules do the file splitting instead. `index.html` holds the markup/CSS/two small classic scripts; almost all app logic lives under `js/*.js`, loaded via `<script type="module" src="./js/app.js">` (see "`js/` module layout" below). It used to be one 7000-line inline `<script type="module">` ‚Äî split apart specifically so a change to (say) the debt tab doesn't require reading through finance/calendar/settings code to get there; see that section for why this was safe to do without a bundler and what to watch for when editing across files.

## Commands

There is no root `package.json`, no bundler, and no linter ‚Äî this is intentional, not an oversight. Treat the app as plain static HTML/CSS/JS. There are two persisted automated checks, `tests/smoke.mjs` and `tests/unit.mjs` (see below), run in CI on every push/PR via `.github/workflows/smoke-test.yml`; everything beyond their coverage is still verified manually with the same recipe.

- **Run locally**: serve the repo root with any static file server (e.g. `python3 -m http.server`) and open `index.html`. Firebase Auth/Firestore calls will hit the real `maxtr-c238f` project unless you intercept the `firebasejs` module imports.
- **Unit tests**: `node tests/unit.mjs` ‚Äî plain-node tests (no browser, no Firebase) for the pure helpers in `functions/lib/pure.js` (currency conversion, timezone-aware date/hour resolution used by the push-notification sweep). These were split out of `functions/index.js` specifically so they're importable without firebase-admin credentials, which `index.js`'s top-level `initializeApp()` call needs.
- **Smoke test**: `node tests/smoke.mjs` ‚Äî a self-contained script that serves `index.html`, stubs the Firebase SDK, signs in, dismisses onboarding, and checks that all 5 tabs render with no console/page errors. No real Firebase project or network access needed (playwright is imported by its absolute sandbox install path since it's a global install, not a project dependency ‚Äî see the script's header comment). In CI (no `/opt/pw-browsers/chromium`), it falls back to Playwright's own installed browser instead ‚Äî see `CHROMIUM_PATH` in the script. This is a smoke test, not full coverage ‚Äî verify anything beyond its scope manually with the same recipe below.
- **CI**: `.github/workflows/smoke-test.yml` runs on every push to `main` and every PR ‚Äî installs Playwright globally (no root `package.json` to declare it as a dependency, matching the sandbox setup above), then runs `tests/unit.mjs`, `tests/smoke.mjs`, and a `node --check` syntax pass on `functions/index.js`.
- **Firestore rules test**: `tests/firestore-rules.mjs` ‚Äî not run in CI (needs a JDK + the Firestore emulator + two npm packages this repo doesn't vendor), but run this manually before *any* `firestore.rules` change; see the script's header for the exact recipe. It asserts against a real emulator, not a read-through of the rules file ‚Äî it caught a real bug during development (two `allow write` blocks for the same match, where Firestore's OR-of-all-`allow`-statements semantics made the stricter one a silent no-op) that looked correct on inspection.
- **Verify a change**: for anything the smoke test doesn't cover, the established approach in this repo is to drive `index.html` with Playwright against a **stubbed Firebase SDK** ‚Äî intercept the three `https://www.gstatic.com/firebasejs/.../firebase-{app,firestore,auth}.js` (and `firebase-messaging.js` if touching push) module imports via `page.route()` and fulfill them with minimal hand-written stub modules, then exercise the UI with real clicks/fills and assert on DOM state. This lets you test auth-gated flows without network access or real credentials. Always also do a syntax check before testing, since `<script type="module">` bodies aren't validated by naive `node --check` on the whole file ‚Äî extract the module script's contents with a regex and check that in isolation.
  - Operational notes from actually running this recipe in this repo's sandbox: serve via `python3 -m http.server` (not `file://` ‚Äî ES module imports of the Firebase gstatic URLs hit CORS issues over `file://`); `playwright` is installed globally, not as a project dependency, and Node's ESM resolver (unlike `require()`) doesn't consult `NODE_PATH`, so import it by its absolute path (`/opt/node22/lib/node_modules/playwright/index.mjs`) rather than a bare `import ... from 'playwright'`; launch Chromium with `executablePath: '/opt/pw-browsers/chromium'`. Sign-in is simulated by making the stubbed `onAuthStateChanged(auth, cb)` fire `cb({uid, email})` itself (asynchronously ‚Äî see the TDZ gotcha below) rather than calling any app-exposed sign-in function; there is no `window.__authSignIn` in this codebase. Dismiss the first-run onboarding overlay (`window.finishOnboarding()` or click `#onboard-skip`) before interacting with tabs, or clicks land on the overlay instead.
  - Stub auth needs to export whatever `firebase-auth.js` imports actually are, including ones not exercised by the specific test at hand (`linkWithPhoneNumber`, `unlink`, etc.) ‚Äî a missing export fails the whole module import and blanks the page, not just the untested code path.
  - `updateRatesOnline()` auto-fires on load to fetch live NBU exchange rates from a real external API and fails silently in any sandbox without a route to that host ‚Äî a resulting `net::ERR_*`/"live rates fetch failed" console error is expected sandbox noise, not a regression, when asserting on "no console errors."
- **Cloud Function deploy**: `cd functions && npm install && firebase deploy --only functions,firestore:rules` from a machine with `firebase-tools` authenticated against the project (Blaze plan required). See `SETUP.md` for the full one-time Firebase Console setup this depends on (enabling sign-in providers, Cloud Messaging VAPID key, required GCP APIs).
- **Hosting deploy**: `firebase deploy --only hosting --project maxtr-c238f` from a git worktree checked out at `origin/main` (deploy from a clean merged tree, not a local feature branch). This is a **separate IAM permission** from the Cloud Functions deploy service account below ‚Äî expect a first-time 403 and need to grant `roles/firebasehosting.admin` before it works. Note: **GitHub Pages is also enabled on this repo** (Settings ‚Üí Pages, "Deploy from a branch" ‚Üí `main` / root) and auto-deploys on every push to `main` with no agent action needed ‚Äî `https://sudak666.github.io/Maxtr/` and `https://maxtr-c238f.web.app` are both live production mirrors serving the same code; only the Firebase Hosting one needs a manual deploy step.

## Architecture

### Layout
- `index.html` ‚Äî markup, CSS, and two small classic (non-module) `<script>` blocks; loads `js/app.js` as the app's actual logic (see "`index.html` script structure" below).
- `js/` ‚Äî the app's logic, split into 16 native ES modules (no bundler ‚Äî the browser resolves the relative `import`s directly). See "`js/` module layout" below for the file list, the shared-state pattern, and gotchas for anyone splitting a file further or adding a new cross-file call.
- `functions/` ‚Äî a separate Node 20 npm project (Cloud Functions v2 / `firebase-admin` + `firebase-functions`), deployed independently of the static site.
- `firebase.json` / `firestore.rules` / `manifest.json` ‚Äî Firebase Hosting, Firestore security rules, and PWA manifest.
- `sw.js` ‚Äî the single service worker, handling **both** PWA asset caching (network-first for HTML, cache-first for everything else) **and** Firebase Cloud Messaging background push display. It used to be split into `sw.js` (caching) + a separate `firebase-messaging-sw.js` (push), both registered at the site's root scope ‚Äî but a single scope can only ever have one active service worker, so registering `firebase-messaging-sw.js` (which had no `fetch` handler) after `sw.js` silently replaced `sw.js` at that scope and disabled offline caching the moment a user enabled push. Merged into `sw.js` alone (bumped to v4) to remove the conflict; there is no `firebase-messaging-sw.js` file anymore ‚Äî don't recreate it as a separate registration. `sw.js`'s `FIREBASE_SDK_ASSETS` also precaches (and cache-first-serves) the exact four gstatic SDK URLs `index.html`'s `<script type="module">` statically imports ‚Äî a static ES module import is all-or-nothing, so without this, a cold start with no network (even with `index.html` itself already cached) would fail the whole module and leave the app inert past its HTML shell. This is a narrow exact-URL allowlist inside the fetch handler's normal cross-origin bail-out, not a general override ‚Äî if the pinned SDK version in `index.html`'s imports ever changes, update `FIREBASE_SDK_ASSETS` in `sw.js` to match (and bump `CACHE_NAME`). `STATIC_ASSETS` also explicitly lists every file under `js/` for the same reason (same-origin this time, so no allowlist trick needed ‚Äî they're precached alongside `index.html` itself) ‚Äî add any new `js/*.js` file to this list too, or a cold start offline can fail to load it.
- `privacy.html` / `terms.html` ‚Äî standalone static pages, not part of the SPA.
- `SETUP.md` ‚Äî the manual, non-code Firebase Console setup required before auth/push/phone-sign-in work in production (enabling sign-in providers, authorized domains, deploying Firestore rules, VAPID key, required GCP APIs for the scheduled function). Keep this in sync whenever a change adds a new external dependency.

### `index.html` script structure
Three `<script>` blocks, in load order:
1. Two tiny inline classic scripts near the top: apply the saved theme before first paint, and register `sw.js`.
2. A classic (non-module) `<script>` containing the custom monochrome SVG icon library (`ICON_PATHS`, `window.Icon(name)`, `setIcon(id, name)`) plus the I18N object/`window.tr()`/`window.translateStaticDOM()`/`window.setLang()` and theme toggle wiring. Runs before the module script. **This is genuinely outside the `js/` module graph** ‚Äî `tr`/`Icon`/`setIcon`/`translateStaticDOM` are real globals (via `window`), not something any `js/*.js` file imports; `window.__applyLangDynamic` is the one hook back the other way (`js/app-init.js` assigns it so `setLang()` can re-render dynamic content after a language switch ‚Äî grep for it before assuming the two script blocks don't talk to each other at all).
3. `<script type="module" src="./js/app.js">` ‚Äî loads the whole `js/` module graph (see below). Functions called from inline `onclick`/`onchange` HTML attributes must be attached to `window` explicitly (`window.foo = function(){...}`) since module-scope declarations aren't global ‚Äî this is why you'll see both a bare `function foo(){}`/`const foo = function(){}` and a `window.foo=foo;` line for anything wired to markup. **This bit us for real**: a function only called from an inline `onclick` attribute (not from other JS) must be `window.foo`, even a "just renders a list" helper ‚Äî `renderProfilesUI` was defined as a bare module-scope `function` and worked fine everywhere it was called from other JS, but silently threw `ReferenceError` every time the one inline `onclick="...; renderProfilesUI()"` fired, until it was also assigned to `window`.

### `js/` module layout
`index.html` used to be one 7000-line inline `<script type="module">`; it's now 16 files under `js/`, split along the same `// ‚îÄ‚îÄ SECTION ‚îÄ‚îÄ` banners that organized it before (`state.js`/`core.js`/`firebase-sync.js`/`color-picker.js`/`auth.js`/`app-init.js`/`ui-widgets.js`/`calendar.js`/`settings-managers.js`/`goals-profile.js`/`notifications.js`/`finance.js`/`analytics-csv.js`/`debt.js`/`shopping.js`, plus `app.js` as the entry point). This was a mechanical, AST-verified transformation (parsed with `espree` + `eslint-scope`, not manually traced), not a rewrite ‚Äî the actual code inside each function is unchanged. Three things make cross-file JS in this app different from a typical ES-module codebase, and matter for anyone adding a new cross-file reference or splitting a file further:

1. **All mutable app state lives in one shared object, `AppState` (`js/state.js`)** ‚Äî `transactions`, `wallets`, `shifts`, `currentUser`, `activeProfileId`, all ~60 of what used to be individual top-level `let` bindings, are now `AppState.transactions`, `AppState.wallets`, etc. **Why one object instead of `export let x` per field**: ES modules only let the module that declared a binding reassign it, but many of these fields are reassigned from several different files ‚Äî e.g. `fbLoadNow()` (`js/firebase-sync.js`) resets almost all of them, `switchProfile()`/`fbLoadNow()` write `AppState.transactions` from one file while `addTransaction()`/`deleteTransaction()` (`js/finance.js`) write it from another. A plain property write (`AppState.x = value`) has no such restriction; `export let x` per field would. **`js/state.js` has zero imports of its own, deliberately** ‚Äî it's the one file every other file depends on, so if it depended on anything back, that circular edge risks a temporal-dead-zone `ReferenceError` on its own top-level `const AppState = {...}` initialization (this happened during development: `AppState`'s `widgetOrder`/`MONTHS`/`WEEKDAYS` fields needed `WIDGET_ORDER_DEFAULT`/`LANG_CALENDAR`, which briefly lived in `core.js` ‚Äî which itself transitively imports `AppState` ‚Äî and that circular edge threw). Both constants were moved into `state.js` itself instead. **If you add a new module-level mutable field, put it in `AppState`, not a fresh top-level `let`** ‚Äî a bare new `let` works fine within its own file but silently can't be reassigned from anywhere else without going through this same problem.
2. **Every chunk that does something immediately at the top level (not inside a function) has that code wrapped in an exported `__init_<chunk>__()` function, called from `app.js` after all 16 files have finished importing.** The original single file ran strictly top-to-bottom, so e.g. `const auth = getAuth(fbApp)` (near the top) had always already run by the time a much later `onAuthStateChanged(auth, ...)` line executed. Once split across files with a real (partly circular) import graph, module evaluation order is not guaranteed to match the original file's order ‚Äî an unwrapped top-level statement in one file can run before another file's declaration it depends on has executed, which is a temporal-dead-zone crash (hit during development with exactly this `auth`/`onAuthStateChanged` pair). **If you add new top-level side-effecting code to one of these files** (a `document.addEventListener(...)` call, a `setInterval(...)`, anything that isn't a `function`/`class`/`const`/`let` declaration), put it inside that file's `__init_X__()` function, not bare at the top level, or it risks running before some other file it depends on has initialized.
3. **A name only ever exposed as `window.foo = function(){...}` (no separate bare `function foo(){}`) still needs a real export/import if another file references the bare identifier `foo`** ‚Äî this looks like it should be safe to skip (relying on `window` being shared across every script on the page) and an earlier version of this split tried exactly that, but it breaks for the specific case of the `window.foo = foo;` assignment statement itself: that line needs `foo` to already be a correctly-resolving lexical binding in the file it's written in, and "resolves via `window` later" doesn't help the statement that's supposed to be *setting* `window.foo` in the first place. Every `window.foo = function(){...}`-only name was rewritten to a real `const foo = function(){...}; window.foo = foo;` pair during the split for exactly this reason (except `closeManagers`, which already had a separate same-named bare wrapper function with different code ‚Äî see that function if you need the exact shape). **When adding a new function only meant to be called from inline HTML (`onclick=`), still give it a real top-level declaration** (`function foo(){}` or `const foo = function(){}`) in whichever file it lives in, plus the `window.foo = foo;` line ‚Äî don't write `window.foo = function(){...}` as the only place `foo` is defined, since any other `js/*.js` file that later needs to call it by name (not just via `onclick=`) will need to `import` that declaration, and there won't be one to import.

Verification for this split (documented for whoever next needs to add a 17th file or further split an existing one): `tests/smoke.mjs` + a manual Playwright pass exercising a transaction add, opening a settings modal, a language switch, and a shift-detail modal, all with zero console/page errors ‚Äî see git history around the module-split commit(s) for the exact verification script if you need to reproduce it. The AST-analysis + code-generation scripts used to produce this split were one-off tooling (not committed ‚Äî they read a since-deleted single-file snapshot of the module script) rather than a repo-maintained codegen step; treat `js/*.js` as hand-edited source from here on, the same as `index.html` always was.

### Firebase data model
- Firestore layout: `users/{uid}/max_tracker/{doc}` where `{doc}` is one of `shifts`, `finance`, `debt`, `backup_v2` (a one-time raw snapshot taken whenever config gets re-seeded), `profiles_meta` (see Multiple profiles below). `DCOL = 'max_tracker'`; `userDoc(name)` builds the path. **The Firestore rule for this collection is no longer a full wildcard** ‚Äî `firestore.rules` matches a single path segment (`match /users/{uid}/max_tracker/{document}`, not `{document=**}`, since there are no subcollections here) and requires `document` to match `^(shifts|finance|debt|backup_v2)(@[A-Za-z0-9_]+)?$|^profiles_meta$`, plus shallow validation (top-level key count, `updatedAt`/`at` must be numbers if present ‚Äî see the rules file's own header comment for why this stops short of a full per-field schema). **Adding a new doc *base name* now needs a matching `firestore.rules` change**, unlike before; a new `@<profileId>`-suffixed variant of an existing base name needs none. Before touching this file, run `tests/firestore-rules.mjs` (see Commands above) ‚Äî it asserts against a real emulator and has already caught a rules bug (duplicate `allow write` blocks silently canceling validation via OR semantics) that a read-through missed. A legacy pre-auth shared `max_tracker/*` collection is explicitly denied in `firestore.rules` ‚Äî do not resurrect it.
- The `finance` doc is the big one: transactions, wallets, categories, budgets, subcategories, currencyRates, tags, autoRules, recurring, `shoppingList`, goals, profile, subscription, widgets, **and `notifSettings`** (synced here specifically so the server-side Cloud Function can see each user's notification preferences ‚Äî it used to be localStorage-only; now also carries `notifSettings.timeZone`, an IANA zone name captured client-side for the push sweep's local-time calculations).
- `push_tokens/{uid}` is a separate top-level collection (not nested under `users/{uid}`) holding `{token, updatedAt, sentDaily, sentBudget, sentRecurring, profileState}` for the FCM push sweep; it has its own Firestore rule scoped to `request.auth.uid == uid`, plus type/size checks on the client-writable `token`/`updatedAt` fields (not a full key allowlist, since the Cloud Function's Admin SDK writes its own dedup fields onto the same doc, bypassing rules entirely).
- Client-side caching/sync: every piece of state is mirrored into `localStorage` under per-user keys via `lsKey(name)` ‚Üí `mx_<name>_<uid>` (loaded first for instant paint), then reconciled against Firestore in `fbLoadNow()`. Writes go through `scheduleSave()` (debounced) ‚Üí `fbSaveNow()`, which uses optimistic concurrency (`lastKnownUpdatedAt` per doc, compared against each doc's `updatedAt`) and prompts the user on conflict rather than silently overwriting another device's newer save. A few internal localStorage keys still carry the old `xamss*` prefix from before the Zminka rename (e.g. `xamssTheme`, `xamssLang`) ‚Äî these were left as-is intentionally since renaming them would silently reset existing users' local prefs for zero benefit.

### Multiple profiles per account
One Firebase Auth account can hold several independent data-profiles (e.g. "–Ø"/"–î—Ä—É–∂–∏–Ω–∞") with their own finances/shifts/debts, switchable in Settings ‚Üí –ü—Ä–æ—Ñ—ñ–ª—ñ.
- `activeProfileId` (module-level, default `'default'`) drives both `userDoc(name)` and `lsKey(name)`: for the **default** profile these resolve to the original unsuffixed doc names/localStorage keys (zero migration, existing users are untouched); for any other profile, `userDoc` suffixes the doc name `"@<profileId>"` (e.g. `finance@abc123`) and `lsKey` suffixes the localStorage key the same way. Because every save/load path already funnels through these two functions, nothing else needed to change to make the whole app profile-aware.
- `profiles_meta` (account-wide, never suffixed even when a non-default profile is active) holds `{list:[{id, name, avatar, createdAt}], updatedAt}` ‚Äî just labels/avatars for the switcher, not real data. Each profile's actual identity (nickname + avatar shown in Settings ‚Üí –ü—Ä–æ—Ñ—ñ–ª—å and the topbar) lives inside *that profile's own* `finance` doc, since `profile.nickname`/`profile.avatar` is per-profile automatically once `finance` is suffixed.
- `activeProfileId` itself is a **per-device** choice (`localStorage['mx_activeProfile_'+uid]`), not synced ‚Äî switching profiles on one device doesn't affect others.
- `switchProfile(id)` flushes the current profile (`fbSaveNow()`) before reassigning `activeProfileId` and calling `fbLoadNow()` again ‚Äî this reuses all the existing seeding/rendering logic verbatim, so a brand-new profile gets the same default wallets/categories a brand-new account gets.
- The scheduled Cloud Function (`functions/index.js` `notificationSweep`) reads `profiles_meta` and sweeps every profile on the account (default + suffixed), not just the default one ‚Äî each profile's `finance`/`finance@<id>` doc is checked independently. Per-profile dedup state lives in `push_tokens/{uid}.profileState.<profileId>` (`{sentDaily, sentBudget, sentRecurring}`); pre-existing token docs' flat `sentDaily`/`sentBudget`/`sentRecurring` fields are read as the default profile's initial state until the sweep writes `profileState.default` for the first time, so old single-profile users need no migration. Local in-app notifications work fine for whichever profile is active since they read the same in-memory state that switching already repopulates.
- Any render function that bails out early when its collection is empty (`if(!items.length) return;`) needs to explicitly clear its previously-rendered DOM first ‚Äî otherwise switching to a profile with no data leaves the *previous* profile's numbers on screen instead of blanking them (this bit us once with the Debt tab; check for the same pattern before adding another profile-scoped collection).

### Auth
Three sign-in methods, all in the AUTH / PHONE SIGN-IN sections: email+password, Google popup, and phone number (SMS code via `signInWithPhoneNumber` + an invisible `RecaptchaVerifier`). Layered on top, independent of Firebase Auth: a local PIN (SHA-256 hash in localStorage, `mx_pin_<uid>`) and optional WebAuthn biometric unlock ‚Äî these gate the UI locally but never replace the Firebase session.

Settings ‚Üí –ê–∫–∞—É–Ω—Ç additionally lets an already-signed-in user (e.g. one that signed in via Google) **link/unlink a phone number** as an extra credential (`linkWithPhoneNumber(currentUser, ...)` / `unlink(currentUser,'phone')`) ‚Äî a separate flow from the pre-auth sign-in one, with its own `link-recaptcha-container`/`RecaptchaVerifier` instance so it doesn't collide with the lock-screen's.

**reCAPTCHA retry gotcha**: `RecaptchaVerifier.clear()` can itself throw partway through on a failed attempt, leaving the widget's DOM node behind; the next attempt's fresh `RecaptchaVerifier.render()` then finds a stale widget already in the container and throws `"reCAPTCHA has already been rendered in this element"`. Both phone flows now force the container's `innerHTML` empty on error (`resetRecaptchaContainer(containerId)`) regardless of whether `.clear()` succeeded ‚Äî do this for any new reCAPTCHA-backed retry flow too.

**Phone auth troubleshooting** (not code issues, but easy to mistake for one): (1) needs the **Blaze** plan (documented below already); (2) needs **Authentication ‚Üí Settings ‚Üí SMS region policy** to explicitly allow the country being tested (Firebase blocks SMS to unlisted regions by default); (3) **Brave browser disables Web Push by default** (`brave://settings/privacy` ‚Üí "Use Google services for push messaging") ‚Äî causes `AbortError: Registration failed - push service error` on `getToken()`/`PushManager.subscribe()` that looks like a bug but isn't; (4) **GitHub Pages mirror isn't auto-authorized** ‚Äî `*.web.app`/`*.firebaseapp.com` are automatically in Firebase's Authorized domains, but `sudak666.github.io` is not, so invisible reCAPTCHA fails (flashes and vanishes) and phone sign-in/link dies with a generic error specifically on that mirror while working fine on `maxtr-c238f.web.app`. Add `sudak666.github.io` to Authentication ‚Üí Settings ‚Üí Authorized domains manually (see SETUP.md ¬ß2). `authErrorMessage()` in `index.html` now maps `auth/captcha-check-failed`, `auth/unauthorized-domain`, and `auth/network-request-failed` to specific messages instead of the generic fallback, to make this diagnosable from the UI itself next time; (5) **`auth/error-code:-39` (503 on `sendVerificationCode`) for every number tried** ‚Äî not billing, not the API key, not the region policy, not a specific number (confirmed by process of elimination and then by Firebase Support directly, case 10414309, July 2026): Firebase's automated anti-toll-fraud system has temporarily blocked the carrier/region for low SMS-success rates. `authErrorMessage()` maps this (and `auth/internal-error`) to a dedicated message, scoped to phone-flow call sites only via an `isPhoneFlow` param ‚Äî see SETUP.md ¬ß5 for the real fix (enable reCAPTCHA SMS Defense, which is why the Firebase JS SDK is pinned to `12.11.0` rather than `10.x`: SMS Defense requires Web SDK 11+).

### Premium / free-tier limits
`subscription = {plan, expiresAt}` lives in the `finance` doc. `FREE_LIMITS`, `canAddMore(kind, count)`, and `showPremiumUpsell(kind)` still exist (kept for when a payment provider ships), but **no "add" action currently calls `canAddMore`** ‚Äî the wallets/categoriesPerType/autoRules/recurring/goals caps were deliberately removed, because with no way to actually purchase premium, a hard cap was just a dead end for every user (this is what was silently blocking "add category" for any account with ‚â•8 categories ‚Äî a brand-new account already starts at exactly 8 default expense categories). If a payment provider gets wired up, the caps should come back by re-adding the `if(!canAddMore(...)){ showPremiumUpsell(...); return; }` guard at the top of each `add*()` function (see git history around the "Remove remaining free-tier caps" commit for the exact call sites) rather than reinventing the gating pattern.

### Finance-tab widgets
`widgets = {rates, converter, analytics, chart, goals}` (synced in the `finance` doc) toggles which optional sections render on the Finance tab; `applyWidgetVisibility()` is the single place that reconciles the toggle state with the DOM. Adding a new toggleable widget means updating this object's default in every place it's read (there are a few: the Firestore doc load, the localStorage cache load, and the in-memory default) ‚Äî grep for `widgets.goals` to find all of them.

### Notifications (two independent delivery paths, same three conditions)
Both paths check: daily "did you log anything today" reminder, budget-exceeded, and upcoming-recurring-payment.
1. **Local** (`NOTIFICATIONS` section): fires only while the app is open, via the `Notification` API, checked on load / tab-visibility-change / a 5-minute interval. Dedup uses localStorage keys.
2. **Push** (`PUSH NOTIFICATIONS` section + `functions/index.js`): the client registers an FCM token (`push_tokens/{uid}`) via `getToken()` + `sw.js` (see Layout above ‚Äî `sw.js` now also carries the FCM background-message handler). A scheduled Cloud Function (`notificationSweep`, hourly) re-implements the same three checks server-side against the `finance` doc and sends real pushes via the Admin SDK, with its own dedup fields stored back on the token doc. If you change the local-notification logic, check whether `functions/index.js` needs the equivalent change ‚Äî the two are intentionally kept in sync but are separate implementations (client JS vs. Cloud Function), not shared code.

### UI conventions worth following
- Custom-styled `<select>`/`<input type=date>`: any new one just needs `enhanceSelect()` / `enhanceDateInput()` called on it (or add it before `enhanceAllSelects()` runs) ‚Äî don't hand-roll another dropdown.
- **Never use `<input type=color>`** ‚Äî it opens the browser's native, unstylable OS color-picker popup, which looks jarringly out of place against the app's dark custom UI. Use the shared `openColorPicker(kind, id)` / `color-pick-modal` pattern instead (a `PALETTE`-swatch grid modal, same idea as the avatar picker) ‚Äî wire a new color-editable list (wallets/shiftTypes/tags all already do this) by adding a case to `selectPickedColor()` and rendering a plain `<button class="mgr-color" style="background:${color}" onclick="openColorPicker('kind','${id}')">` instead of a color input.
- Similarly, don't rely on native browser affordances that can't be restyled or that read as "not part of the app" ‚Äî e.g. no `title="..."` tooltips (native OS tooltip box); prefer a visible label/subtitle in the UI itself, like the rest of the settings rows do.
- "Settings manager" modals (wallets, categories, budgets, goals, recurring, tags, rules, rates, widgets, pin, premium, profiles, link-phone, avatar/color pickers) all follow the same pattern: a `.modal-overlay` + `.modal-card`, an `open*Manager()`-style function that renders a list into it and sets `display:flex`, and the shared `closeManagers()` that hides all of them by id ‚Äî add new manager ids to that shared close-list (an easy miss: forgetting this leaves a modal that won't close on backdrop-click).
- All icons are the custom monochrome SVG set (`window.Icon(name)` / `setIcon(id, name)`) ‚Äî no emoji, no icon fonts. Add new glyphs to `ICON_PATHS` rather than reaching for an emoji. **Check for duplicates within the same visible list/row-group** before reusing an existing glyph ‚Äî two *different* features sitting next to each other (e.g. two quick-action buttons, two settings-card icons in the same screen) with the same icon reads as a bug to users even though it isn't one; reusing an icon for the *same* feature across contexts (a tab + its settings entry + its quick-action) is fine and intentional. Generic action icons (`plus` for add, `trash` for delete, `pencil` for edit, `chevron`, `refresh`) are meant to repeat everywhere ‚Äî that's a consistent affordance, not a duplicate.
- i18n: a single `I18N = {uk:{...}, en:{...}}` object plus `tr(key)`; static markup uses `data-i18n`/`data-i18n-placeholder` attributes applied by `translateStaticDOM()`. There is no `data-i18n-title` ‚Äî dynamic titles/tooltips must be set in JS.
- The bottom nav (`.tabs`) currently has 5 tabs (–§—ñ–Ω–∞–Ω—Å–∏/–ì—Ä–∞—Ñ—ñ–∫ –∑–º—ñ–Ω/–Ý–æ–∑—Ä–∞—Ö—É–Ω–∫–∏/–ü–æ–∫—É–ø–∫–∏/–ù–∞–ª–∞—à—Ç—É–≤–∞–Ω–Ω—è) ‚Äî `.tab-icon-wrap` is 42px (38px under the ‚â§600px breakpoint). Adding a 6th tab means re-checking these still fit comfortably; there's no overflow/scroll handling on that bar.
- `sw.js`'s `fetch` handler must bail out (`return` without calling `respondWith`) for any cross-origin request and let the browser handle it natively ‚Äî it used to re-issue *every* request including third-party ones through `respondWith(fetch(event.request))`, which throws for some no-cors/redirect combinations (this broke reCAPTCHA's own CSP-report beacon with a console `TypeError: Failed to fetch`). Don't reintroduce a `respondWith` path for anything outside `self.location.origin`.

### Mobile UI redesign (multi-session pass, July 2026)
A design review (screenshots + before/after mockups) plus several rounds of live-phone-screenshot feedback drove a rework of the densest/most-cluttered screens. The patterns below are now the established convention ‚Äî follow them for anything new rather than reverting to the older per-screen styles they replaced.
- **Hero metric + chip row** (`.hero-metric`/`.hero-balance` + `.chip-stat-row`/`.chip-stat`): replaces "3-4 equal-weight stat cards in a grid" (the old `.stats-row` of `.stat-card`s). One large primary number up top, everything else demoted to a horizontal scrollable strip of small pill chips. Used on Finance (balance + wallet chips), Shifts (earned + hours/shifts/days-off), Debt (balance + start/paid/count), Shopping (remaining/bought). `.stat-card`/`.stats-row` still exist and render fine (Analytics tab still uses them for its 4 equal metrics, where equal weighting is actually correct) ‚Äî they're not deprecated, just no longer the default for "here's a bunch of numbers" on a primary tab.
- **Compact manager row** (`.budget-row`, `.rate-row`, `.cat-row`): replaces "one full-width card per list item with a giant input" in settings managers. Single line: icon badge, name (flex, ellipsis-truncated ‚Äî needs `min-width:0` on *every* ancestor in the flex chain up to the row itself, not just the text element, or a nowrap row can silently overflow its own card on long text; see the `.cat-row{min-width:0}` fix) + a small right-aligned input/value. Used in Budgets and –ö—É—Ä—Å–∏ –≤–∞–ª—é—Ç (rates) managers. Categories manager uses the same row shape but with a two-line text block (name + subcategory/amount caption) instead of a single value, since it has more to show per row.
- **FAB + bottom-sheet modal** (`.fin-fab` + `#tx-form-modal`): the Finance tab's "–ù–æ–≤–∞ –æ–ø–µ—Ä–∞—Ü—ñ—è" form used to render permanently expanded inline, eating a quarter of the screen on every visit. It's now a modal (same `.modal-overlay`/`.modal-card` pattern as every other manager) opened by a floating purple `+` button bottom-right. `editTransaction(id)` opens the same modal pre-filled instead of scrolling to an inline form. Same idea applies if any other "always-expanded form on a primary tab" shows up again.
- **Floating pill bottom nav**: `.tabs` is a detached rounded capsule (`left/right:14px`, `bottom:calc(14px + safe-area)`, `border-radius:26px` all around) rather than a bar flush against the screen edge with only top corners rounded. `.app`'s reserved bottom padding (112px) and `.fin-fab`'s offset (104px) are both tuned to this bar's height + gap ‚Äî if the bar's padding/height ever changes, re-check both.
- **Icon-based avatars, not emoji**: `BUILTIN_AVATARS` (profile avatar picker) uses `window.Icon(a.icon)` glyphs from the same monochrome set as everything else, not literal emoji characters ‚Äî matches the "no emoji" UI convention above, which the avatar picker used to be the one exception to. **There are exactly three render call sites for `BUILTIN_AVATARS`** (`renderAvatarPicker`, `renderProfileAvatarPickerGrid`, `renderProfilesUI`) plus `fillAvatarEl` for the topbar/preview circles ‚Äî when this array's shape changes again, grep `BUILTIN_AVATARS` and check all four; the `emoji`‚Üí`icon` rename shipped with `renderProfilesUI` still reading the old field name (rendered the literal text "undefined" in the –ü—Ä–æ—Ñ—ñ–ª—ñ switcher) because it was missed in the first pass. Lesson: after renaming a field on a shared data array, grep the field name itself (`.emoji`), not just the array name ‚Äî the array name search doesn't catch every destructured/aliased read.
- **Collapsed-by-default secondary panels**: Shifts tab's "–®–≤–∏–¥–∫–µ –∑–∞–ø–æ–≤–Ω–µ–Ω–Ω—è" quick-fill panel is collapsed by default (`toggleQuickFill()`, chevron rotates), expanding to full-width stacked `.fg` fields rather than a cramped inline row. This is also what fixed a real truncation bug ("–õ–∏–ø–µ–Ω—å" clipping to "–õ–∏–ø‚Ä¶") ‚Äî a compact custom-select variant (`.cs-sm`) had `min-width:0` on mobile, which let it shrink below its own text's width instead of the row wrapping; don't reuse `min-width:0` on a `.cs-sm` without also either giving it a real minimum or guaranteeing its container wraps.
- **Haptic feedback**: `showToast(msg, icon)` now fires `navigator.vibrate(...)` (feature-detected, wrapped in try/catch) ‚Äî a short pulse when `icon==='check'`, a sharper double-pulse when `icon==='xmark'`. Every existing `showToast()` call site gets this for free; no need to call it separately.
- All of the above was verified with the repo's stubbed-Firebase Playwright recipe (see Commands above) at both 420px and 1280px viewports, light + dark theme, plus targeted bounding-box diagnostics (`getBoundingClientRect()`/`getComputedStyle()` via `page.evaluate`) where a screenshot alone couldn't confirm *why* something overflowed ‚Äî worth reaching for that over guessing when a flex/overflow bug isn't visually obvious from a screenshot.

### Android / Google Play packaging (TWA, July 2026)
The app ships to Google Play as a **Trusted Web Activity (TWA)** ‚Äî a thin Android wrapper that opens the live `maxtr-c238f.web.app` site full-screen, not a separate native codebase. There is no local Android project checked into this repo; it's generated on demand.
- **Canonical domain for the Android package is `maxtr-c238f.web.app`**, not the GitHub Pages mirror ‚Äî chosen because Play's Digital Asset Links verification needs one fixed origin, and the Firebase domain has no subpath (GitHub Pages serves under `/Maxtr/`, which would complicate `start_url`/asset-link matching).
- **`.well-known/assetlinks.json`** (repo root) is what lets Android show the app without a browser URL bar ‚Äî it declares the app's package ID + the signing certificate's SHA-256 fingerprint, and Google Play/Android verifies it against `https://maxtr-c238f.web.app/.well-known/assetlinks.json`. `firebase.json`'s hosting `ignore` list originally excluded all dotfiles (`**/.*`), which would have silently dropped this file from every deploy ‚Äî it now has an explicit `"!/.well-known/**"` un-ignore plus a `Content-Type: application/json` header override for that path. If this file's fingerprint and the actual signing key ever drift apart (e.g. the key gets regenerated), the app degrades to showing a Chrome URL bar instead of failing outright ‚Äî annoying but not broken, and the fix is always "update the fingerprint in this file, redeploy hosting."
- **Package ID is `ua.zminka.app`** ‚Äî currently a placeholder (app name wasn't finalized when this was set up). It's freely changeable in whatever tool generates the package (twa-manifest.json / PWABuilder's Package ID field) up until the *first* real Play Store upload; after that, changing it means publishing as a brand-new listing, not an update.
- **The actual APK/AAB build cannot happen from a Claude Code cloud sandbox.** Confirmed by hitting explicit 403 policy denials from this environment's egress proxy on `dl.google.com` (Android SDK/build-tools ‚Äî blocks even the offline `bubblewrap build` path), `pwabuilder.com`, and even `maxtr-c238f.web.app` itself. Only `github.com`/npm/pypi-class hosts and Google's OAuth/Firebase *API* endpoints (as opposed to the Hosting *content* domain) were reachable. What **did** work fully offline from the sandbox: generating the Android Studio project skeleton via `@bubblewrap/core`'s `TwaManifest`/`TwaGenerator` classes directly (bypassing the CLI's interactive prompts, feeding it the local `manifest.json` instead of fetching it live, and serving the icon files over a `127.0.0.1` static server so `TwaGenerator` had something reachable to pull them from), and generating the release signing keystore with the locally-installed JDK's `keytool` (no network needed at all for that part). The actual compile step then has to happen either in Android Studio on a real machine, or ‚Äî the path actually used ‚Äî **pwabuilder.com** from the user's own browser, pointed at `https://maxtr-c238f.web.app`.
- **Signing keystore is intentionally not committed to this repo** ‚Äî it was generated once (`keytool -genkeypair`, RSA/3072, PKCS12 format) and handed directly to the account owner as a downloadable file; losing it means never being able to publish an update under `ua.zminka.app` again. If it ever needs regenerating, `.well-known/assetlinks.json`'s fingerprint has to be updated and hosting redeployed to match, or the "no browser URL bar" behavior breaks.
- **PKCS12 keystore gotcha, cost real debugging time**: `keytool` on JDK 9+ defaults to PKCS12-format keystores, which ‚Äî unlike the older JKS format ‚Äî **do not support a separate password for the private key entry**; `keytool -genkeypair -keypass X -storepass Y` with `X != Y` silently *succeeds* but prints `Warning: Different store and key passwords not supported for PKCS12 KeyStores. Ignoring user-specified -keypass value.` and just reuses the store password for the key too. Any consumer that's later told "the key password is X" (PWABuilder, `apksigner`, etc.) will then fail with `javax.crypto.BadPaddingException: Given final block not properly padded` ‚Äî a decryption error, not an authentication-rejected error, which reads like file corruption rather than a wrong-password problem. **Always generate PKCS12 signing keys with the same value for both `-storepass` and `-keypass`**, and only document/hand off one password, not two.
- **`android-icon-rounded.png`** (repo root) is a squircle-rounded-corner variant of `icon-512.png`, generated with Pillow ‚Äî used *only* by pasting its URL into PWABuilder's Android package form "Icon URL" field, not referenced anywhere in `manifest.json`. Reason: the TWA's launch splash screen centers the raw icon on the manifest's `background_color` with **no adaptive-icon masking applied** (unlike the home-screen launcher icon, which Android masks itself), so `icon-512.png`'s hard square corners looked jarring specifically on that one screen. The form's separate "**Maskable icon URL**" field must keep pointing at the original full-bleed `icon-512.png` ‚Äî pre-rounding that one would break the safe-zone contract Android's own adaptive-icon masking expects.
- Rebuilding the Android package from scratch (e.g. after a manifest/icon change) is a manual, browser-based process: pwabuilder.com ‚Üí enter `https://maxtr-c238f.web.app` ‚Üí Package For Stores ‚Üí Google Play ‚Üí fill Package ID / Icon URL / Maskable icon URL ‚Üí Signing key ‚Üí "Mine" ‚Üí upload the existing `.jks` and its one password ‚Üí Download Package. There's no CI/automation for this yet.
- As of this session: Google Play Console developer account **not yet created** by the account owner ($25 one-time fee, manual step at play.google.com/console) ‚Äî nothing has been uploaded to Play yet, only the hosting-side asset-link verification and a locally-tested `.apk` exist so far.
