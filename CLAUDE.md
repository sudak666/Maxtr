# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

**Zminka** (formerly "Xamss") is a single-file PWA: a personal tracker for work shifts, multi-currency finances, and debt/settlement schedules. It's Ukrainian-first (`uk`/`en` toggle), backed by Firebase (Auth + Firestore + Cloud Messaging + one scheduled Cloud Function), and deployed as static files via Firebase Hosting ‚Äî there is no build step, bundler, or frontend framework. Almost all product code lives in `index.html`.

## Commands

There is no root `package.json`, no bundler, no linter, and no automated test suite ‚Äî this is intentional, not an oversight. Treat the app as plain static HTML/CSS/JS.

- **Run locally**: serve the repo root with any static file server (e.g. `python3 -m http.server`) and open `index.html`. Firebase Auth/Firestore calls will hit the real `maxtr-c238f` project unless you intercept the `firebasejs` module imports.
- **Verify a change**: there's no test runner. The established approach in this repo is to drive `index.html` with Playwright against a **stubbed Firebase SDK** ‚Äî intercept the three `https://www.gstatic.com/firebasejs/.../firebase-{app,firestore,auth}.js` (and `firebase-messaging.js` if touching push) module imports via `page.route()` and fulfill them with minimal hand-written stub modules, then exercise the UI with real clicks/fills and assert on DOM state. This lets you test auth-gated flows without network access or real credentials. Always also do a syntax check before testing, since `<script type="module">` bodies aren't validated by naive `node --check` on the whole file ‚Äî extract the module script's contents with a regex and check that in isolation.
  - Operational notes from actually running this recipe in this repo's sandbox: serve via `python3 -m http.server` (not `file://` ‚Äî ES module imports of the Firebase gstatic URLs hit CORS issues over `file://`); `playwright` is installed globally, not as a project dependency, so run node scripts with `NODE_PATH=/opt/node22/lib/node_modules` set; launch Chromium with `executablePath: '/opt/pw-browsers/chromium'`. After signing in via the stub (`window.__authSignIn(uid, email)`), dismiss the first-run onboarding overlay (`#onboard-skip`) before interacting with tabs, or clicks land on the overlay instead.
  - Stub auth needs to export whatever `firebase-auth.js` imports actually are, including ones not exercised by the specific test at hand (`linkWithPhoneNumber`, `unlink`, etc.) ‚Äî a missing export fails the whole module import and blanks the page, not just the untested code path.
- **Cloud Function deploy**: `cd functions && npm install && firebase deploy --only functions,firestore:rules` from a machine with `firebase-tools` authenticated against the project (Blaze plan required). See `SETUP.md` for the full one-time Firebase Console setup this depends on (enabling sign-in providers, Cloud Messaging VAPID key, required GCP APIs).
- **Hosting deploy**: `firebase deploy --only hosting --project maxtr-c238f` from a git worktree checked out at `origin/main` (deploy from a clean merged tree, not a local feature branch). This is a **separate IAM permission** from the Cloud Functions deploy service account below ‚Äî expect a first-time 403 and need to grant `roles/firebasehosting.admin` before it works. Note: **GitHub Pages is also enabled on this repo** (Settings ‚Üí Pages, "Deploy from a branch" ‚Üí `main` / root) and auto-deploys on every push to `main` with no agent action needed ‚Äî `https://sudak666.github.io/Maxtr/` and `https://maxtr-c238f.web.app` are both live production mirrors serving the same code; only the Firebase Hosting one needs a manual deploy step.

## Architecture

### Layout
- `index.html` ‚Äî the entire app (styles, markup, and logic in three `<script>` blocks ‚Äî see below).
- `functions/` ‚Äî a separate Node 20 npm project (Cloud Functions v2 / `firebase-admin` + `firebase-functions`), deployed independently of the static site.
- `firebase.json` / `firestore.rules` / `manifest.json` ‚Äî Firebase Hosting, Firestore security rules, and PWA manifest.
- `sw.js` ‚Äî PWA asset-caching service worker (network-first for HTML, cache-first for everything else).
- `firebase-messaging-sw.js` ‚Äî a **second, separate** service worker, registered only when push is enabled, that handles background FCM push display. Keep these two service workers' responsibilities separate; don't merge them.
- `privacy.html` / `terms.html` ‚Äî standalone static pages, not part of the SPA.
- `SETUP.md` ‚Äî the manual, non-code Firebase Console setup required before auth/push/phone-sign-in work in production (enabling sign-in providers, authorized domains, deploying Firestore rules, VAPID key, required GCP APIs for the scheduled function). Keep this in sync whenever a change adds a new external dependency.

### `index.html` script structure
Three `<script>` blocks, in load order:
1. Two tiny inline classic scripts near the top: apply the saved theme before first paint, and register `sw.js`.
2. A classic (non-module) `<script>` containing the custom monochrome SVG icon library (`ICON_PATHS`, `window.Icon(name)`, `setIcon(id, name)`) plus theme/language toggle wiring. Runs before the module script.
3. The main `<script type="module">` ‚Äî everything else: Firebase init, all app state, and all feature logic, organized under `// ‚îÄ‚îÄ SECTION ‚îÄ‚îÄ` banner comments (search for `// ‚îÄ‚îÄ` to jump between them: STATE, FIREBASE, AUTH, PHONE SIGN-IN, LINK PHONE NUMBER, PIN LOCK, ONBOARDING, BIOMETRIC UNLOCK, INIT, TABS, custom SELECT/DATE PICKER/DIALOGS, COLOR PICKER, CALENDAR, charts, SETTINGS MANAGERS, GOALS, PROFILE, NOTIFICATIONS, PUSH NOTIFICATIONS, FINANCE, ANALYTICS, CSV EXPORT, DEBT, SHOPPING LIST). Functions called from inline `onclick`/`onchange` HTML attributes must be attached to `window` explicitly (`window.foo = function(){...}`) since module-scope declarations aren't global ‚Äî this is why you'll see both a bare `function foo(){}` and a `window.foo=foo;` line for anything wired to markup. **This bit us for real**: a function only called from an inline `onclick` attribute (not from other JS) must be `window.foo`, even a "just renders a list" helper ‚Äî `renderProfilesUI` was defined as a bare module-scope `function` and worked fine everywhere it was called from other JS, but silently threw `ReferenceError` every time the one inline `onclick="...; renderProfilesUI()"` fired, until it was also assigned to `window`.

### Firebase data model
- Firestore layout: `users/{uid}/max_tracker/{doc}` where `{doc}` is one of `shifts`, `finance`, `debt`, `backup_v2` (a one-time raw snapshot taken whenever config gets re-seeded), `profiles_meta` (see Multiple profiles below). `DCOL = 'max_tracker'`; `userDoc(name)` builds the path. The Firestore rule for this collection is a full wildcard (`match /users/{uid}/max_tracker/{document=**}`), so any doc name is already allowed ‚Äî no rules change is needed to add a new doc name. A legacy pre-auth shared `max_tracker/*` collection is explicitly denied in `firestore.rules` ‚Äî do not resurrect it.
- The `finance` doc is the big one: transactions, wallets, categories, budgets, subcategories, currencyRates, tags, autoRules, recurring, `shoppingList`, goals, profile, subscription, widgets, **and `notifSettings`** (synced here specifically so the server-side Cloud Function can see each user's notification preferences ‚Äî it used to be localStorage-only).
- `push_tokens/{uid}` is a separate top-level collection (not nested under `users/{uid}`) holding `{token, sentDaily, sentBudget, sentRecurring}` for the FCM push sweep; it has its own Firestore rule scoped to `request.auth.uid == uid`.
- Client-side caching/sync: every piece of state is mirrored into `localStorage` under per-user keys via `lsKey(name)` ‚Üí `mx_<name>_<uid>` (loaded first for instant paint), then reconciled against Firestore in `fbLoadNow()`. Writes go through `scheduleSave()` (debounced) ‚Üí `fbSaveNow()`, which uses optimistic concurrency (`lastKnownUpdatedAt` per doc, compared against each doc's `updatedAt`) and prompts the user on conflict rather than silently overwriting another device's newer save. A few internal localStorage keys still carry the old `xamss*` prefix from before the Zminka rename (e.g. `xamssTheme`, `xamssLang`) ‚Äî these were left as-is intentionally since renaming them would silently reset existing users' local prefs for zero benefit.

### Multiple profiles per account
One Firebase Auth account can hold several independent data-profiles (e.g. "–Ø"/"–î—Ä—É–∂–∏–Ω–∞") with their own finances/shifts/debts, switchable in Settings ‚Üí –ü—Ä–æ—Ñ—ñ–ª—ñ.
- `activeProfileId` (module-level, default `'default'`) drives both `userDoc(name)` and `lsKey(name)`: for the **default** profile these resolve to the original unsuffixed doc names/localStorage keys (zero migration, existing users are untouched); for any other profile, `userDoc` suffixes the doc name `"@<profileId>"` (e.g. `finance@abc123`) and `lsKey` suffixes the localStorage key the same way. Because every save/load path already funnels through these two functions, nothing else needed to change to make the whole app profile-aware.
- `profiles_meta` (account-wide, never suffixed even when a non-default profile is active) holds `{list:[{id, name, avatar, createdAt}], updatedAt}` ‚Äî just labels/avatars for the switcher, not real data. Each profile's actual identity (nickname + avatar shown in Settings ‚Üí –ü—Ä–æ—Ñ—ñ–ª—å and the topbar) lives inside *that profile's own* `finance` doc, since `profile.nickname`/`profile.avatar` is per-profile automatically once `finance` is suffixed.
- `activeProfileId` itself is a **per-device** choice (`localStorage['mx_activeProfile_'+uid]`), not synced ‚Äî switching profiles on one device doesn't affect others.
- `switchProfile(id)` flushes the current profile (`fbSaveNow()`) before reassigning `activeProfileId` and calling `fbLoadNow()` again ‚Äî this reuses all the existing seeding/rendering logic verbatim, so a brand-new profile gets the same default wallets/categories a brand-new account gets.
- **Known gap**: the scheduled Cloud Function (`functions/index.js` `notificationSweep`) only ever checks the **default** profile's `finance` doc ‚Äî extending server-side push to loop over every profile hasn't been done. Local in-app notifications work fine for whichever profile is active since they read the same in-memory state that switching already repopulates.
- Any render function that bails out early when its collection is empty (`if(!items.length) return;`) needs to explicitly clear its previously-rendered DOM first ‚Äî otherwise switching to a profile with no data leaves the *previous* profile's numbers on screen instead of blanking them (this bit us once with the Debt tab; check for the same pattern before adding another profile-scoped collection).

### Auth
Three sign-in methods, all in the AUTH / PHONE SIGN-IN sections: email+password, Google popup, and phone number (SMS code via `signInWithPhoneNumber` + an invisible `RecaptchaVerifier`). Layered on top, independent of Firebase Auth: a local PIN (SHA-256 hash in localStorage, `mx_pin_<uid>`) and optional WebAuthn biometric unlock ‚Äî these gate the UI locally but never replace the Firebase session.

Settings ‚Üí –ê–∫–∞—É–Ω—Ç additionally lets an already-signed-in user (e.g. one that signed in via Google) **link/unlink a phone number** as an extra credential (`linkWithPhoneNumber(currentUser, ...)` / `unlink(currentUser,'phone')`) ‚Äî a separate flow from the pre-auth sign-in one, with its own `link-recaptcha-container`/`RecaptchaVerifier` instance so it doesn't collide with the lock-screen's.

**reCAPTCHA retry gotcha**: `RecaptchaVerifier.clear()` can itself throw partway through on a failed attempt, leaving the widget's DOM node behind; the next attempt's fresh `RecaptchaVerifier.render()` then finds a stale widget already in the container and throws `"reCAPTCHA has already been rendered in this element"`. Both phone flows now force the container's `innerHTML` empty on error (`resetRecaptchaContainer(containerId)`) regardless of whether `.clear()` succeeded ‚Äî do this for any new reCAPTCHA-backed retry flow too.

**Phone auth troubleshooting** (not code issues, but easy to mistake for one): (1) needs the **Blaze** plan (documented below already); (2) needs **Authentication ‚Üí Settings ‚Üí SMS region policy** to explicitly allow the country being tested (Firebase blocks SMS to unlisted regions by default); (3) **Brave browser disables Web Push by default** (`brave://settings/privacy` ‚Üí "Use Google services for push messaging") ‚Äî causes `AbortError: Registration failed - push service error` on `getToken()`/`PushManager.subscribe()` that looks like a bug but isn't; (4) **GitHub Pages mirror isn't auto-authorized** ‚Äî `*.web.app`/`*.firebaseapp.com` are automatically in Firebase's Authorized domains, but `sudak666.github.io` is not, so invisible reCAPTCHA fails (flashes and vanishes) and phone sign-in/link dies with a generic error specifically on that mirror while working fine on `maxtr-c238f.web.app`. Add `sudak666.github.io` to Authentication ‚Üí Settings ‚Üí Authorized domains manually (see SETUP.md ¬ß2). `authErrorMessage()` in `index.html` now maps `auth/captcha-check-failed`, `auth/unauthorized-domain`, and `auth/network-request-failed` to specific messages instead of the generic fallback, to make this diagnosable from the UI itself next time; (5) **`auth/error-code:-39` / 503 on `sendVerificationCode` for every phone number tried, not just a repeat of the same one** ‚Äî this is not the per-number SMS rate limit (that's the same underlying Firebase error code but only reproduces on one specific number after repeated attempts); when it reproduces immediately on multiple different fresh numbers back-to-back, it means the project needs **reCAPTCHA Enterprise** wired up (Google has been migrating phone auth off classic invisible reCAPTCHA v2) and the Enterprise site key isn't configured/bound to the project's domains yet ‚Äî a Firebase Console + GCP Console configuration step, not an `index.html` bug; see SETUP.md ¬ß5 for the exact console path. `authErrorMessage()` maps this raw code to the "too many attempts" message client-side either way, since the client can't distinguish the two causes from the error code alone.

### Premium / free-tier limits
`subscription = {plan, expiresAt}` lives in the `finance` doc. `FREE_LIMITS`, `canAddMore(kind, count)`, and `showPremiumUpsell(kind)` still exist (kept for when a payment provider ships), but **no "add" action currently calls `canAddMore`** ‚Äî the wallets/categoriesPerType/autoRules/recurring/goals caps were deliberately removed, because with no way to actually purchase premium, a hard cap was just a dead end for every user (this is what was silently blocking "add category" for any account with ‚â•8 categories ‚Äî a brand-new account already starts at exactly 8 default expense categories). If a payment provider gets wired up, the caps should come back by re-adding the `if(!canAddMore(...)){ showPremiumUpsell(...); return; }` guard at the top of each `add*()` function (see git history around the "Remove remaining free-tier caps" commit for the exact call sites) rather than reinventing the gating pattern.

### Finance-tab widgets
`widgets = {rates, converter, analytics, chart, goals}` (synced in the `finance` doc) toggles which optional sections render on the Finance tab; `applyWidgetVisibility()` is the single place that reconciles the toggle state with the DOM. Adding a new toggleable widget means updating this object's default in every place it's read (there are a few: the Firestore doc load, the localStorage cache load, and the in-memory default) ‚Äî grep for `widgets.goals` to find all of them.

### Notifications (two independent delivery paths, same three conditions)
Both paths check: daily "did you log anything today" reminder, budget-exceeded, and upcoming-recurring-payment.
1. **Local** (`NOTIFICATIONS` section): fires only while the app is open, via the `Notification` API, checked on load / tab-visibility-change / a 5-minute interval. Dedup uses localStorage keys.
2. **Push** (`PUSH NOTIFICATIONS` section + `functions/index.js`): the client registers an FCM token (`push_tokens/{uid}`) via `getToken()` + `firebase-messaging-sw.js`. A scheduled Cloud Function (`notificationSweep`, hourly) re-implements the same three checks server-side against the `finance` doc and sends real pushes via the Admin SDK, with its own dedup fields stored back on the token doc. If you change the local-notification logic, check whether `functions/index.js` needs the equivalent change ‚Äî the two are intentionally kept in sync but are separate implementations (client JS vs. Cloud Function), not shared code.

### UI conventions worth following
- Custom-styled `<select>`/`<input type=date>`: any new one just needs `enhanceSelect()` / `enhanceDateInput()` called on it (or add it before `enhanceAllSelects()` runs) ‚Äî don't hand-roll another dropdown.
- **Never use `<input type=color>`** ‚Äî it opens the browser's native, unstylable OS color-picker popup, which looks jarringly out of place against the app's dark custom UI. Use the shared `openColorPicker(kind, id)` / `color-pick-modal` pattern instead (a `PALETTE`-swatch grid modal, same idea as the avatar picker) ‚Äî wire a new color-editable list (wallets/shiftTypes/tags all already do this) by adding a case to `selectPickedColor()` and rendering a plain `<button class="mgr-color" style="background:${color}" onclick="openColorPicker('kind','${id}')">` instead of a color input.
- Similarly, don't rely on native browser affordances that can't be restyled or that read as "not part of the app" ‚Äî e.g. no `title="..."` tooltips (native OS tooltip box); prefer a visible label/subtitle in the UI itself, like the rest of the settings rows do.
- "Settings manager" modals (wallets, categories, budgets, goals, recurring, tags, rules, rates, widgets, pin, premium, profiles, link-phone, avatar/color pickers) all follow the same pattern: a `.modal-overlay` + `.modal-card`, an `open*Manager()`-style function that renders a list into it and sets `display:flex`, and the shared `closeManagers()` that hides all of them by id ‚Äî add new manager ids to that shared close-list (an easy miss: forgetting this leaves a modal that won't close on backdrop-click).
- All icons are the custom monochrome SVG set (`window.Icon(name)` / `setIcon(id, name)`) ‚Äî no emoji, no icon fonts. Add new glyphs to `ICON_PATHS` rather than reaching for an emoji. **Check for duplicates within the same visible list/row-group** before reusing an existing glyph ‚Äî two *different* features sitting next to each other (e.g. two quick-action buttons, two settings-card icons in the same screen) with the same icon reads as a bug to users even though it isn't one; reusing an icon for the *same* feature across contexts (a tab + its settings entry + its quick-action) is fine and intentional. Generic action icons (`plus` for add, `trash` for delete, `pencil` for edit, `chevron`, `refresh`) are meant to repeat everywhere ‚Äî that's a consistent affordance, not a duplicate.
- i18n: a single `I18N = {uk:{...}, en:{...}}` object plus `tr(key)`; static markup uses `data-i18n`/`data-i18n-placeholder` attributes applied by `translateStaticDOM()`. There is no `data-i18n-title` ‚Äî dynamic titles/tooltips must be set in JS.
- The bottom nav (`.tabs`) currently has 5 tabs (–§—ñ–Ω–∞–Ω—Å–∏/–ì—Ä–∞—Ñ—ñ–∫ –∑–º—ñ–Ω/–Ý–æ–∑—Ä–∞—Ö—É–Ω–∫–∏/–ü–æ–∫—É–ø–∫–∏/–ù–∞–ª–∞—à—Ç—É–≤–∞–Ω–Ω—è) ‚Äî `.tab-icon-wrap` is 42px (38px under the ‚â§600px breakpoint). Adding a 6th tab means re-checking these still fit comfortably; there's no overflow/scroll handling on that bar.
- `sw.js`'s `fetch` handler must bail out (`return` without calling `respondWith`) for any cross-origin request and let the browser handle it natively ‚Äî it used to re-issue *every* request including third-party ones through `respondWith(fetch(event.request))`, which throws for some no-cors/redirect combinations (this broke reCAPTCHA's own CSP-report beacon with a console `TypeError: Failed to fetch`). Don't reintroduce a `respondWith` path for anything outside `self.location.origin`.

### Mobile UI redesign (multi-session pass, July 2026)
A design review (screenshots + before/after mockups) plus several rounds of live-phone-screenshot feedback drove a rework of the densest/most-cluttered screens. The patterns below are now the established convention ‚Äî follow them for anything new rather than reverting to the older per-screen styles they replaced.
- **Hero metric + chip row** (`.hero-metric`/`.hero-balance` + `.chip-stat-row`/`.chip-stat`): replaces "3-4 equal-weight stat cards in a grid" (the old `.stats-row` of `.stat-card`s). One large primary number up top, everything else demoted to a horizontal scrollable strip of small pill chips. Used on Finance (balance + wallet chips), Shifts (earned + hours/shifts/days-off), Debt (balance + start/paid/count), Shopping (remaining/bought). `.stat-card`/`.stats-row` still exist and render fine (Analytics tab still uses them for its 4 equal metrics, where equal weighting is actually correct) ‚Äî they're not deprecated, just no longer the default for "here's a bunch of numbers" on a primary tab.
- **Compact manager row** (`.budget-row`, `.rate-row`, `.cat-row`): replaces "one full-width card per list item with a giant input" in settings managers. Single line: icon badge, name (flex, ellipsis-truncated ‚Äî needs `min-width:0` on *every* ancestor in the flex chain up to the row itself, not just the text element, or a nowrap row can silently overflow its own card on long text; see the `.cat-row{min-width:0}` fix) + a small right-aligned input/value. Used in Budgets and –ö—É—Ä—Å–∏ –≤–∞–ª—é—Ç (rates) managers. Categories manager uses the same row shape but with a two-line text block (name + subcategory/amount caption) instead of a single value, since it has more to show per row.
- **FAB + bottom-sheet modal** (`.fin-fab` + `#tx-form-modal`): the Finance tab's "–ù–æ–≤–∞ –æ–ø–µ—Ä–∞—Ü—ñ—è" form used to render permanently expanded inline, eating a quarter of the screen on every visit. It's now a modal (same `.modal-overlay`/`.modal-card` pattern as every other manager) opened by a floating purple `+` button bottom-right. `editTransaction(id)` opens the same modal pre-filled instead of scrolling to an inline form. Same idea applies if any other "always-expanded form on a primary tab" shows up again.
- **Floating pill bottom nav**: `.tabs` is a detached rounded capsule (`left/right:14px`, `bottom:calc(14px + safe-area)`, `border-radius:26px` all around) rather than a bar flush against the screen edge with only top corners rounded. `.app`'s reserved bottom padding (112px) and `.fin-fab`'s offset (104px) are both tuned to this bar's height + gap ‚Äî if the bar's padding/height ever changes, re-check both.
- **Icon-based avatars, not emoji**: `BUILTIN_AVATARS` (profile avatar picker) uses `window.Icon(a.icon)` glyphs from the same monochrome set as everything else, not literal emoji characters ‚Äî matches the "no emoji" UI convention above, which the avatar picker used to be the one exception to. **There are exactly three render call sites for `BUILTIN_AVATARS`** (`renderAvatarPicker`, `renderProfileAvatarPickerGrid`, `renderProfilesUI`) plus `fillAvatarEl` for the topbar/preview circles ‚Äî when this array's shape changes again, grep `BUILTIN_AVATARS` and check all four; the `emoji`‚Üí`icon` rename shipped with `renderProfilesUI` still reading the old field name (rendered the literal text "undefined" in the –ü—Ä–æ—Ñ—ñ–ª—ñ switcher) because it was missed in the first pass. Lesson: after renaming a field on a shared data array, grep the field name itself (`.emoji`), not just the array name ‚Äî the array name search doesn't catch every destructured/aliased read.
- **Collapsed-by-default secondary panels**: Shifts tab's "–®–≤–∏–¥–∫–µ –∑–∞–ø–æ–≤–Ω–µ–Ω–Ω—è" quick-fill panel is collapsed by default (`toggleQuickFill()`, chevron rotates), expanding to full-width stacked `.fg` fields rather than a cramped inline row. This is also what fixed a real truncation bug ("–õ–∏–ø–µ–Ω—å" clipping to "–õ–∏–ø‚Ä¶") ‚Äî a compact custom-select variant (`.cs-sm`) had `min-width:0` on mobile, which let it shrink below its own text's width instead of the row wrapping; don't reuse `min-width:0` on a `.cs-sm` without also either giving it a real minimum or guaranteeing its container wraps.
- **Haptic feedback**: `showToast(msg, icon)` now fires `navigator.vibrate(...)` (feature-detected, wrapped in try/catch) ‚Äî a short pulse when `icon==='check'`, a sharper double-pulse when `icon==='xmark'`. Every existing `showToast()` call site gets this for free; no need to call it separately.
- All of the above was verified with the repo's stubbed-Firebase Playwright recipe (see Commands above) at both 420px and 1280px viewports, light + dark theme, plus targeted bounding-box diagnostics (`getBoundingClientRect()`/`getComputedStyle()` via `page.evaluate`) where a screenshot alone couldn't confirm *why* something overflowed ‚Äî worth reaching for that over guessing when a flex/overflow bug isn't visually obvious from a screenshot.

- **No payment provider wired up.** Premium is upsell-only right now (and its free-tier caps are currently all disabled app-wide for exactly that reason ‚Äî see Premium / free-tier limits above). Plan discussed: Stripe first (still a PWA), RevenueCat later once/if this ships to app stores (Apple/Google require native IAP there).
- A GCP service account `claude-deploy@maxtr-c238f.iam.gserviceaccount.com` exists specifically so an agent session without interactive `firebase login` access can run `firebase deploy`. Its roles were granted incrementally to get things working (Cloud Functions Developer/Admin, Firebase Rules Admin, Cloud Run Admin, Service Account User, and now also **Firebase Hosting Admin** for `firebase deploy --only hosting`) ‚Äî expect to hit a "permission denied" / 403 error on the *first* deploy of a kind that hasn't been done before and grant the missing role/API as prompted rather than pre-guessing the full set.
- Multi-profile push notifications: `notificationSweep` only sweeps the default profile (see Multiple profiles per account above).

- Facebook sign-in and a "savings goal" style debt reminder were discussed and deliberately deferred, not forgotten.
