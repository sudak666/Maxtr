**Every session that takes a real action — a deploy, an IAM/config change, a merged PR, a fix — must record it in `CHANGELOG.md` (or a more specific doc like `AUDIT.md` when one already exists for that topic) before ending the turn.** This repo has no other persistent memory between sessions — a container restart wipes everything not committed. Follow the style already used throughout `CHANGELOG.md`: what was done, why, the exact commit/PR reference, and any gotcha hit along the way (e.g. the IAM-propagation-delay note, the CI-fix root cause) — terse enough to skim, complete enough that a future session doesn't have to re-derive it from git log or re-discover the same bug. Do this even for actions that felt routine (a redeploy with "no changes detected", a docs-only PR) — the point is a continuous written record, not just documenting surprises. **`CHANGELOG.md` is a working log, not curated documentation** — it exists so a future session (or a human debugging a regression) can trace exactly what happened and why; it is intentionally separate from this file so `CLAUDE.md` itself stays a lean, current reference rather than growing unbounded. If a fact from the log becomes permanent/architectural (a new convention, a new module, a new gotcha that will recur), promote it into the relevant section of this file — don't just leave it buried in the log.

## Project overview

**Rytm** (formerly "Xamss", then "Zminka" — renamed to Rytm 2026-07-16, the account owner's final branding decision) is a static PWA: a personal tracker for work shifts, multi-currency finances, and debt/settlement schedules. It's Ukrainian-first (`uk`/`en` toggle), backed by Firebase (Auth + Firestore + Cloud Messaging + one scheduled Cloud Function), and deployed as static files via Firebase Hosting — there is no build step, bundler, or frontend framework; native browser ES modules do the file splitting instead. `index.html` holds the markup/CSS and loads four small classic (non-module) scripts plus the module graph; almost all app logic lives under `js/*.js`, loaded via `<script type="module" src="./js/app.js">` (see "`js/` module layout" below). It used to be one 7000-line inline `<script type="module">` — split apart specifically so a change to (say) the debt tab doesn't require reading through finance/calendar/settings code to get there; see that section for why this was safe to do without a bundler and what to watch for when editing across files.

## Architecture

### Layout
- `index.html` — markup, CSS, and four `<script src="...">` tags loading small classic (non-module) scripts from `js/`; loads `js/app.js` as the app's actual logic (see "`index.html` script structure" below).
- `js/` — the app's logic, split into 16 native ES modules (no bundler — the browser resolves the relative `import`s directly). See "`js/` module layout" below for the file list, the shared-state pattern, and gotchas for anyone splitting a file further or adding a new cross-file call.
- `functions/` — a separate Node 20 npm project (Cloud Functions v2 / `firebase-admin` + `firebase-functions`), deployed independently of the static site.
- `firebase.json` / `firestore.rules` / `manifest.json` — Firebase Hosting, Firestore security rules, and PWA manifest.
- `sw.js` — the single service worker, handling **both** PWA asset caching (network-first for HTML, cache-first for everything else) **and** Firebase Cloud Messaging background push display. It used to be split into `sw.js` (caching) + a separate `firebase-messaging-sw.js` (push), both registered at the site's root scope — but a single scope can only ever have one active service worker, so registering `firebase-messaging-sw.js` (which had no `fetch` handler) after `sw.js` silently replaced `sw.js` at that scope and disabled offline caching the moment a user enabled push. Merged into `sw.js` alone (bumped to v4) to remove the conflict; there is no `firebase-messaging-sw.js` file anymore — don't recreate it as a separate registration. `sw.js`'s `FIREBASE_SDK_ASSETS` also precaches (and cache-first-serves) the exact four gstatic SDK URLs `index.html`'s `<script type="module">` statically imports — a static ES module import is all-or-nothing, so without this, a cold start with no network (even with `index.html` itself already cached) would fail the whole module and leave the app inert past its HTML shell. This is a narrow exact-URL allowlist inside the fetch handler's normal cross-origin bail-out, not a general override — if the pinned SDK version in `index.html`'s imports ever changes, update `FIREBASE_SDK_ASSETS` in `sw.js` to match (and bump `CACHE_NAME`). `STATIC_ASSETS` also explicitly lists every file under `js/` for the same reason (same-origin this time, so no allowlist trick needed — they're precached alongside `index.html` itself) — add any new `js/*.js` file to this list too, or a cold start offline can fail to load it.
- `privacy.html` / `terms.html` — standalone static pages, not part of the SPA.
- `SETUP.md` — the manual, non-code Firebase Console setup required before auth/push/phone-sign-in work in production (enabling sign-in providers, authorized domains, deploying Firestore rules, VAPID key, required GCP APIs for the scheduled function). Keep this in sync whenever a change adds a new external dependency.

### `index.html` script structure
Five `<script>` tags, in load order — **all classic (non-module) scripts are now external files under `js/`, not inline** (moved there in the CSP-hardening pass, see the CSP entries in `CHANGELOG.md`; `sw.js`'s `STATIC_ASSETS` precaches all four same as every `js/*.js` module):
1. `js/theme-preinit.js` — applies the saved theme before first paint (must stay first, must stay a blocking classic script, not `defer`/`async`/`type=module`).
2. `js/touch-active-fix.js` — registers a no-op `touchstart` listener; iOS Safari only applies `:active` CSS states to a tap when at least one touch listener exists somewhere in the document, or every `:active`-based press animation (bottom nav, buttons, FAB) silently never shows on iPhone.
3. `js/sw-register.js` — registers `sw.js` on `window load`, **and drives reliable auto-update**: a `controllerchange` listener reloads the open page exactly once when a newly-deployed worker takes control (guarded against reload loops, and skipped on the first-ever install when there was no prior controller), plus a `reg.update()` poll on load and on tab-refocus. This exists because the app is a long-lived PWA/TWA users keep open without navigating: `sw.js` already uses `skipWaiting()`+`clients.claim()` so a new worker activates immediately, but the already-open page keeps executing the JS it loaded at first paint until it actually reloads — so without this, a user on an open session never picks up a deploy's new JS (this caused repeated "still broken" reports for bugs that were already fixed — their tab was running the previous version). **The transition onto this mechanism costs one manual reload/reopen** (the already-running old code can't retroactively gain the listener); every deploy after that lands automatically. Paired with `firebase.json` serving `sw.js` and `/js/**` as `Cache-Control: no-cache` (revalidate-before-use, not "never cache") so Firebase Hosting's default `max-age=3600` can't (a) delay the SW update check or (b) let a freshly-installed worker's `cache.addAll()` re-cache hour-stale JS from the browser HTTP cache. **When debugging a "my fix isn't working on the phone" report, suspect stale JS delivery first** — reproduce against the actual deployed behavior (or under the real CSP, see the CSP note below) before assuming the code is wrong; the local test suite serves fresh files every run and is blind to caching.
4. `js/classic-globals.js` — the custom monochrome SVG icon library (`ICON_PATHS`, `window.Icon(name)`, `setIcon(id, name)`) plus the I18N object/`window.tr()`/`window.translateStaticDOM()`/`window.setLang()`/`window.setTheme()`, theme/lang toggle wiring, and (added in the CSP pass) a small delegated-click listener for `.theme-opt`/`.lang-opt`/`.modal-card`/`.dlg-card`/`#profile-avatar-preview` that used to be inline `onclick=""` attributes. Runs before the module script. **This is genuinely outside the `js/` module graph** despite living in the same directory as the 16 ES modules — it's a classic script with immediate top-level execution, no `import`/`export`, no `__init_X__()` wrapper. `tr`/`Icon`/`setIcon`/`translateStaticDOM` are real globals (via `window`), not something any `js/*.js` ES module imports; `window.__applyLangDynamic` is the one hook back the other way (`js/core.js` assigns it so `setLang()` can re-render dynamic content after a language switch — grep for it before assuming these scripts don't talk to each other at all).
5. `<script type="module" src="./js/app.js">` — loads the whole `js/` module graph (see below). Functions called from inline `onclick`/`onchange` HTML attributes must be attached to `window` explicitly (`window.foo = function(){...}`) since module-scope declarations aren't global — this is why you'll see both a bare `function foo(){}`/`const foo = function(){}` and a `window.foo=foo;` line for anything wired to markup. **This bit us for real**: a function only called from an inline `onclick` attribute (not from other JS) must be `window.foo`, even a "just renders a list" helper — `renderProfilesUI` was defined as a bare module-scope `function` and worked fine everywhere it was called from other JS, but silently threw `ReferenceError` every time the one inline `onclick="...; renderProfilesUI()"` fired, until it was also assigned to `window`. **As of the CSP-hardening pass, this is now the exception, not the rule** — `index.html`/`js/*.js` templates have essentially zero inline `onclick=""` left (see the window.*/onclick-removal audit entries in `CHANGELOG.md`); this paragraph's advice only still applies to the ~10 deliberately-kept cross-file `window.*` exports documented there.

**A strict Content-Security-Policy IS deployed** (`firebase.json` hosting header): `script-src 'self' https://www.gstatic.com https://apis.google.com` — crucially **no `'unsafe-inline'` and no `'unsafe-eval'`**. This means **inline event-handler attributes (`onclick=""`/`onchange=""` baked into rendered HTML) SILENTLY DO NOT EXECUTE on the live site** (Firebase Hosting / the `.web.app` TWA), and `eval`/`new Function` are blocked too. It's dangerously invisible: the sandbox's `python3 -m http.server` sends no CSP, so inline handlers work in every local test, then no-op only in production. This shipped a real bug — the tx-list delete button (inline `onclick="event.stopPropagation();deleteTransaction()"`) no-op'd on the live site so the tap fell through to the row's click→edit listener ("delete opened edit"), and every `emptyStateHtml` "add first item" CTA was dead ("add doesn't work" on an empty list). All were converted to `data-action` (real `addEventListener` delegation, which CSP allows). **NEVER add an inline `onclick=""`/`onchange=""` to `index.html` or a `js/*.js` template string** — wire it as a `data-action` dispatched by the file's delegated capture-phase listener (or `el.onclick = fn` set in JS, which CSP does NOT block since it isn't an inline attribute), the same as `deleteTransaction`/`openColorPicker`/the empty-state CTAs now are. `tests/csp-handlers.mjs` (in CI) loads the page under the real firebase.json CSP and exercises these flows specifically so this can't regress silently again — the rest of the suite is blind to it (no CSP header locally). `el.onclick=`/`cell.onclick=`/`chip.onclick=` DOM-property assignments in JS are fine (not inline attributes); only attribute strings in HTML are blocked.

### `js/` module layout
`index.html` used to be one 7000-line inline `<script type="module">`; it's now 16 files under `js/`, split along the same `// ── SECTION ──` banners that organized it before (`state.js`/`core.js`/`firebase-sync.js`/`color-picker.js`/`auth.js`/`app-init.js`/`ui-widgets.js`/`calendar.js`/`settings-managers.js`/`goals-profile.js`/`notifications.js`/`finance.js`/`analytics-csv.js`/`debt.js`/`shopping.js`, plus `app.js` as the entry point). This was a mechanical, AST-verified transformation (parsed with `espree` + `eslint-scope`, not manually traced), not a rewrite — the actual code inside each function is unchanged. Three things make cross-file JS in this app different from a typical ES-module codebase, and matter for anyone adding a new cross-file reference or splitting a file further:

1. **All mutable app state lives in one shared object, `AppState` (`js/state.js`)** — `transactions`, `wallets`, `shifts`, `currentUser`, `activeProfileId`, all ~60 of what used to be individual top-level `let` bindings, are now `AppState.transactions`, `AppState.wallets`, etc. **Why one object instead of `export let x` per field**: ES modules only let the module that declared a binding reassign it, but many of these fields are reassigned from several different files — e.g. `fbLoadNow()` (`js/firebase-sync.js`) resets almost all of them, `switchProfile()`/`fbLoadNow()` write `AppState.transactions` from one file while `addTransaction()`/`deleteTransaction()` (`js/finance.js`) write it from another. A plain property write (`AppState.x = value`) has no such restriction; `export let x` per field would. **`js/state.js` has zero imports of its own, deliberately** — it's the one file every other file depends on, so if it depended on anything back, that circular edge risks a temporal-dead-zone `ReferenceError` on its own top-level `const AppState = {...}` initialization (this happened during development: `AppState`'s `widgetOrder`/`MONTHS`/`WEEKDAYS` fields needed `WIDGET_ORDER_DEFAULT`/`LANG_CALENDAR`, which briefly lived in `core.js` — which itself transitively imports `AppState` — and that circular edge threw). Both constants were moved into `state.js` itself instead. **If you add a new module-level mutable field, put it in `AppState`, not a fresh top-level `let`** — a bare new `let` works fine within its own file but silently can't be reassigned from anywhere else without going through this same problem.
2. **Every chunk that does something immediately at the top level (not inside a function) has that code wrapped in an exported `__init_<chunk>__()` function, called from `app.js` after all 16 files have finished importing.** The original single file ran strictly top-to-bottom, so e.g. `const auth = getAuth(fbApp)` (near the top) had always already run by the time a much later `onAuthStateChanged(auth, ...)` line executed. Once split across files with a real (partly circular) import graph, module evaluation order is not guaranteed to match the original file's order — an unwrapped top-level statement in one file can run before another file's declaration it depends on has executed, which is a temporal-dead-zone crash (hit during development with exactly this `auth`/`onAuthStateChanged` pair). **If you add new top-level side-effecting code to one of these files** (a `document.addEventListener(...)` call, a `setInterval(...)`, anything that isn't a `function`/`class`/`const`/`let` declaration), put it inside that file's `__init_X__()` function, not bare at the top level, or it risks running before some other file it depends on has initialized.
3. **A name only ever exposed as `window.foo = function(){...}` (no separate bare `function foo(){}`) still needs a real export/import if another file references the bare identifier `foo`** — this looks like it should be safe to skip (relying on `window` being shared across every script on the page) and an earlier version of this split tried exactly that, but it breaks for the specific case of the `window.foo = foo;` assignment statement itself: that line needs `foo` to already be a correctly-resolving lexical binding in the file it's written in, and "resolves via `window` later" doesn't help the statement that's supposed to be *setting* `window.foo` in the first place. Every `window.foo = function(){...}`-only name was rewritten to a real `const foo = function(){...}; window.foo = foo;` pair during the split for exactly this reason (except `closeManagers`, which already had a separate same-named bare wrapper function with different code — see that function if you need the exact shape). **When adding a new function only meant to be called from inline HTML (`onclick=`), still give it a real top-level declaration** (`function foo(){}` or `const foo = function(){}`) in whichever file it lives in, plus the `window.foo = foo;` line — don't write `window.foo = function(){...}` as the only place `foo` is defined, since any other `js/*.js` file that later needs to call it by name (not just via `onclick=`) will need to `import` that declaration, and there won't be one to import.

Verification for this split (documented for whoever next needs to add a 17th file or further split an existing one): `tests/smoke.mjs` + a manual Playwright pass exercising a transaction add, opening a settings modal, a language switch, and a shift-detail modal, all with zero console/page errors — see git history around the module-split commit(s) for the exact verification script if you need to reproduce it. The AST-analysis + code-generation scripts used to produce this split were one-off tooling (not committed — they read a since-deleted single-file snapshot of the module script) rather than a repo-maintained codegen step; treat `js/*.js` as hand-edited source from here on, the same as `index.html` always was.

### Build tooling (Vite bundler)
The app gained a real build step after being pure hand-served static files for its whole history — prompted by direct feedback relayed from the account owner ("не професійно написан код") and a deliberate decision to start with the lowest-risk, smallest piece of "doing it properly" (bundler only, no TypeScript, no framework yet — see the TypeScript adoption section below for the followup) rather than a big-bang rewrite, while the app is still pre-Play-Store-approval and has time to absorb tooling changes safely. Landed across several small, independently-verified, independently-deployed phases rather than one PR — full blow-by-blow reasoning, gotchas found, and verification detail live in `CHANGELOG.md`; this section is the durable summary.
- **`vite.config.js` bundles only the ES module graph rooted at `js/app.js`** (`build.rollupOptions.input`, not `index.html`) — deliberately not the more obvious "feed Vite the whole `index.html`" approach, which was tried first and found to silently fingerprint/rewrite `<link rel="manifest">`/`<link rel="apple-touch-icon">` to hashed paths, breaking the GitHub Pages mirror. The 4 classic non-module scripts (`theme-preinit.js`/`touch-active-fix.js`/`sw-register.js`/`classic-globals.js`) and `js/vendor/tesseract/**` are completely untouched by this — Vite never even sees them.
- **`js/receipt-ocr.js`'s dynamic `import()` of the vendored Tesseract library needed two fixes to survive bundling**: `/* @vite-ignore */` on the import call (so Vite never tries to bundle the vendor library itself), and `entryFileNames`/`chunkFileNames` output config forcing the bundled chunk to land under `dist/js/` (matching the source `js/` depth the import's relative specifier `./vendor/tesseract/...` depends on) instead of Vite's default `dist/assets/`. This is exactly the kind of regression a bundler can introduce silently — caught by `tests/receipt-scan-ui.mjs`, which is why that test (among others) gets run against the real built output, not just source.
- **`scripts/build-site.mjs`** (run via `npm run build`, after `vite build`) turns `dist/` from "just the bundled JS" into a complete deployable copy of the whole site: copies every static file Firebase Hosting's `public` dir needs (mirroring `firebase.json`'s own `hosting.ignore` patterns, plus this repo's dev-only tooling and the 20 original `js/*.js` module-graph files, now superseded by the bundle), then patches `dist/index.html`'s one `<script type="module">` tag to reference the real built bundle filename, and regenerates `dist/sw.js`'s `STATIC_ASSETS` list from that same build output instead of the hand-maintained 20-file array — this is the concrete payoff of the whole effort: `sw.js`'s "forgot to add a new file" bug class (see the Layout section's `STATIC_ASSETS` note) is now structurally impossible for anything served via `dist/`. `sw.js`'s `CACHE_NAME` for the `dist/`-served variant is derived from hashing **every** locally-precached file's actual bytes (not just the JS bundle's own hashed filename) — a change to e.g. `manifest.json` or an icon alone (none of those are content-hashed by Vite, they're copied verbatim) would otherwise leave `sw.js`'s text byte-identical across two different deploys, meaning an already-installed service worker would never detect the update (browsers detect a new SW purely by byte-comparing the registered script) and would keep serving the stale asset forever.
- **`firebase.json`'s `hosting.public` is `dist`** (was `.`), with a `hosting.predeploy: ["npm install", "npm run build"]` hook so a deploy from a clean checkout always regenerates `dist/` rather than depending on it already existing/being current on disk.
- **GitHub Pages still serves the unbundled source tree directly** — no tool available to an agent session can flip the repo's Settings → Pages build-source toggle to an Actions-based build, so this is a standing, intentional divergence between the two live mirrors (`maxtr-c238f.web.app` = bundled `dist/`, `sudak666.github.io/Maxtr/` = raw source), not something left half-finished. Revisit only if the account owner explicitly wants Pages moved onto the same pipeline.
- **`window.__RYTM_TEST_HOOKS__`** (`js/app.js`, attached unconditionally after every `__init_X__()` call): `{ AppState, addTransaction, scanReceiptImage, setMonobankSyncGapMsForTesting }`. A handful of Playwright tests used to reach into the live app's module-scoped internals via `page.evaluate(() => import('./js/state.js'))` — this only works when `js/*.js` files are served individually, not once inlined into one bundled chunk. This hook is the fix: it's attached the same way whether `js/app.js` ran as 20 separate files or 1 bundled chunk, so tests use `window.__RYTM_TEST_HOOKS__.X` uniformly and pass identically against source or `dist/`. **If a test ever needs one more module-scoped internal**, add it to this hook (import it into `js/app.js`, add it to the object) rather than reaching for a dynamic `import('./js/...')` — the latter will silently break the moment CI's `dist-smoke` job (which runs the full suite against the bundled build) picks it up.
- **CI now has 3 jobs, not 2** (see the CI bullet under Commands above): `smoke` (source), `dist-smoke` (bundled `dist/`, full parity with `smoke`'s browser-test coverage as of the `__RYTM_TEST_HOOKS__` fix above), `firestore-rules` (emulator, unrelated to this).

### TypeScript adoption (gradual, JSDoc-based)
The natural followup to the Vite bundler work above, same reasoning for going incrementally rather than a big-bang rewrite — converting a ~20-module, zero-type-annotation codebase to real `.ts` files all at once would be a much larger, much riskier undertaking than anything the Vite phases did (every file rename risks breaking `sw.js` precache paths, test `page.route()` stub URL matching, and dynamic import specifiers all at once). Started instead with TypeScript's own built-in mechanism for exactly this situation: type-checking plain `.js` files via inferred types + JSDoc comments, with zero runtime behavior change and zero file renames.
- **`tsconfig.json`**: `allowJs: true`, **`checkJs: false` project-wide** (`include: ["js/**/*.js"]`, vendor/tests/dist excluded), `noEmit: true` (this never compiles/emits anything — Vite's esbuild transform still handles the real build; `tsc` here is purely a checking tool). `checkJs: false` project-wide is deliberate, not a placeholder — a global `checkJs: true` sweep against a codebase with zero prior type annotations and heavy dynamic patterns (the ~60-field `AppState` object, direct DOM manipulation, Firebase SDK types) would likely surface an unmanageable flood of errors on day one.
- **Per-file opt-in via a `// @ts-check` pragma at the top of the file** — this is a real, standard TypeScript feature (the inverse of `// @ts-nocheck` when `checkJs` is globally `true`) designed for exactly this gradual-adoption case, not a workaround. `js/tx-validation.js` is the first (and, as of this section being written, only) file opted in — chosen deliberately as the lowest-risk possible starting point: zero imports, touches no `AppState`/DOM (see its own header comment), so checking it surfaces zero cross-file resolution noise. Gained a `@typedef {Object} TransactionDraft` and `@param`/`@returns` JSDoc annotations on `validateTransactionDraft()`; typechecks clean.
- **`npm run typecheck`** (`tsc --noEmit`) runs in CI (`smoke` job, right after the build step) — only ever reports on opted-in files, so it stays fast and low-noise as adoption grows one file at a time.
- **When to add `// @ts-check` to another file**: pick self-contained, low-import files first (mirrors the Vite phases' own "prove it small, then expand" pattern) — `functions/lib/pure.js`/`functions/lib/sweep.js`/`functions/lib/monobank.js` are similarly good candidates (already split out for framework-independence, same reasoning as `tx-validation.js`). A file that imports `AppState` or the Firebase SDK's `https://` URL specifiers will need real thought about how far to type those (a full `AppState` type, sensible or not, is a bigger design decision than this section resolves) — don't opt in a heavily-coupled file casually expecting a quick, clean pass.

### Firebase data model
- Firestore layout: `users/{uid}/max_tracker/{doc}` where `{doc}` is one of `shifts`, `finance`, `debt`, `backup_v2` (a one-time raw snapshot taken whenever config gets re-seeded), `profiles_meta` (see Multiple profiles below). `DCOL = 'max_tracker'`; `userDoc(name)` builds the path. **The Firestore rule for this collection is no longer a full wildcard** — `firestore.rules` matches a single path segment (`match /users/{uid}/max_tracker/{document}`, not `{document=**}`) and requires `document` to match `^(shifts|finance|debt|backup_v2)(@[A-Za-z0-9_]+)?$|^profiles_meta$`, plus shallow validation (top-level key count, `updatedAt`/`at` must be numbers if present — see the rules file's own header comment for why this stops short of a full per-field schema). **Adding a new doc *base name* now needs a matching `firestore.rules` change**, unlike before; a new `@<profileId>`-suffixed variant of an existing base name needs none. Before touching this file, run `tests/firestore-rules.mjs` (see Commands above) — it asserts against a real emulator and has already caught a rules bug (duplicate `allow write` blocks silently canceling validation via OR semantics) that a read-through missed. A legacy pre-auth shared `max_tracker/*` collection is explicitly denied in `firestore.rules` — do not resurrect it.
- **Transactions live in their own subcollection, not the `finance` doc.** `users/{uid}/max_tracker/{financeDocName}/transactions/{txId}` (`txId` = `String(tx.id)`, the same `Date.now()`-based id transactions have always used) — this is the one subcollection under `max_tracker`, with its own sibling `match` block in `firestore.rules` scoped to a `finance`/`finance@<profileId>` parent only. See `MIGRATION_PLAN_transactions.md` for the full history/rationale (1 MiB doc-size risk, whole-doc rewrite on every save, doc-level-not-field-level optimistic concurrency) — completed as a **direct cutover**, not the originally-planned phased dual-write/dual-read rollout, since the app was confirmed pre-launch with no real users to protect through a slow migration (see Operating conventions above). All Firestore access for transactions goes through `js/firebase-sync.js`'s TRANSACTIONS SUBCOLLECTION helpers (`txCollection()`, `saveTransactionDoc()`, `deleteTransactionDoc()`, `loadTransactionsFromSubcollection()`, `batchWriteTransactions()`, `batchDeleteTransactionsByIds()`, `deleteAllTransactionDocs()`) — every function that adds/edits/deletes a transaction, or cascades a change onto many transactions at once (tag deletion, category rename), calls one of these directly instead of `scheduleSave()`, so a single transaction edit no longer rewrites the whole `finance` doc. `fbLoadNow()` does a one-time client-side migration per account/profile: if the subcollection is still empty but the legacy `finance.data` array has entries, it batch-copies them in once and sets `AppState.txMigrated`/`finance.txMigrated` (mirrors the existing `catBackfillDone`/`catLegacyMerged` flag pattern) so a later load — even after every transaction has since been deleted, legitimately leaving the subcollection empty again — never re-migrates the stale array. `fbSaveNow()` no longer writes a `data` field onto the `finance` doc at all (dropped automatically by `{merge:false}` replacing the whole doc). Account deletion (`js/auth.js`'s `deleteAccountUser()`) explicitly deletes every doc in the subcollection too — Firestore never cascade-deletes a subcollection when its parent doc is deleted. The Cloud Function (`functions/lib/sweep.js`) reads the subcollection via the Admin SDK in parallel with the finance/debt doc reads it already did, falling back to the legacy `finance.data` array only when the subcollection is empty (a pre-migration account, or a race with a client that hasn't loaded yet). `scripts/backfill-transactions.mjs` (an operator-run one-time bulk migration script) predates this client-side automatic migration and is now optional — useful only for pre-migrating in bulk before rollout rather than waiting for each account's next app load.
- **CSV import (audit-followup session), `js/analytics-csv.js`**: round-trips `exportTransactionsCSV()`'s own file format back in — Settings → Фінанси → "Імпорт CSV" (`data-action="trigger-import-csv"` clicks the hidden `#csv-import-input`; its `change` event is handled by this file's own `FIELD_ACTIONS`/`dispatchFieldAction`-style listener, the same per-file delegated-event pattern every other module uses). Deliberately scoped to **only** this app's own export dialect — no generic bank-statement column-mapping UI, which would be a materially bigger feature (and a bigger way to get transaction data wrong) than a v1 needs. `parseCSV()` is a minimal parser for `csvEscape()`'s exact dialect (`;` delimiter, `"`-quoted fields, BOM), not a general CSV parser. `buildTransactionsFromCSV()` matches columns **positionally** against `exportTransactionsCSV()`'s header order, never by header text (labels are `tr()`-localized, so text-matching would break importing a uk-exported file after switching the UI to en). Each row resolves its wallet **name** back to a wallet **id** via `AppState.wallets.find(w=>w.name===...)` and revalidates through the same `validateTransactionDraft()` every manual add goes through (`js/tx-validation.js`) — a row that fails (unknown wallet, bad date/amount, ...) is skipped and counted, not fatal to the rest of the file. Valid rows get fresh ids via `finance.js`'s `newTransactionId()` (now exported specifically for this — `analytics-csv.js` already imported from `finance.js` for `deleteTransaction`/`editTransaction`/etc., so this doesn't introduce a new circular edge, just extends an existing one) and go through the same `batchWriteTransactions()` the transactions-subcollection migration already uses. **No id-based dedup**: the export format has no id column (never meant to round-trip — it's an internal detail), so re-importing the same file twice creates duplicates; this is a documented limitation surfaced in the confirm-dialog copy, not a silent trap — a content-based heuristic (same date+amount+comment) was deliberately not built, since it would also silently drop genuinely-repeated real transactions (two coffees at the same price on the same day). Needed a new `upload` glyph in `classic-globals.js`'s `ICON_PATHS` (mirrors the existing `download` icon's shape, arrow flipped) — there wasn't one. Covered by `tests/csv-import-export.mjs` (export → delete → re-import round-trip, plus an unknown-wallet-row-is-reported case).
- The `finance` doc still holds everything else: wallets, categories, budgets, subcategories, currencyRates, tags, autoRules, recurring, `shoppingList`, goals, profile, subscription, widgets, `integrations` (see the Monobank section below), **and `notifSettings`** (synced here specifically so the server-side Cloud Function can see each user's notification preferences — it used to be localStorage-only; now also carries `notifSettings.timeZone`, an IANA zone name captured client-side for the push sweep's local-time calculations).
- **`renderFinance()`'s transaction list (`js/analytics-csv.js`) does a targeted DOM update, not a full rebuild.** `renderFinance()` is called from ~20 places across `js/*.js` for state changes that often have nothing to do with any specific transaction (renaming a wallet, editing a budget, switching tabs back), so tearing down and recreating every `.tx-item` on every call was wasteful and also visibly replayed the row entrance animation on unrelated re-renders. It now matches existing `.tx-item` nodes by `data-id` (`dataset.txId`, a stringified transaction id) against the new `visible` list: a matched row gets its inner content regenerated in place via `txItemInnerHtml()` (so nothing ever goes stale — wallet/tag/category names are recomputed fresh every render, same as before) but keeps the *same outer element*, so its click/keydown listeners stay bound without re-registration and the CSS entrance animation (which only triggers on fresh insertion, not on content/attribute mutation of an already-connected node) doesn't replay; only genuinely new transactions get a freshly created node. Existing nodes are repositioned via `insertBefore` if their order changed, and any node whose transaction is no longer in `visible` (deleted, or filtered/collapsed out) is removed. One gotcha this pattern has to guard against explicitly: transitioning *from* the empty-state (`emptyStateHtml()`) or cold-start skeleton (`renderFinanceSkeleton()`) markup leaves non-`.tx-item` children in `#tx-list-container` that the diff wouldn't otherwise know to clear (it only tracks `.tx-item`/`.tx-view-all-btn`) — handled by an explicit `:not(.tx-item):not(.tx-view-all-btn)` cleanup pass before the diff runs. Covered by `tests/tx-list-targeted-update.mjs`, which tags rows with a custom attribute that only survives on a *reused* node (proving identity, not just matching content) across an unrelated re-render, an add, and a delete.
- `push_tokens/{uid}` is a separate top-level collection (not nested under `users/{uid}`) holding `{token, updatedAt, sentDaily, sentBudget, sentRecurring, profileState}` for the FCM push sweep; it has its own Firestore rule scoped to `request.auth.uid == uid`, plus type/size checks on the client-writable `token`/`updatedAt` fields (not a full key allowlist, since the Cloud Function's Admin SDK writes its own dedup fields onto the same doc, bypassing rules entirely).
- `error_reports/{uid}` (another separate top-level collection, added in the audit-followup session) mirrors the capped 20-entry client-side error log that `js/core.js`'s `logAppError()` already kept in `localStorage` (`mx_errlog_<uid>`) — before this, a crash on a real user's phone was invisible to the account owner unless the user sent a screenshot. `logAppError()` now also (best-effort, rate-limited to once per 30s via a module-level `lastErrorSyncAt` so an error loop can't turn into a write storm) whole-doc-replaces `error_reports/{uid}` with `{entries:[...], updatedAt}`, stack traces truncated to 500 chars. Only fires when `AppState.currentUser` is set — signed-out/anonymous errors stay local-only, not worth relaxing the per-uid rule for. Nothing in the client ever *reads* this collection — the account owner checks it via the Firebase console (Admin SDK bypasses rules), so `firestore.rules`' `error_reports/{uid}` match block only needs to gate writes (owner-only, `entries` must be a ≤20-item list, `updatedAt` a number). Deliberately **not** added to `deleteAccountUser()`'s cleanup `Promise.all` — same reasoning as `push_tokens/{uid}` already being left alone there: it's a small, capped, harmless-if-orphaned doc outside the `max_tracker` doc-name scoping that cleanup targets, and account deletion already has enough moving parts.
- **Empty `catch(e){}` audit (audit-followup session)**: all 34 empty catch blocks in `js/*.js` were individually reviewed against `logAppError()` above — none of them turned out to be masking a real, unexpected failure; every one falls into exactly one of four safe, best-effort categories: (1) `localStorage`/`sessionStorage` read/write (~20 sites — private-browsing mode or a full/disabled storage quota; the app already runs fine on in-memory defaults when this fails, there's nothing else to do), (2) `el.setPointerCapture(e.pointerId)` in the swipe-gesture handlers (`js/analytics-csv.js`, `js/debt.js`, `js/ui-widgets.js` — best-effort touch capture, safe to skip on a browser that doesn't support it), (3) `Intl.DateTimeFormat()`/`document.referrer`/`matchMedia()` reads (timezone/context detection with an already-safe fallback), (4) `navigator.vibrate()` (`js/ui-widgets.js`, already guarded by an `'vibrate' in navigator` feature check one line above). **The convention going forward**: a *new* catch block only stays silently empty if it's genuinely one of these four categories (or an equally inconsequential best-effort operation) — anything that could represent a real, unexpected app failure should call `logAppError(kind, detail)` (`js/core.js`) instead, which is exactly the tool the account owner now has to see it (local capped list + `error_reports/{uid}` mirror, see above). Not worth sprinkling a repeated one-line comment at all 34 near-identical call sites (`js/privacy-cache.js` in particular is *entirely* a localStorage-safety wrapper by design — every catch in that file is this by construction, not something to annotate individually) — this paragraph is the canonical record instead.
- Client-side caching/sync: every piece of state is mirrored into `localStorage` under per-user keys via `lsKey(name)` → `mx_<name>_<uid>` (loaded first for instant paint), then reconciled against Firestore in `fbLoadNow()`. Writes go through `scheduleSave()` (debounced) → `fbSaveNow()`, which uses optimistic concurrency (`lastKnownUpdatedAt` per doc, compared against each doc's `updatedAt`) and prompts the user on conflict rather than silently overwriting another device's newer save. A handful of device-global (not per-user) preference/dedup keys — theme, language, hide-amounts toggle, exchange-rate source/timestamp, and local-notification dedup state — live under a separate `mx<Name>` naming (e.g. `mxTheme`, `mxLang`, `mxNotif`; see `js/theme-preinit.js`, `js/classic-globals.js`, `js/settings-managers.js`, `js/notifications.js`) rather than through `lsKey()`, since they apply before any user is authenticated (`theme-preinit.js` runs pre-auth) or aren't meaningfully per-account. These used to carry a legacy `xamss*` prefix from before the Zminka (now Rytm) rename; renamed to `mx*` while the app was still pre-launch with no real users to reset (see Operating conventions above) — don't reintroduce the old prefix. `mx*` is an internal storage prefix decoupled from the marketing brand name on purpose, so it did not need to change again when the brand was renamed Zminka→Rytm.

### Multiple profiles per account
One Firebase Auth account can hold several independent data-profiles (e.g. "Я"/"Дружина") with their own finances/shifts/debts, switchable in Settings → Профілі.
- `activeProfileId` (module-level, default `'default'`) drives both `userDoc(name)` and `lsKey(name)`: for the **default** profile these resolve to the original unsuffixed doc names/localStorage keys (zero migration, existing users are untouched); for any other profile, `userDoc` suffixes the doc name `"@<profileId>"` (e.g. `finance@abc123`) and `lsKey` suffixes the localStorage key the same way. Because every save/load path already funnels through these two functions, nothing else needed to change to make the whole app profile-aware.
- `profiles_meta` (account-wide, never suffixed even when a non-default profile is active) holds `{list:[{id, name, avatar, createdAt}], updatedAt}` — just labels/avatars for the switcher, not real data. Each profile's actual identity (nickname + avatar shown in Settings → Профіль and the topbar) lives inside *that profile's own* `finance` doc, since `profile.nickname`/`profile.avatar` is per-profile automatically once `finance` is suffixed.
- `activeProfileId` itself is a **per-device** choice (`localStorage['mx_activeProfile_'+uid]`), not synced — switching profiles on one device doesn't affect others.
- `switchProfile(id)` flushes the current profile (`fbSaveNow()`) before reassigning `activeProfileId` and calling `fbLoadNow()` again — this reuses all the existing seeding/rendering logic verbatim, so a brand-new profile gets the same default wallets/categories a brand-new account gets.
- The scheduled Cloud Function (`functions/index.js` `notificationSweep`) reads `profiles_meta` and sweeps every profile on the account (default + suffixed), not just the default one — each profile's `finance`/`finance@<id>` doc is checked independently. Per-profile dedup state lives in `push_tokens/{uid}.profileState.<profileId>` (`{sentDaily, sentBudget, sentRecurring}`); pre-existing token docs' flat `sentDaily`/`sentBudget`/`sentRecurring` fields are read as the default profile's initial state until the sweep writes `profileState.default` for the first time, so old single-profile users need no migration. Local in-app notifications work fine for whichever profile is active since they read the same in-memory state that switching already repopulates.
- Any render function that bails out early when its collection is empty (`if(!items.length) return;`) needs to explicitly clear its previously-rendered DOM first — otherwise switching to a profile with no data leaves the *previous* profile's numbers on screen instead of blanking them (this bit us once with the Debt tab; check for the same pattern before adding another profile-scoped collection).

### Shared profiles (audit-followup session)
A second Firebase Auth account can be invited to read/write one of this account's own profiles (shifts/finance/debt + transactions) — built for a couple sharing one household's finances, not a general multi-tenant model. This was the largest, highest-risk item in the audit-followup session (a new cross-account write path on a real production Firestore project), so the design below was verified against a real local emulator (`tests/firestore-rules.mjs`, 25 new cases covering the full invite→redeem→join→leave lifecycle plus every rejection path) before any client code was written, not after.

- **Reuses the existing `users/{uid}/max_tracker/{doc}@{profileId}` tree — deliberately not a parallel `shared_profiles` top-level collection.** The owner's side of a shared profile is stored and behaves exactly like any other profile of theirs; nothing about `fbSaveNow()`/`fbLoadNow()`/`renderFinance()`/etc. needed to change. The only new client-side generalization is in `js/firebase-sync.js`: `userDoc(name)` now resolves against `AppState.activeProfileOwnerUid || AppState.currentUser.uid` instead of always the current user's own uid (never for `profiles_meta`, which stays account-wide/own-uid always), and `lsKey(name)` returns `null` (skip local caching entirely) whenever `activeProfileOwnerUid` is set — a shared profile's data always loads fresh from Firestore rather than reasoning about cache invalidation across two different accounts' devices. `txCollection()` generalizes for free since it's built on `userDoc('finance')`.
- **Membership lives in a new doc-name kind, `shared_members`** (suffixed `@{profileId}` the same as shifts/finance/debt; unsuffixed for a shared *default* profile, matching how `'default'` already omits the suffix everywhere else), holding `{members:[ownerUid, ...], updatedAt}` under the **owner's own** `max_tracker` tree.
- **Joining is invite-code based, never open self-join.** `profile_invites/{code}` (a new top-level collection, not nested — a joining user has no relationship to the owner's account yet, so has to be able to read one before anything else) holds `{ownerUid, profileId, profileName, createdBy, createdAt, expiresAt, usedBy}`. The code (the doc ID) is an 8-char unambiguous-alphabet string (`ABCDEFGHJKLMNPQRSTUVWXYZ23456789` — no `0/O/1/I`), generated client-side, 24h TTL, single-use. **Redeeming is a genuine two-step, rules-enforced sequence, not one write**: step 1 marks the invite used (`usedBy: null → joinerUid`, only possible while still null and unexpired); only *after* that commits does step 2 — joining `shared_members@{profileId}` — become possible, because that write's rule does a `get()` on the invite doc (referenced by a `lastJoinInviteCode` field the client includes in the same write) and requires `usedBy == request.auth.uid`. This was verified empirically against the emulator before being adopted as the design (see the three standalone probe scripts used to confirm Firestore rules actually resolve dynamic `get()` paths built via string concatenation, `arrayUnion`/`arrayRemove` resolve correctly for `request.resource.data` comparisons, and `list.removeAll()` works, none of which were assumed) — a single-write "just check the members array shape" rule would have left a real hole: anyone could self-add to `shared_members@{profileId}` for *any* guessed/leaked `{ownerUid, profileId}` pair without ever possessing a real invite, since Firestore rules can't require "this write only succeeds if a *different*, specific other document was also written" any other way. Two joiners racing the same code can't both win: Firestore commits are serialized per document, so the second one's rule evaluation sees `resource.data.usedBy` already non-null and fails.
- **`firestore.rules`' main `users/{uid}/max_tracker/{document}` block** gained: a `isSharedMember(ownerUid, document)` helper (does the requester's uid appear in the relevant `shared_members` doc, derived positionally from `document`'s `@suffix`); a read/write grant for `shifts`/`finance`/`debt` (never `backup_v2`/`profiles_meta`/`shared_members` itself) to any such member; and the join/leave branches for `shared_members` described above. **Deletion of the profile's data docs stays owner-only even when shared** — a member can edit contents or leave, never delete the doc outright or forcibly remove another member (also rules-enforced: `request.resource.data.members == resource.data.members.removeAll([request.auth.uid])` only ever permits removing *your own* uid). The transactions subcollection match block got the identical `isSharedMember()` extension.
- **Client functions** (`js/firebase-sync.js`): `shareCurrentProfile()` (owner-only — throws if called while viewing someone else's shared profile; idempotent, reissues a fresh code if the profile is already shared), `redeemSharedInvite(code)` (returns `{ok:false, reason}` for every rejected case — unknown/used/expired code, or trying to redeem your own — rather than throwing, since these are all plain-typo-reachable and the UI just needs to show a message), `leaveSharedProfile(ownerUid, profileId)`. **UI**: Settings → Профілі gained "Поділитись поточним профілем" (shows the code in a plain `uiAlert`, not a dedicated share sheet — matches this repo's preference for reusing existing dialog primitives over single-purpose UI) and "Приєднатися за кодом" (`uiPrompt`); each profile row shows a "Спільний" badge and a leave (not delete) button when `kind==='shared'`. A shared-profile reference's `name`/`avatar` are still stored in the *joining* account's own `profiles_meta` (cosmetic only, per-account, same as any other profile) — editable the normal way, doesn't affect the underlying shared data doc at all.
- **Still out of scope** (documented gaps, not oversights): **`notifSettings` lives inside the shared `finance` doc**, so it's effectively shared too — a member changing the daily-reminder time changes it for the owner as well, a minor, disclosed UX quirk rather than a bug, not worth restructuring `notifSettings` out of `finance` just for this; no cleanup of `shared_members`/`profile_invites` on account deletion (`deleteAccountUser()`), matching the same already-established precedent as `push_tokens`/`error_reports` being left alone there (small, harmless-if-orphaned docs, outside the deletion routine's actual scope). Granular permissions and push-notification-sweep support (both originally listed here as "deliberately out of scope for v1") were implemented in a later session — see the two dedicated sections immediately below.

### Granular permissions on shared profiles (a later session)
Every member used to get full read/write on shifts/finance/debt — the account owner asked for a real "view only" tier. Two roles: **editor** (default, unchanged behavior) and **viewer** (read-only).
- **`shared_members`'s shape gained a `roles` map**: `{members:[...], roles:{uid:'editor'|'viewer'}, updatedAt}`. A member absent from `roles` (every pre-existing member, and any new joiner until the owner explicitly downgrades them) defaults to `'editor'` — this is what makes the rollout backward-compatible: no migration step, no existing shared profile silently loses write access the moment this deployed.
- **The real enforcement is in `firestore.rules`, not the client** — `memberRole()`/`isEditorMember()` (new helpers alongside the existing `isSharedMember()`) gate the main doc's and the transactions subcollection's `create`/`update`/`delete` branches; `isSharedMember()` alone (any role) still gates `read`. The client-side `canEditActiveProfile()` (`js/core.js`) is a UX convenience that mirrors this, never a second source of truth — a viewer's client is also given read access to the `shared_members` doc itself (a new `allow read` branch) specifically so it can look up its own role. Emulator-verified: 17 new cases in `tests/firestore-rules.mjs` (a freshly-joined member defaults to editor; owner downgrades to viewer; viewer can still read everything including `shared_members`; viewer is rejected on every write path — finance/shifts/transactions create/update/delete; a viewer can still leave).
- **Role changes are owner-only, applied after the fact, never during redemption** — deliberately simpler than trying to let a joiner's own redeem-invite write also set their role atomically: Firestore rules can't cleanly verify "this write only touches your own key in a shared map" without real complexity/risk, so every new member always joins as editor (unchanged join rule, zero new attack surface there) and the owner downgrades them afterward via a small **"Учасники" (Members) manager** (`js/color-picker.js`'s `openSharedMembersManagerUI()`/`renderSharedMembersList()`/`toggleMemberRoleUI()`, `#shared-members-modal` in `index.html`) — a button on each of the owner's own profile rows in the Profiles manager, following the same list-manager pattern as wallets/categories/tags rather than the single-shot `uiAlert` pattern `shareCurrentProfileUI()` uses (a member list genuinely needs a per-row action, which one dialog can't express). Members are shown by a shortened uid — this app has no cross-account contacts/display-name lookup, the same limitation the invite-code flow already has.
- **Client-side gating, defense-in-depth, not the security boundary**: `AppState.activeProfileRole` (`js/state.js`, default `'editor'`) is loaded via `loadActiveProfileRole()` (`js/firebase-sync.js`) on `switchProfile()` and once at cold-init (covers launching straight into a previously-active shared profile). `canEditActiveProfile()` guards the app's primary mutating entry points — `addTransaction()`/`deleteTransaction()` (`js/finance.js`), `saveModalSelection()` (`js/calendar.js`), `addDebtEntry()`/`deleteDebtEntry()` (`js/debt.js`) — with an immediate `shared_profile_readonly` toast instead of a doomed write attempt. `applyProfileReadOnlyUI()` (`js/ui-widgets.js`) toggles a `body.profile-readonly` class that hides `.fin-fab` (the Finance/Debt "add" buttons; Shifts has no dedicated FAB — a day-cell tap opens the shift modal directly, guarded at `saveModalSelection()` instead). **Deliberately not individually guarded**: secondary settings mutations (renaming a wallet, editing a budget/category/tag/rule). A viewer attempting one of those still fails safely — `firestore.rules` rejects the write regardless of what the client thinks — just via the existing generic `sync_autosave_error` toast rather than a role-specific message; scoped out to keep this pass to the primary, most-used mutating paths rather than auditing every settings-manager call site.

### Push notification support for shared profiles (a later session)
`functions/lib/sweep.js` used to only ever read `users/{token holder's own uid}/max_tracker/{doc}` for every profile in that account's `profiles_meta.list` — which doesn't exist for a *shared* profile someone else owns, so a member enabling push never got reminders for it (silently, not brokenly — `financeSnap.exists` guards already made this a safe no-op, just an incomplete feature).
- **Fix**: `profiles_meta.list` already carries `{id, name, kind:'shared', ownerUid}` for a shared-profile reference (`redeemSharedInvite()`, unchanged) — `sweepToken()` now keeps the full entry, not just `.id`, and resolves a `dataOwnerByProfile` map (`dataOwnerFor()`: the entry's `ownerUid` for a `kind:'shared'` entry, the token holder's own uid otherwise) before fetching each profile's finance/debt/transactions docs — every `users/${uid}/...` path in the "other profiles" parallel-fetch block became `users/${dataOwnerByProfile[id]}/...`. `sweepProfile()` itself gained a `dataOwnerUid` parameter (default `= uid`, so every existing call site/test that never passes it is unaffected) for its own two fallback fetches (only reachable when a caller doesn't pre-supply `financeSnap`/`debtSnap`, which `sweepToken()` always does — this matters for direct `sweepProfile()` tests).
- **The Admin SDK bypasses `firestore.rules` entirely**, so this needed no security-rules change — the sweep already has unrestricted read access to every document; the only real change was *which path* it reads for a shared entry. Whether the *member* is a viewer or editor doesn't affect their push notifications either — they see reminders for the shared profile's own state regardless of their write permission on it.
- **`notifSettings` is still the owner's** (lives inside the shared `finance` doc, see the "still out of scope" note above) — a member's push for a shared profile uses whatever reminder time/toggles the *owner* last saved, not anything the member can independently configure. Disclosed pre-existing quirk, not new with this change.
- 2 new `tests/functions-sweep.mjs` cases (now 30 total): a shared-profile entry's daily reminder correctly reads from the owner's uid (proven by the member's own default-profile data being deliberately different, so a wrong-uid read would produce a detectably wrong result, not just an accidentally-matching one); a shared-profile budget-exceeded push sums spending from the *owner's* transactions subcollection. Both construct the token holder's own profile with different data specifically so a regression back to the old same-uid-only read path would fail loudly, not coincidentally pass.
- **Testing**: `tests/firestore-rules.mjs` (25 new cases, emulator-verified — the security-critical part) and `tests/shared-profiles-ui.mjs` (client wiring only, single stubbed uid — proves a real code gets generated and that both rejection paths surface as visible feedback rather than silently failing or crashing; a true two-account join/access flow isn't testable with this repo's single-stubbed-uid-per-page-load Playwright recipe, so that full lifecycle's correctness rests on the emulator tests instead).

### Monobank integration (a later session)
"Прив'язати Monobank" (Settings → Фінанси, right after Гаманці) — pulls transactions from the account owner's real Monobank cards/jars into the app via Monobank's Open API (`api.monobank.ua`). Scope decided upfront with the account owner via `AskUserQuestion` before writing any code (recorded here since both were real architectural forks, not just implementation details): **manual "Синхронізувати" button, not a webhook** (simpler, no public always-listening endpoint to register/maintain — a real-time webhook is a documented possible future upgrade, not built), and **links every card and jar on the token at once**, not a per-account picker.

- **Why a Cloud Function proxy at all**: `api.monobank.ua` never sends CORS headers, so a browser `fetch()` straight to it is blocked — identical root cause to `PRIVAT_RATES_URL` in `functions/index.js` (see that function's own comment). `functions/index.js`'s new `monobankProxy` (`onRequest`, exposed same-origin via the `"/api/monobank"` Hosting rewrite in `firebase.json`, mirroring `"/api/privat-rates"`) relays `action=client-info` and `action=statement&account=&from=&to=` calls to Monobank, attaching the caller-supplied `X-Monobank-Token` header. **Unlike `privatRates` (public data, no auth), this proxy requires a valid Firebase ID token first** (`Authorization: Bearer <idToken>`, verified via `firebase-admin`'s `getAuth().verifyIdToken()`) — without it, this endpoint would be a free anonymous relay into Monobank's API for anyone who found the URL, using *their own* bank token but still riding on this project's Cloud Function/quota. The query-building/window-validation logic (`buildMonobankUrl()`) is split into `functions/lib/monobank.js` for the same reason `lib/pure.js`/`lib/sweep.js` are split out — unit-testable (`tests/monobank-proxy.mjs`) without firebase-admin credentials.
- **The token itself is a live bank credential, handled like any other per-profile secret this app already stores**: `AppState.integrations.monobank = {token, clientName, accounts:[{id, kind:'account'|'jar', label, currencyAlpha}], mapping:{monobankAccountId: walletId}, lastSyncAt}`, synced inside the existing `finance` doc (`fbSaveNow()`/`seedConfigFromDocs()` in `js/color-picker.js`, plus the `mx_cfg_*` local-cache mirror in `js/app-init.js`/`js/color-picker.js`) — same Firestore-rules-enforced per-uid protection as every other field in that doc, no new doc kind/rules change needed. The token is **only ever sent to this app's own `monobankProxy`**, never anywhere else client-side.
- **Connecting** (`connectMonobankUI()`, `js/monobank.js`) calls `client-info` once, then auto-creates one new wallet per returned account **and** jar (`createWalletForMonobankAccount()` — named `"Monobank <masked PAN or jar title>"`, currency mapped from Monobank's numeric ISO 4217 code via a small hand-maintained table, defaulting to UAH for anything outside this app's 5 supported currencies) and records `monobankAccountId → walletId` in `mapping`. No manual per-account wallet picker — this *is* the "link everything at once" decision above.
- **Syncing** (`syncMonobankUI()`) walks `mapping`, one account at a time, and for each calls `statement` for `[lastSyncAt ?? now-31days, now)`, **chunked into ≤31-day-+1h windows** (`MONOBANK_MAX_WINDOW_SEC`, Monobank's own hard limit on that endpoint) if the gap is wider — capped at `MAX_SYNC_CHUNKS_PER_ACCOUNT` (14, ~14 months) as a defensive backstop against a corrupted `lastSyncAt`, not a real-world expectation. **Every single call this module makes — `client-info` or `statement`, any account — is paced at least `SYNC_REQUEST_GAP_MS` (61s) apart** via `paceMonobankRequest()`, because Monobank enforces "1 request per 60 seconds per token" *across the whole API*, not per-endpoint; a multi-account/multi-chunk sync can genuinely take minutes, which is exactly the tradeoff the account owner accepted by choosing "manual button" over "webhook" — `#monobank-sync-status` shows live "Синхронізую рахунок N/M…" text so this doesn't read as a hung UI. Each returned entry with `hold:true` (a pending/not-yet-settled preauth) is skipped — same reasoning as every other "don't trust not-yet-final data" decision in this app. **Dedup is by Monobank's own stable per-entry `id`**, stored as `monobankId` on the created transaction (`AppState.transactions.some(t=>t.monobankId===id)`) — re-syncing an overlapping window (which happens on every sync after the first, by design) never creates duplicates, verified explicitly by `tests/monobank-connect.mjs`'s second-sync case.
- **Category**: the same keyword auto-rules a manual add already applies (`js/finance.js`'s `findMatchingRule()`, exported specifically for this reuse) get first say; an unmatched entry falls back to the generic `"Інше"` category, exactly like a manual entry with nothing picked. No MCC-code-to-category mapping in v1 — Monobank's `mcc` field is returned but unused; a real mapping table would be a reasonable v2, not built here to keep this session's scope to what was actually asked.
- **No "transfer" detection**: an internal move between two of the user's own Monobank accounts/jars shows up as a separate expense on one and income on the other (not this app's `type:'transfer'`), same simplification `js/analytics-csv.js`'s CSV import already makes rather than trying to correlate paired entries by amount/timestamp heuristically across accounts.
- **Testing**: `tests/monobank-proxy.mjs` (plain node, no browser — `buildMonobankUrl()`'s query validation and 31-day-window enforcement) and `tests/monobank-connect.mjs` (full stubbed-Firebase Playwright recipe, plus a `page.route()` stub for `"/api/monobank"` itself since this repo's test setup has no Functions emulator — same principle as stubbing the Firebase SDK's gstatic imports). The real Cloud Function's ID-token verification and the actual `api.monobank.ua` fetch aren't exercised by either — same split as `notificationSweep`'s `lib/sweep.js` (unit-tested) vs. `index.js`'s wiring (not). `setMonobankSyncGapMsForTesting()` (`js/monobank.js`) exists purely so the Playwright test doesn't need to wait through the real 61s pacing gap per request — production code never calls it.
- **CSP**: none needed — the client only ever calls the same-origin `/api/monobank` path (already covered by `connect-src 'self'`), never `api.monobank.ua` directly; that cross-origin call happens entirely server-side inside the Cloud Function.
- **Not done in this session, documented gaps rather than oversights**: no webhook/real-time sync (see the scope decision above); no per-account "don't import this one" toggle (all-or-nothing per the "link everything" decision); no MCC→category mapping; no cleanup of `integrations.monobank` in `deleteAccountUser()` (the whole `finance` doc is deleted there regardless, so this is moot — nothing extra needed).

### Firebase App Check (audit-followup session)
Integrated end-to-end, but deliberately left in **monitor mode, not enforced** — see below for exactly why and what the one remaining manual step is.

- **What was provisioned, and how** (all done live via REST calls under `claude-deploy@maxtr-c238f.iam.gserviceaccount.com`'s credentials — `GOOGLE_APPLICATION_CREDENTIALS`, already present in the deploy sandbox — using `google-auth-library` loaded from firebase-tools' own `node_modules` by absolute path, since Node's ESM resolver doesn't consult `NODE_PATH`; no `gcloud` CLI is installed in this sandbox, so this went through the REST APIs directly rather than `gcloud`/`firebase` subcommands): confirmed `recaptchaenterprise.googleapis.com` and `firebaseappcheck.googleapis.com` were already-enabled APIs on the project (live check, not assumed); self-granted `roles/recaptchaenterprise.admin` + `roles/firebaseappcheck.admin` to the deploy SA (see the projectIamAdmin entry above — kept permanently, not revoked after use, matching this SA's other standing infra roles); created a **reCAPTCHA Enterprise** key (`projects/311094677098/keys/6LdDGlctAAAAAB3lX0HQS9ao_B4Bn8w0O_KFJDYk`, `integrationType: SCORE`, `allowedDomains: [maxtr-c238f.web.app, maxtr-c238f.firebaseapp.com, sudak666.github.io]`) via `recaptchaenterprise.googleapis.com`; registered its site key with the web app's App Check config via `firebaseappcheck.googleapis.com`'s `recaptchaEnterpriseConfig` endpoint. Functionally equivalent for App Check's purposes (both are score-based bot-detection signals); Enterprise has its own free tier (1M assessments/month) well beyond what this app's traffic will hit.
- **Client integration**: `js/core.js` imports `initializeAppCheck`/`ReCaptchaEnterpriseProvider` from `firebase-app-check.js` (pinned to the same `12.11.0` SDK version as every other Firebase import) and calls `initializeAppCheck(fbApp, {provider: new ReCaptchaEnterpriseProvider(siteKey), isTokenAutoRefreshEnabled:true})` right after `initializeApp()`, ahead of `getFirestore()`/`getAuth()` — App Check has to be initialized before those services issue their first request. The site key is embedded directly in the client (`export const appCheck = ...` in `core.js`) — like the Firebase `apiKey` already sitting a few lines above it, this is a *public* key meant to ship in client code, not a secret; it identifies which reCAPTCHA key configuration to use, it doesn't authorize anything by itself.
- **CSP**: `firebase.json`'s `script-src` gained `https://www.google.com` (reCAPTCHA Enterprise's client script loads from there) — `connect-src`/`frame-src` already covered `https://www.google.com` and `https://*.googleapis.com` from the existing phone-auth reCAPTCHA setup, so nothing else needed changing. Verified against the real deployed CSP via `tests/csp-handlers.mjs`, not just read through.
- **`sw.js`**: `firebase-app-check.js` added to `FIREBASE_SDK_ASSETS` (same reasoning as the other 4 gstatic SDK URLs already there — a static ES module import is all-or-nothing, so a cold offline start needs this file precached too) and `CACHE_NAME` bumped.
- **Test suite impact — this was the one genuinely tricky part**: `js/core.js`'s new top-level `firebase-app-check.js` import meant every one of this repo's 14 stubbed-Firebase Playwright tests (they intercept the Firebase SDK's gstatic module URLs via `page.route()` and fulfill them with hand-written stub modules — see "Verify a change" below) needed a matching stub for the new import too, or the whole module graph fails to load and every test hangs/times out on a blank lock-screen. Added an inert `STUB_APP_CHECK` (`initializeAppCheck` returns `{}`, `ReCaptchaEnterpriseProvider`'s constructor does nothing) plus the matching `page.route(...firebase-app-check.js...)` line to all 14 files, mechanically, since they already duplicate every other Firebase stub per-file rather than sharing a module (this repo's established test-file style). **If a 15th Firebase SDK module import is ever added to `js/core.js` (or any other `js/*.js` file), grep every `tests/*.mjs` file for `firebase-app.js` to find every place a matching stub route needs adding** — this is exactly the kind of gap that fails loud (every test hangs) rather than quiet, but better to know the pattern in advance.
- **Deliberately NOT enforced — the one remaining manual step**: checked live via the App Check Management API (`firebaseappcheck.googleapis.com/v1/projects/maxtr-c238f/services`), both `firestore.googleapis.com` and `identitytoolkit.googleapis.com` (Auth) are `UNENFORCED` — the default — and this session left them that way on purpose. In this mode, App Check tokens are attached to requests when available, but a request without one (or one that fails verification — e.g. this repo's own local/sandbox test recipe, which serves from `localhost`, outside the reCAPTCHA key's `allowedDomains`) still goes through exactly as before, same as today. This is Google's own recommended rollout order: monitor real traffic's token-verification rate in the Firebase console's App Check metrics tab for a while, *then* flip to `ENFORCED` once confident real users aren't being false-positived out of their own data. Flipping enforcement before that verification is a real way to lock the account owner out of their own app — a decision this session deliberately left for the account owner to make explicitly (Firebase Console → App Check → Firestore/Authentication → "Enforce"), not something to do unilaterally from an audit checklist. If a future session is asked to "finish App Check" or "turn on enforcement," that PATCH is the only remaining step — everything else above is already live in production.

### Auth
Three sign-in methods, all in the AUTH / PHONE SIGN-IN sections: email+password, Google popup, and phone number (SMS code via `signInWithPhoneNumber` + an invisible `RecaptchaVerifier`). Layered on top, independent of Firebase Auth: a local PIN (SHA-256 hash in localStorage, `mx_pin_<uid>`) and optional WebAuthn biometric unlock — these gate the UI locally but never replace the Firebase session.

Settings → Акаунт additionally lets an already-signed-in user (e.g. one that signed in via Google) **link/unlink a phone number** as an extra credential (`linkWithPhoneNumber(currentUser, ...)` / `unlink(currentUser,'phone')`) — a separate flow from the pre-auth sign-in one, with its own `link-recaptcha-container`/`RecaptchaVerifier` instance so it doesn't collide with the lock-screen's.

**reCAPTCHA retry gotcha**: `RecaptchaVerifier.clear()` can itself throw partway through on a failed attempt, leaving the widget's DOM node behind; the next attempt's fresh `RecaptchaVerifier.render()` then finds a stale widget already in the container and throws `"reCAPTCHA has already been rendered in this element"`. Both phone flows now force the container's `innerHTML` empty on error (`resetRecaptchaContainer(containerId)`) regardless of whether `.clear()` succeeded — do this for any new reCAPTCHA-backed retry flow too.

**Phone auth troubleshooting** (not code issues, but easy to mistake for one): (1) needs the **Blaze** plan (documented below already); (2) needs **Authentication → Settings → SMS region policy** to explicitly allow the country being tested (Firebase blocks SMS to unlisted regions by default); (3) **Brave browser disables Web Push by default** (`brave://settings/privacy` → "Use Google services for push messaging") — causes `AbortError: Registration failed - push service error` on `getToken()`/`PushManager.subscribe()` that looks like a bug but isn't; (4) **GitHub Pages mirror isn't auto-authorized** — `*.web.app`/`*.firebaseapp.com` are automatically in Firebase's Authorized domains, but `sudak666.github.io` is not, so invisible reCAPTCHA fails (flashes and vanishes) and phone sign-in/link dies with a generic error specifically on that mirror while working fine on `maxtr-c238f.web.app`. Add `sudak666.github.io` to Authentication → Settings → Authorized domains manually (see SETUP.md §2). `authErrorMessage()` in `index.html` now maps `auth/captcha-check-failed`, `auth/unauthorized-domain`, and `auth/network-request-failed` to specific messages instead of the generic fallback, to make this diagnosable from the UI itself next time; (5) **`auth/error-code:-39` (503 on `sendVerificationCode`) for every number tried** — not billing, not the API key, not the region policy, not a specific number (confirmed by process of elimination and then by Firebase Support directly, case 10414309, July 2026): Firebase's automated anti-toll-fraud system has temporarily blocked the carrier/region for low SMS-success rates. `authErrorMessage()` maps this (and `auth/internal-error`) to a dedicated message, scoped to phone-flow call sites only via an `isPhoneFlow` param — see SETUP.md §5 for the real fix (enable reCAPTCHA SMS Defense, which is why the Firebase JS SDK is pinned to `12.11.0` rather than `10.x`: SMS Defense requires Web SDK 11+).

### On-device AI category suggestion (audit-followup session)
`js/finance.js`'s `maybeSuggestCategoryWithAI()`, wired to the same `#fin-comment` `input` listener as `applyAutoRuleToForm()` (`js/app-init.js`'s cold-init block, right after it). Uses Chrome's built-in **Prompt API** (`self.LanguageModel`, stable from Chrome 138) — a small language model that ships with the browser and runs the whole categorization **on-device**: no network call, no API key, no per-request cost, and the transaction comment text never leaves the browser. Feature-detected (`typeof self.LanguageModel`) and a total no-op everywhere unsupported, which today is every browser this repo's own test suite runs against (this sandbox's Chromium has no on-device model installed) — this can only ever enhance the form, never block or break it, and `tests/ai-category-suggestion.mjs` proves the real code path by stubbing `self.LanguageModel` via `page.addInitScript` rather than only proving "does nothing when absent" (every other test already covers that implicitly).
- **Keyword rules always win.** `findMatchingRule()` (the existing deterministic, user-authored auto-rules) is checked first; the AI path only runs when nothing matched. A probabilistic guess should never override a rule the user explicitly configured.
- **Debounced (700ms) and staleness-checked at every await boundary** (`stillFresh()`, re-read live from the DOM each time): session creation and the `prompt()` call are both real async work with unpredictable latency, so a suggestion computed for text the user has since changed, cleared, or already picked a category over is discarded rather than landing on top of whatever they're doing now.
- **The model's answer is only ever applied if it exactly matches (case-insensitively) one of the finance type's real category names** (`AppState.categories[type]`) — never lets the model invent a new category string or write anything free-form into the category field. An out-of-list answer is silently ignored.
- **Comments under 8 characters never trigger a call at all** — short notes ("кава", "хліб") are exactly what the keyword rules already handle well, and skipping them avoids a session-create + prompt round-trip on every couple of keystrokes.
- On a match, behaves identically to `applyAutoRuleToForm()`'s own UX (sets the `<select>` value, dispatches `change`, `csSync()`s the custom-select display, shows a `sparkle`-icon toast) but with its own `ai_category_suggested` i18n string so a user can tell an AI guess apart from a rule they configured themselves.

### Receipt scanning (OCR) — a later session
"Сканувати чек" button at the top of the new-transaction form (`#receipt-scan-group`/`#fin-scan-receipt-btn`, hidden for transfers — a transfer has no counterparty receipt). Picks/photographs an image (`<input type=file accept=image/* capture=environment>`, so on mobile this opens the native camera app directly) and runs it through **Tesseract.js** (WASM, on-device OCR) to prefill the amount + date fields — never blocks the form; an unreadable photo just leaves the fields as they were, with a toast saying so.
- **Self-hosted, not CDN-loaded — this was a hard requirement, not a style choice.** Tesseract.js's own defaults pull its library/WASM core/language data from `cdn.jsdelivr.net` at runtime, but this app's CSP `script-src` has no CDN allowance (`'self'` + a short Google-origin allowlist only, no `'unsafe-inline'`/`'unsafe-eval'` — see the CSP section above), so a CDN-default setup would just fail silently under CSP. Vendored the whole dependency chain into `js/vendor/tesseract/` instead (~9MB: `tesseract.esm.min.js` the library, `worker.min.js` the worker script, `tesseract-core-lstm.wasm.js` the WASM core — LSTM-only, non-SIMD, chosen deliberately for universal browser compat over the faster SIMD variant — plus `lang-data/{ukr,eng}.traineddata.gz`, the quantized `best_int` language packs). Pulled via `npm pack` (registry.npmjs.org is reachable from this sandbox; `unpkg`/`jsdelivr` are not) rather than committed from a browser download.
  - **`tesseract-core-lstm.wasm.js` alone is sufficient — no separate `.wasm` binary file needed.** The npm package also ships a same-named-minus-`.js` `tesseract-core-lstm.wasm` (2.8MB) binary; initial vendoring included it too on the assumption `corePath`'s `.js` file would fetch it separately at runtime. A real (non-stubbed) OCR run against a synthetic receipt image, driven through this repo's Playwright recipe with network-request logging, showed the `.wasm` file is **never actually requested** — the `.wasm.js` wrapper already embeds the WASM binary inline as base64 (its size, ~3.9MB, is almost exactly the `.wasm` file's 2.8MB inflated by base64's ~33% overhead, confirming this). Deleted the redundant `.wasm` file after confirming the real E2E OCR run still succeeded identically without it — cut the vendored footprint from ~12MB to ~9MB for zero functional loss. Worth re-checking this assumption if the vendored Tesseract.js version is ever upgraded, rather than assuming the current file set stays minimal.
  - **`corePath`/`workerPath`/`langPath` deliberately point at exact same-origin files, and `workerBlobURL:false` is required.** Tesseract.js's browser default spawns its worker by wrapping `workerPath` in a `Blob`+`URL.createObjectURL()` (`new Worker(blobURL)`) rather than `new Worker(workerPath)` directly — this app's CSP `worker-src` is `'self'` only, with no `blob:` allowance, so the default would be silently blocked. `js/receipt-ocr.js`'s `scanReceiptImage()` passes `workerBlobURL:false` to force the direct same-origin-URL path instead, which needs no CSP change. Passing `corePath` as an exact `.js` file path (not a directory) also sidesteps Tesseract.js's own SIMD-feature-detection branching (`wasm-feature-detect`), which would otherwise require vendoring *both* SIMD and non-SIMD core variants.
  - **`script-src` needed one real CSP addition: `'wasm-unsafe-eval'`** (`firebase.json`) — `WebAssembly.instantiate()` is blocked by a strict CSP without it (a narrower, WASM-only counterpart to `'unsafe-eval'`; it does not enable `eval()`/`new Function()` for ordinary JS). No other CSP directive needed changing — confirmed empirically (not just reasoned about) that image data never touches `img-src`/`connect-src`/blob URLs at all: a `File`/`Blob` passed to `worker.recognize()` is read via `FileReader.readAsArrayBuffer()` internally, never rendered through an `<img>`/canvas/object-URL.
  - **`js/receipt-ocr.js` uses two different base paths for the same directory, on purpose**: a dynamic `import()` specifier resolves relative to *the importing module's own URL* (so `./vendor/tesseract/...` from `js/receipt-ocr.js` correctly resolves to `js/vendor/tesseract/...`), while the plain path strings Tesseract.js's `createWorker()` takes (`workerPath`/`corePath`/`langPath`) are resolved by the library itself relative to `window.location.href` (the page, always at the app's root whichever host serves it — `/` on Firebase Hosting, `/Maxtr/` on GitHub Pages), so those need the `./js/vendor/tesseract/...` form instead. Mixing the two up 404s silently.
  - **Real end-to-end OCR was verified once by hand, under the actual deployed CSP, before this shipped** — generated a synthetic receipt PNG (Pillow), served `index.html` with the exact `Content-Security-Policy` header read live from `firebase.json` (same technique `tests/csp-handlers.mjs` already uses), and drove a real (non-stubbed) scan through Playwright. Caught two real things this way that a stubbed test alone would have missed entirely: the CSP/worker-spawn/WASM-compile chain actually working end-to-end (confirmed the design above is sound, not just plausible), and a real parsing bug (next bullet).
  - **Amount-extraction fallback bug, found by the real E2E run, not by unit tests written in advance**: when no line matches the total/сума keyword regex, the parser falls back to "largest number in the text" — but a receipt's own date (e.g. `16.07.2026`) contains a bare `2026`, which is numerically larger than a typical amount and was winning over the real total. Fixed in `parseReceiptText()` (`js/receipt-ocr.js`) two ways: the matched date span is masked out of the text before amount extraction runs at all, and the fallback now prefers decimal-formatted numbers (`X.XX`/`X,XX` — how receipts actually print money) over bare integers, only falling back to bare integers if literally no decimal-formatted number exists. Locked in as a named regression case in `tests/receipt-ocr-parse.mjs`.
  - **CI/local Playwright test stubs the one dynamic-`import()` boundary** (`js/vendor/tesseract/tesseract.esm.min.js`, via `page.route()` — same technique every other test here uses for the Firebase SDK's gstatic URLs) rather than running real WASM OCR every run, which would be slow. `tests/receipt-scan-ui.mjs` exercises the real `js/receipt-ocr.js`/`js/finance.js` code against a fake `createWorker()`/`recognize()` backend. **Needed `browser.newContext({serviceWorkers:'block'})`, not the plain `browser.newPage()` every other test here uses**: `sw.js` calls `clients.claim()` on activation, and by the time this test's later "pick a receipt" step fires its dynamic import, the SW has often already claimed the page and its `fetch` handler intercepts the request *before* Playwright's `page.route()` ever sees it (silently loading the real ~9MB vendored library instead of the stub). Every other test here gets lucky by stubbing Firebase SDK imports that resolve immediately on page load, before the SW finishes installing — this is the first test in the suite whose stub target is reached late enough to hit the race for real; worth remembering for any future test whose stubbed fetch doesn't happen in the first moment or two of page load.
  - **Real-device follow-up bug, found by the account owner on an actual phone within hours of shipping**: tapping the button opened the camera and took the photo fine, but the scan then never completed — the "Розпізнаю чек…" toast (a fixed 2.8s auto-hide, `showToast()`'s own generic timing) vanished with no further feedback, and the button stayed disabled until the tab was closed and reopened (no in-app recovery). Root-caused to two compounding issues neither of which showed up against this repo's small synthetic test image: (1) a real phone camera photo is routinely 3000-4000px on the long side, several MB — feeding that straight into the non-SIMD LSTM-only WASM core on a memory-constrained mobile browser is a known way to make Tesseract.js extremely slow or hang the worker outright with no JS-catchable error ever posted back; (2) even when it does eventually finish, a multi-second-to-tens-of-seconds real scan has zero UI feedback once the fixed-duration toast auto-hides, reading as "nothing happening" even when it's actually still working. Fixed both in `js/receipt-ocr.js`/`js/finance.js`: `downscaleImage()` (createImageBitmap + canvas, capped at 1800px on the long side, re-encoded as JPEG) runs before every `recognize()` call — verified for real against a synthetic 3024×4032 "phone photo," confirming it actually gets downscaled to 1350×1800 and still recognizes correctly, total wall time ~20s including cold vendor-asset download; and `handleReceiptFile()` now swaps the scan button's own label to the "processing" text for the whole duration (a persistent indicator, not a transient toast) rather than relying on the toast alone. **Also added a hard 45s timeout** (`withTimeout()` wrapping the whole `scanReceiptImageInner()` call) as a defensive backstop regardless of root cause — a genuinely stuck call (hung fetch, wedged WASM, anything) now rejects on its own after 45s, re-enabling the button and showing a dedicated `receipt_scan_timeout` toast, instead of leaving the UI stuck until a manual reload. The abandoned original promise isn't force-cancelled (JS can't do that) but is harmless — `scanReceiptImageInner`'s own `finally` still terminates the worker whenever/if it eventually settles on its own. Verified the recovery path for real (not just reasoned about) by temporarily patching `OCR_TIMEOUT_MS` down and a hung-forever stubbed `recognize()`: confirmed the button disables during the scan, then automatically re-enables with the correct toast and restored label after the timeout, with no page reload needed. `scanReceiptImage()` takes an optional `timeoutMs` override (unused by `finance.js`'s real call site, default unchanged) purely so `tests/receipt-scan-ui.mjs` could turn this into a fast (~1s) permanent regression test instead of a 45s one. `sw.js` `CACHE_NAME` bumped `rytm-v66`→`rytm-v67`.
  - `js/vendor/tesseract/*` is **not** added to `sw.js`'s `STATIC_ASSETS` (unlike every other `js/*.js` file) — it's only ever reached via a dynamic `import()` inside a function body, never a static import in the core module graph, so eagerly precaching ~9MB on every install for a feature most sessions never touch would be wasteful. It gets cached opportunistically by the generic same-origin cache-first fetch handler the first time a user actually scans a receipt. `js/receipt-ocr.js` itself (the small wrapper module, statically imported by `js/finance.js`) **is** added to `STATIC_ASSETS`, same as any other core module.
  - Apache-2.0 licensed (Tesseract.js and tesseract.js-core both) — `LICENSE-tesseract.js.md`/`LICENSE-tesseract.js-core.txt` kept alongside the vendored files for attribution.

### UI conventions worth following
- Custom-styled `<select>`/`<input type=date>`: any new one just needs `enhanceSelect()` / `enhanceDateInput()` called on it (or add it before `enhanceAllSelects()` runs) — don't hand-roll another dropdown.
- **Never use `<input type=color>`** — it opens the browser's native, unstylable OS color-picker popup, which looks jarringly out of place against the app's dark custom UI. Use the shared `openColorPicker(kind, id)` / `color-pick-modal` pattern instead (a `PALETTE`-swatch grid modal, same idea as the avatar picker) — wire a new color-editable list (wallets/shiftTypes/tags all already do this) by adding a case to `selectPickedColor()` and rendering a plain `<button class="mgr-color" style="background:${color}" onclick="openColorPicker('kind','${id}')">` instead of a color input.
- Similarly, don't rely on native browser affordances that can't be restyled or that read as "not part of the app" — e.g. no `title="..."` tooltips (native OS tooltip box); prefer a visible label/subtitle in the UI itself, like the rest of the settings rows do.
- "Settings manager" modals (wallets, categories, budgets, goals, recurring, tags, rules, rates, widgets, pin, premium, profiles, link-phone, avatar/color pickers) all follow the same pattern: a `.modal-overlay` + `.modal-card`, an `open*Manager()`-style function that renders a list into it and sets `display:flex`, and the shared `closeManagers()` that hides all of them by id — add new manager ids to that shared close-list (an easy miss: forgetting this leaves a modal that won't close on backdrop-click). **Separately, every `.modal-overlay` in `index.html` (23 of them, including the two — `#shift-modal`, `#ui-dialog` — that `closeManagers()`'s id list doesn't cover) already gets focus-trap/Escape-to-close/focus-restore for free**, via `setupModalAccessibility()` (`js/settings-managers.js`, called once from `js/app-init.js`'s cold-init block): a `MutationObserver` watches every `.modal-overlay`'s `style` attribute, and on open pushes the overlay onto `AppState.openModalStack`, remembers `document.activeElement` as the element to restore focus to, and auto-focuses the first focusable element inside the `.modal-card`; a single `document`-level `keydown` listener traps `Tab`/`Shift+Tab` within the topmost open modal's card and closes it on `Escape` (by synthetically `.click()`-ing the overlay itself, which reuses whatever close logic that specific modal already has, rather than duplicating it). This is a *different*, more robust choke point than `closeManagers()`'s hard-coded id list — any new modal gets this accessibility behavior automatically just by having `.modal-overlay`/`.modal-card` markup present in the DOM at `init()` time, no id-list registration needed (though it still needs adding to `closeManagers()` separately for backdrop-click/close-all). Covered by `tests/e2e-modals.mjs`.
- All icons are the custom monochrome SVG set (`window.Icon(name)` / `setIcon(id, name)`) — no emoji, no icon fonts. Add new glyphs to `ICON_PATHS` rather than reaching for an emoji. **Check for duplicates within the same visible list/row-group** before reusing an existing glyph — two *different* features sitting next to each other (e.g. two quick-action buttons, two settings-card icons in the same screen) with the same icon reads as a bug to users even though it isn't one; reusing an icon for the *same* feature across contexts (a tab + its settings entry + its quick-action) is fine and intentional. Generic action icons (`plus` for add, `trash` for delete, `pencil` for edit, `chevron`, `refresh`) are meant to repeat everywhere — that's a consistent affordance, not a duplicate.
- **"Hide amounts" (topbar `#btn-hide-amounts`, `body.amounts-hidden` in `index.html`) is an opt-in CSS class list, not a generic "blur anything that looks like money" rule** — it works by `filter:blur(7px)` on a fixed list of selectors (`.stat-val`, `.tx-amount`, `.hero-balance-val`, `.wallet-chip-val`, `.budget-widget-val`, `.goal-widget-val`, `.debt-amount-val`, `.analytics-donut-total`, etc. — grep `body.amounts-hidden` in `index.html` for the current full list), so **any new money display needs its own dedicated class added to that list, or it silently leaks** — this shipped for real (reported via screenshot, "ще не всі суми приховані"): the Finance-tab Budgets/Goals widgets, the Debt tab's payment-history rows + discrepancy hint + payoff-forecast average, and a cross-currency transfer's target-amount note had no class at all and were never blurred despite the rest of the screen being hidden. **Don't reuse a shared row/field class already used for non-money text** (e.g. `.debt-field-view` is used for a debt entry's amount, balance, *and* date fields) — blurring the whole shared class would also hide the date, which isn't sensitive and reads as a bug the other way; give the money-only span/div its own class (`.debt-amount-val` in that example) instead of blanket-covering the shared one. Covered by `tests/hide-amounts.mjs`, which also asserts the date field specifically stays visible (the over-broad-blur regression this pattern guards against).
- **Calendar day cells show shifts as compact colored letter tokens** (`.shift-token`, a 1-2 char "keycap"), not word-badges — `renderCalendar()` (`js/calendar.js`) renders `shiftCode(t)` per shift, wrapped in a `.day-tokens` flex row. `shiftCode()` prefers an explicit `code` field on the shift type (set on the built-ins in `js/core.js`: Д/Д+/Н/Нр/Нп/В — explicit because several share a first letter) and falls back to a derived first-letter for custom/legacy types. `short`/`name` stay the readable labels (legend, managers). When adding a built-in shift type, give it a `code`.
- **Debt tab has a payoff-forecast card** (`#debt-forecast`, `renderDebtForecast()` in `js/debt.js`): an inline-SVG balance-burndown mini chart + a "≈ N payments to go at this pace" estimate (average paydown-per-payment; by payment count, not date, since `entry.date` is free-form text). Shown only with enough signal (startAmount set, ≥2 payments, positive downward-trending balance). Pure-derived from existing entries — no schema change.
- i18n: a single `I18N = {uk:{...}, en:{...}}` object plus `tr(key)`; static markup uses `data-i18n`/`data-i18n-placeholder` attributes applied by `translateStaticDOM()`. There is no `data-i18n-title` — dynamic titles/tooltips must be set in JS.
- The bottom nav (`.tabs`) currently has 5 tabs (Фінанси/Графік змін/Розрахунки/Покупки/Налаштування) — `.tab-icon-wrap` is 42px (38px under the ≤600px breakpoint). Adding a 6th tab means re-checking these still fit comfortably; there's no overflow/scroll handling on that bar.
- `sw.js`'s `fetch` handler must bail out (`return` without calling `respondWith`) for any cross-origin request and let the browser handle it natively — it used to re-issue *every* request including third-party ones through `respondWith(fetch(event.request))`, which throws for some no-cors/redirect combinations (this broke reCAPTCHA's own CSP-report beacon with a console `TypeError: Failed to fetch`). Don't reintroduce a `respondWith` path for anything outside `self.location.origin`.

### Mobile UI redesign (multi-session pass, July 2026)
A design review (screenshots + before/after mockups) plus several rounds of live-phone-screenshot feedback drove a rework of the densest/most-cluttered screens. The patterns below are now the established convention — follow them for anything new rather than reverting to the older per-screen styles they replaced.
- **Hero metric + chip row** (`.hero-metric`/`.hero-balance` + `.chip-stat-row`/`.chip-stat`): replaces "3-4 equal-weight stat cards in a grid" (the old `.stats-row` of `.stat-card`s). One large primary number up top, everything else demoted to a horizontal scrollable strip of small pill chips. Used on Finance (balance + wallet chips), Shifts (earned + hours/shifts/days-off), Debt (balance + start/paid/count), Shopping (remaining/bought). `.stat-card`/`.stats-row` still exist and render fine (Analytics tab still uses them for its 4 equal metrics, where equal weighting is actually correct) — they're not deprecated, just no longer the default for "here's a bunch of numbers" on a primary tab.
- **Compact manager row** (`.budget-row`, `.rate-row`, `.cat-row`): replaces "one full-width card per list item with a giant input" in settings managers. Single line: icon badge, name (flex, ellipsis-truncated — needs `min-width:0` on *every* ancestor in the flex chain up to the row itself, not just the text element, or a nowrap row can silently overflow its own card on long text; see the `.cat-row{min-width:0}` fix) + a small right-aligned input/value. Used in Budgets and Курси валют (rates) managers. Categories manager uses the same row shape but with a two-line text block (name + subcategory/amount caption) instead of a single value, since it has more to show per row. **Auto-rules and recurring managers (a later session)** were the last two holdouts still using the pre-redesign `.debt-row` always-expanded stacked-fields pattern (`.debt-field`/`.debt-field-label`, `flex-wrap:wrap`) — converted to the same collapsed-summary-row-with-a-pencil-toggle-to-expand shape the shift-types/categories managers already used (`AppState.expandedRuleId`/`.expandedRecurringId`, mirroring `expandedShiftTypeId` exactly): a `.mgr-row` with an icon badge + two-line `.cat-row-body` summary (rule: `"keyword" → category`; recurring: `category · amount` / `frequency · next date`) by default, expanding inline below the summary row when the pencil is tapped. **Found and fixed a real pre-existing bug while doing this**: `.mgr-num` (the shared "compact number input" class, used by shift-types' Pay/Hours fields too, not something this pass introduced) was `flex:1 1 90px;width:90px` — every real usage nests it inside `.mgr-field` (`display:flex;flex-direction:column`), where `flex-basis:90px` applies to the *column's* main axis (vertical), silently making the input 90px **tall** as well as 90px wide (rendered as a solid near-black square with no visible text) — caught via live screenshot verification of the new recurring-manager fields, not by reading the CSS. Fixed by dropping the `flex` shorthand entirely (`width:90px` alone is sufficient; the grow/shrink behavior wasn't doing anything useful in this context), which also silently fixed the same latent bug in the shift-types manager's Pay/Hours inputs.
- **FAB + bottom-sheet modal** (`.fin-fab` + `#tx-form-modal`): the Finance tab's "Нова операція" form used to render permanently expanded inline, eating a quarter of the screen on every visit. It's now a modal (same `.modal-overlay`/`.modal-card` pattern as every other manager) opened by a floating purple `+` button bottom-right. `editTransaction(id)` opens the same modal pre-filled instead of scrolling to an inline form. Same idea applies if any other "always-expanded form on a primary tab" shows up again.
- **PWA shortcuts + Web Share Target (audit-followup session)**: `manifest.json`'s `shortcuts` array adds a long-press-the-app-icon entry ("Нова операція" → `./index.html?action=new-tx`) and its `share_target` (GET-based — `{title,text,url}` query params, no service worker fetch handler needed) lets another app's native share sheet hand text straight to Rytm. Both land on the same handler, `js/app-init.js`'s `handleLaunchParams()`, called as the last statement of `init()` (after `fbLoadNow()`, so the tx form's wallet/category selects already have real data, not seeded placeholders): it reads `URLSearchParams`, strips the query string immediately via `history.replaceState` (so a later reload/relaunch from history doesn't reopen the modal), then clicks the same `[data-action="open-new-tx-modal"]` FAB the user would tap manually rather than duplicating `openNewTxModal()`'s logic through a new export. A shared `title`+`text` also prefills the comment field with the shared text and, best-effort, the amount field with the first plain number found in it (e.g. a price copied from a receipt/chat) — never blocks the form if nothing numeric is found. Bumped `sw.js`'s `CACHE_NAME` (`rytm-v61`) so already-installed PWAs actually pick up the changed `manifest.json`, since it's cache-first precached like every other `STATIC_ASSETS` entry. Covered by `tests/pwa-launch-params.mjs` (shortcut-launch, share-launch, and a plain-load-opens-nothing case).
- **Swipe-to-delete list rows** (Telegram/Gmail-style): the established pattern for any list row with a destructive delete action, now used by both the Finance transaction list (`.tx-item`/`.tx-item-inner`/`.tx-swipe-delete`, `setupTxSwipe()` in `js/analytics-csv.js`) and the Debt tab's payment-history rows (`.debt-row`/`.debt-row-inner`/`.debt-row-swipe-delete`, `setupDebtRowSwipe()` in `js/debt.js`). The outer element is just a `position:relative;overflow:hidden` clipping shell; the visible card (background/border/padding) lives on an `*-inner` child that shrinks its own `width` (never `transform:translateX` — that clips left-edge content past the shell's `overflow:hidden`, caught via screenshot) on swipe/hover/`:focus-within` to reveal a red delete button sitting behind it as a sibling. Replaced an earlier always-visible small delete icon sitting directly next to edit — on touch devices the delete icon had no visual "this is destructive" cue until `:hover` (which never fires on touch), so it read as visually identical to the adjacent edit icon and was easy to mis-tap; hiding it behind an explicit swipe removes that ambiguity entirely rather than just enlarging/recoloring the icon. The two implementations are intentionally duplicated (not a shared helper) — different outer/inner class names per list, matching this repo's established per-file self-contained render-logic style (same reasoning as the `tests/*.mjs` stub-duplication convention below). Covered by `tests/tx-swipe-delete.mjs` and `tests/debt-swipe-delete.mjs`. **Gotcha if you swipe-clip a row inside a flex list container**: a flex item with any `overflow` other than `visible` gets an automatic minimum size of **0** instead of "the size of its content" (CSS flexbox spec) — `.debt-row` sits inside `.debt-list`, which is `display:flex;flex-direction:column;max-height:...;overflow-y:auto`, so without an explicit `flex-shrink:0` on `.debt-row` the browser silently shrank every row down to a sliver once total content height exceeded the list's `max-height` (any account with more than ~4 entries), clipping almost everything. **`.tx-item` has since hit the exact same bug and now also carries `flex-shrink:0`** — `#tx-list-container` (`.tx-list`) IS a flex column (`display:flex;flex-direction:column;max-height:500px;overflow-y:auto`), so once the expanded "view all" list overflowed 500px every `.tx-item` (also `overflow:hidden` for its swipe-clip) crushed to a ~12px sliver; this shipped and was reported via screenshot, and *also* silently broke pull-to-refresh (a non-overflowing container reads as non-scrollable, so `closestScrollableAncestor()` returned null and PTR fired on swipes over the list). An earlier version of this note wrongly claimed `.tx-item` was immune because the container "isn't a flex container" — it is; don't trust that. Both `.tx-item` and `.debt-row` are covered by geometry regression assertions (`tests/stress-tx-list.mjs`, `tests/debt-swipe-delete.mjs`). Add `flex-shrink:0` to any new swipe-clipped row that lives inside a `display:flex` list — caught only via a live screenshot/geometry check at realistic row counts, not by reading the CSS. **`.tx-swipe-delete`/`.debt-row-swipe-delete` also carry their own `border-radius:18px`** (matching the row) — the outer row's `overflow:hidden` clip already rounds the button's *far* edge (right side) for free, but without a radius on the button itself, its *near* edge (revealed flush against the shrunk inner card) stayed sharp-cornered, reading as a jarring rectangle stuck onto an otherwise fully-rounded card rather than one smooth shape. Round all four corners on any new swipe-reveal button, not just relying on the outer clip.

### Android / Google Play packaging (TWA, July 2026)
The app ships to Google Play as a **Trusted Web Activity (TWA)** — a thin Android wrapper that opens the live `maxtr-c238f.web.app` site full-screen, not a separate native codebase. There is no local Android project checked into this repo; it's generated on demand.
- **Canonical domain for the Android package is `maxtr-c238f.web.app`**, not the GitHub Pages mirror — chosen because Play's Digital Asset Links verification needs one fixed origin, and the Firebase domain has no subpath (GitHub Pages serves under `/Maxtr/`, which would complicate `start_url`/asset-link matching).
- **`.well-known/assetlinks.json`** (repo root) is what lets Android show the app without a browser URL bar — it declares the app's package ID + the signing certificate's SHA-256 fingerprint, and Google Play/Android verifies it against `https://maxtr-c238f.web.app/.well-known/assetlinks.json`. `firebase.json`'s hosting `ignore` list originally excluded all dotfiles (`**/.*`), which would have silently dropped this file from every deploy — it now has an explicit `"!/.well-known/**"` un-ignore plus a `Content-Type: application/json` header override for that path. If this file's fingerprint and the actual signing key ever drift apart (e.g. the key gets regenerated), the app degrades to showing a Chrome URL bar instead of failing outright — annoying but not broken, and the fix is always "update the fingerprint in this file, redeploy hosting."
- **Package ID is `ua.rytm.app`** (changed from the placeholder `ua.zminka.app` on 2026-07-16 once the account owner made the final "Rytm" branding decision — see Project overview). Still freely changeable in whatever tool generates the package (twa-manifest.json / PWABuilder's Package ID field) up until the *first* real Play Store upload, which per the "Google Play packaging" status below hasn't happened yet; after that, changing it means publishing as a brand-new listing, not an update.
- **The actual APK/AAB build cannot happen from a Claude Code cloud sandbox.** Confirmed by hitting explicit 403 policy denials from this environment's egress proxy on `dl.google.com` (Android SDK/build-tools — blocks even the offline `bubblewrap build` path), `pwabuilder.com`, and even `maxtr-c238f.web.app` itself. Only `github.com`/npm/pypi-class hosts and Google's OAuth/Firebase *API* endpoints (as opposed to the Hosting *content* domain) were reachable. What **did** work fully offline from the sandbox: generating the Android Studio project skeleton via `@bubblewrap/core`'s `TwaManifest`/`TwaGenerator` classes directly (bypassing the CLI's interactive prompts, feeding it the local `manifest.json` instead of fetching it live, and serving the icon files over a `127.0.0.1` static server so `TwaGenerator` had something reachable to pull them from), and generating the release signing keystore with the locally-installed JDK's `keytool` (no network needed at all for that part). The actual compile step then has to happen either in Android Studio on a real machine, or — the path actually used — **pwabuilder.com** from the user's own browser, pointed at `https://maxtr-c238f.web.app`.
- **Signing keystore is intentionally not committed to this repo** — it was generated once (`keytool -genkeypair`, RSA/3072, PKCS12 format) and handed directly to the account owner as a downloadable file; losing it means never being able to publish an update under `ua.rytm.app` again (the keystore itself isn't tied to the package ID, so the 2026-07-16 `ua.zminka.app`→`ua.rytm.app` rename didn't require regenerating it — only `.well-known/assetlinks.json`'s `package_name` field needed updating, which it now has). If the keystore itself ever needs regenerating, `.well-known/assetlinks.json`'s fingerprint has to be updated and hosting redeployed to match, or the "no browser URL bar" behavior breaks.
- **PKCS12 keystore gotcha, cost real debugging time**: `keytool` on JDK 9+ defaults to PKCS12-format keystores, which — unlike the older JKS format — **do not support a separate password for the private key entry**; `keytool -genkeypair -keypass X -storepass Y` with `X != Y` silently *succeeds* but prints `Warning: Different store and key passwords not supported for PKCS12 KeyStores. Ignoring user-specified -keypass value.` and just reuses the store password for the key too. Any consumer that's later told "the key password is X" (PWABuilder, `apksigner`, etc.) will then fail with `javax.crypto.BadPaddingException: Given final block not properly padded` — a decryption error, not an authentication-rejected error, which reads like file corruption rather than a wrong-password problem. **Always generate PKCS12 signing keys with the same value for both `-storepass` and `-keypass`**, and only document/hand off one password, not two.
- **`android-icon-rounded.png`** (repo root) is a squircle-rounded-corner variant of `icon-512.png`, generated with Pillow — used *only* by pasting its URL into PWABuilder's Android package form "Icon URL" field, not referenced anywhere in `manifest.json`. Reason: the TWA's launch splash screen centers the raw icon on the manifest's `background_color` with **no adaptive-icon masking applied** (unlike the home-screen launcher icon, which Android masks itself), so `icon-512.png`'s hard square corners looked jarring specifically on that one screen. The form's separate "**Maskable icon URL**" field must keep pointing at the original full-bleed `icon-512.png` — pre-rounding that one would break the safe-zone contract Android's own adaptive-icon masking expects.
- **`icon-192-rounded.png`/`icon-512-rounded.png`** (repo root, same 20%-corner-radius Pillow squircle mask as `android-icon-rounded.png` above, generated independently rather than reusing that file so its own single-purpose documentation above stays accurate) exist for the same underlying reason as `android-icon-rounded.png`, but for the desktop-PWA-install case instead of the Android TWA case: `manifest.json`'s `icons` array used to list `icon-192.png`/`icon-512.png` once each with `"purpose": "any maskable"` combined — but "any" (used as-is, no OS masking — e.g. the Windows desktop shortcut Chrome creates on `chrome://apps` "Install" or a taskbar pin) and "maskable" (Android applies its own adaptive-icon shape mask and needs a full-bleed, un-rounded source) are contradictory requirements on the same file. The combined purpose meant Windows showed the raw square `icon-512.png` with hard corners on desktop shortcuts (reported directly by the account owner via a screenshot). **Fix**: split into four `icons` entries — `icon-*-rounded.png` at `"purpose": "any"`, original `icon-*.png` (unchanged, still full-bleed square) at `"purpose": "maskable"` only. Both new files added to `sw.js`'s `STATIC_ASSETS` (same reasoning as every other same-origin asset there). Deliberately did **not** touch `index.html`/`landing.html`'s `apple-touch-icon` link tags or `js/notifications.js`'s notification icon references — both still point at the original square `icon-192.png` on purpose: iOS applies its own squircle mask to `apple-touch-icon` automatically (pre-rounding it would double-mask/look wrong), and OS notification-tray icons are typically shown small enough that corner rounding doesn't matter and isn't worth another asset variant.
- Rebuilding the Android package from scratch (e.g. after a manifest/icon change) is a manual, browser-based process: pwabuilder.com → enter `https://maxtr-c238f.web.app` → Package For Stores → Google Play → fill Package ID / Icon URL / Maskable icon URL → Signing key → "Mine" → upload the existing `.jks` and its one password → Download Package. There's no CI/automation for this yet.
